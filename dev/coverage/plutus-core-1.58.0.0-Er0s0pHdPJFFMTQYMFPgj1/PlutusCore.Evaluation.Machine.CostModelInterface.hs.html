   39 {- Note [Cost model parameters]
   40 We want to expose to the ledger some notion of the "cost model
   41 parameters". Intuitively, these should be all the numbers that appear in the
   42 cost model.
   43 
   44 However, there are quite a few quirks to deal with.
   45 
   46 1. BuiltinCostModel is stuctured.
   47 
   48 That is, it's a complex data structure and the numbers in question are often
   49 nested inside it.  To deal with this quickly, we take the ugly approach of
   50 operating on the JSON representation of the model.  We flatten this down into a
   51 simple key-value mapping (see 'flattenObject' and 'unflattenObject'), and then
   52 look only at the numbers.
   53 
   54 2. We use CostingIntegers, Aeson uses Data.Scientific.
   55 
   56 The numbers in CostModel objects are CostingIntegers, which are usually the
   57 64-bit SatInt type (but Integer on 32-bit machines).  Numerical values in
   58 Aeson-encoded JSON objects are represented as Data.Scientific (Integer mantissa,
   59 Int exponent). We should be able to convert between these types without loss of
   60 precision, except that Scientific numbers of large magnitude will overflow to
   61 SatInt::MaxBound or underflow to SatInt::MinBound.  This is OK because
   62 CostModelParams objects should never contain such large numbers. Any Plutus Core
   63 programs whose cost reaches MaxBound will fail due to excessive resource usage.
   64 
   65 3. BuiltinCostModel includes the *type* of the model, which isn't a parameter
   66 
   67 We can just strip the type out, but in particular this means that the parameters are
   68 not enough to *construct* a model.  So we punt and say that you can *update* a
   69 model by giving the parameters. So you can take the default model and then
   70 overwrite the parameters, which seems okay.
   71 
   72 This is also implemented in a horrible JSON-y way.
   73 
   74 4. The implementation is not nice.
   75 
   76 Ugly JSON stuff and failure possibilities where there probably shouldn't be any.
   77 
   78 5. The overall cost model now includes two components: a model for the internal
   79 costs of the evaluator and a model for built-in evaluation costs.  We just
   80 re-use the technique mentioned above to extract parameters for the evaluator
   81 costs, merging these with the parameters for the builtin cost model to obtain
   82 parameters for the overall model.  To recover cost model components we assume
   83 that every field in the cost model for the evaluator begins with a prefix (eg
   84 "cek") which is does not occur as a prefix of any built-in function, and use
   85 that to split the map of parameters into two maps.
   86 
   87 -}

   89 {- Note [Cost model parameters from the ledger's point of view]
   90 
   91 A newly-voted protocol update may have some of the following effects:
   92 
   93 a) Introduce a new plutus-language version accompanied by a whole new set of cost model parameters for that new version.
   94 b) Update any values of existing cost model parameters for an existing plutus version.
   95 c) Bring new builtins to an existing plutus version, thus also extending
   96 the set of current cost model parameters of that version with new parameters for the new builtins.
   97 
   98 Note that, removing existing builtins is only possible by issuing a brand-new plutus language version
   99 that would exclude them (and their cost model parameters).
  100 An alternative without issuing a new protocol update and affecting a current version,
  101 would be to set via (b) the builtins' cost model parameters prohibitively high,
  102 so as to make those builtins impossible to execute given the current maximum transaction budget,
  103 thus effectively disabling their execution.
  104 
  105 The ledger-state stores a *mapping* of each plutus-language version to its current set of cost model parameters.
  106 Each set of parameters is represented as an *array of plain integer values*,
  107 without the  cost model parameter names. Overall this ledger-state can be conceptualized as `Map PlutusVersion [Integer]`.
  108 
  109 The implementation of (a) and (b) effects by the node software is straightforward:
  110 for (a) introduce a whole new entry to the Map; for (b) update the affected array in-place with the new values.
  111 As a consequence for implementing (c), we arrive to four restrictions that hold for ALL protocol updates:
  112 
  113 1) In case of (b), the values of the existing parameters may change, but their *meaning*,
  114 i.e. which parameter name and thus which builtin parameter they correspond to,  ABSOLUTELY CANNOT change. In other words,
  115 the existing parameters cannot be re-ordered.
  116 
  117 2) In case of (c) the new parameters (for the new builtins) must be appended **to the end** of the existing array for the affected version.
  118 
  119 3) The node software is responsible to re-create the plutus runtimes when a new protocol is voted in ---
  120 one plutus runtime a.k.a. `plutus-ledger-api.EvaluationContext` per plutus-version.
  121 
  122 4) The node software will always pass the array in its entirety to each plutus-runtime,
  123 and not partially just the updated-parameter values (in case of (b)) or just the new-parameter values (in case of (c)).
  124 
  125 To make (c) work, we must allow a node to continue operating when receiving either more or
  126 fewer cost model parameters than it expects. As an example, suppose at the beginning of
  127 major protocol version 9 (PV9), PlutusV3 has 100 cost model parameters. During PV9, we add
  128 some more builtins to Plutus V3 (to be enabled after the hard fork, at PV10), requiring 20
  129 additional cost model parameters. Then, one submits a proposal updating the number of PlutusV3
  130 cost model parameters to 120.
  131 
  132 During PV9, both node-9.x and node-10.x must operate normally and agree on everything. This means
  133 node-9.x must allow receiving more cost model parameters than it expects (since it may receive
  134 120), and node-10.x must allow receiving fewer than it expects (since it may receive 100).
  135 Node-10.x should fill in the missing parameters with a large enough number to prevent the new
  136 builtins from being used, in case the hard fork to PV10 happens without first updating the number
  137 of PlutusV3 cost model parameters to 120 (which is unlikely to happen, but just in case).
  138 
  139 During PV10, node-9.x stops working.
  140 
  141 The overall logic for the expected number of cost model paremeters is as follows:
  142 
  143 (expected number in node software == received number by ledger) => NOWARNING & NOERROR
  144 (expected number in node software < received number by ledger)  => WARNING
  145 (expected number in node software > received number by ledger)  => WARNING
  146 
  147 If the received number is EQ or GT the expected (WARNING), we
  148 will take the first n from the received cost model parameters (n==expected number),
  149 and create the internal (nameful) representation of cost model parameters, by assigning a parameter name
  150 to its value: see `PlutusLedgerApi.Common.ParamName.tagWithParamNames` and the `ParamName` datatypes in
  151 plutus-ledger-api.
  152 
  153 If the received number is LT the expected (WARNING), we will fill in the missing parameters
  154 with maxBound :: Int64.
  155 
  156 See https://github.com/IntersectMBO/cardano-ledger/issues/2902 for a discussion
  157 of these issues and the rationale for adopting the system described above.
  158 
  159 -}

  161 {- Note [Table of all possible ledger's states w.r.t. cost model parameters update]
  162 What Note [Cost model parameters from the ledger's point of view] says can be summarized in a table.
  163 We'll need to enumerate all possible combinations of
  164 
  165     old_node / new_node
  166     before_HF / after_HF
  167     shorter_list / exact_list / longer_list
  168 
  169 where the last ones are about the list of cost model parameters supplied by the ledger, e.g.
  170 @shorter_list@ means that the ledger supplied a list of cost model parameters that is shorter than
  171 expected by the appropriate Plutus ledger language at the specific protocol version.
  172 
  173 Note that the exact same list may be @exact_list@ for @old_node@ and @shorter_list@ for @new_node@.
  174 And the exact same list may be @longer_list@ for @old_node@ and @exact_list@ for @new_node@.
  175 
  176 We'll also need to consider changing the cost model parameters of old builtins and adding cost model
  177 parameters for new builtins separately, because there are non-trivial differences. For old builtins
  178 we'll only consider the case of adding new cost model parameters, because the case of reducing the
  179 number of cost model parameters is trivial: the only thing we can do about it is simply keeping the
  180 redundant arguments and asking the ledger to continue providing them, as there's no way for us to
  181 remove them form the middle of the cost model parameter list. And if the number of parameters
  182 doesn't change, then there's nothing to worry about.
  183 
  184 We'll also only consider the case when the Plutus ledger language stays the same, because if it
  185 changes, then everything becomes trivial: we don't need to worry about any backward compatibility
  186 and can simply "start over" for a new ledger language.
  187 
  188 "OK" in the table below means "we handle this situation correctly as implied by
  189 Note [Cost model parameters from the ledger's point of view]".
  190 
  191 The table then looks like this:
  192 
  193                                          old_builtin new_builtin
  194     old_node / before_HF / shorter_list    OK [1]      OK [1]
  195     old_node / before_HF / exact_list      OK [2]      OK [3]
  196     old_node / before_HF / longer_list     OK [4]      OK [3]
  197 
  198     old_node / after_HF / shorter_list     OK [5]      OK [5]
  199     old_node / after_HF / exact_list       OK [5]      OK [5]
  200     old_node / after_HF / longer_list      OK [5]      OK [5]
  201 
  202     new_node / before_HF / shorter_list    OK [6]      OK [3]
  203     new_node / before_HF / exact_list      OK [2]      OK [3]
  204     new_node / before_HF / longer_list     OK [7]      OK [7]
  205 
  206     new_node / after_HF / shorter_list     OK [8]      OK [9]
  207     new_node / after_HF / exact_list       OK [2]      OK [2]
  208     new_node / after_HF / longer_list      OK [10]     OK [10]
  209 
  210 [1] this is the same case as `new_node / after_HF / shorter_list`, just in the context of a different hardfork
  211 [2] the most straightforward case, nothing to consider
  212 [3] a new builtin cannot be used before the hardfork as `scriptCBORDecoder` will fail with "built-in function X is not available"
  213 [4] an old builtin does not touch its new cost model parameters (if any) before the hardfork, hence it's safe to ignore those extra parameters
  214 [5] the old node is not alive after the hardfork, hence anything is OK
  215 [6] an old builtin does not touch its new cost model parameters (if any) before the hardfork, hence it's safe to make those extra parameters all `maxBound`
  216 [7] this is a bit weird, but theoretically possible (one can propose protocol parameter updates, including updating the number of PlutusV3 cost model parameters to 100000, or updating the max block size to 0. Those will be rejected, especially if there's a guardrail in place). But even it happens in practice, `longer_list` should never be a problem, since the extra numbers are simply ignored
  217 [8] this is a tricky one: since it's a shorter list, the missing cost model parameters all become 'maxBound' meaning for an old builtin we end up in a situation where some of its cost model parameters are 'maxBound' and some are not, which is not enough to prevent the builtin from running. Which is however not a problem, because it _is_ fine to run the builtin after the hardfork, all we care about is that it's not undercosted and making some of the cost model parameters 'maxBound' ensures that it's not as it's literally the highest value that they can have
  218 [9] same as [8] except this does ensure that the builtin never runs, because all of its cost model parameters are 'maxBound', which makes the cost of the builtin higher than any reasonable budget
  219 [10] this is the same case as `old_node / before_HF / longer_list`, just in the context of a different hardfork
  220 -}

  222 {-| A raw representation of the ledger's cost model parameters.
  223 
  224 The associated keys/names to the parameter values are arbitrarily set by the plutus team; the ledger does not hold any such names.
  225 
  226 See Note [Cost model parameters] -}
  227 type CostModelParams = Map.Map Text.Text Int64
  228 
  229 -- See Note [Cost model parameters]
  230 -- | Extract the model parameters from a model.
  231 extractParams :: ToJSON a => a -> Maybe CostModelParams
  232 extractParams cm = case toJSON cm of
  233   Object o ->
  234     let
  235       flattened = objToHm $ flattenObject "-" o
  236       usingCostingIntegers = HM.mapMaybe (\case Number n -> Just $ ceiling n; _ -> Nothing) flattened
  237       -- \^ Only (the contents of) the "Just" values are retained in the output map.
  238       mapified = Map.fromList $ HM.toList usingCostingIntegers
  239      in
  240       Just mapified
  241   _ -> Nothing
  242 
  243 -- | A fatal error when trying to create a cost given some plain costmodel parameters.
  244 data CostModelApplyError
  245   = -- | a costmodel parameter with the give name does not exist in the costmodel to be applied upon
  246     CMUnknownParamError !Text.Text
  247   | -- | internal error when we are transforming the applyParams' input to json (should not happen)
  248     CMInternalReadError
  249   | -- | internal error when we are transforming the applied params from json with given jsonstring error (should not happen)
  250     CMInternalWriteError !String
  251   deriving stock (Eq, Show, Generic, Data)
  252   deriving anyclass (Exception, NFData, NoThunks)
  253 
  254 -- | A non-fatal warning when trying to create a cost given some plain costmodel parameters.
  255 data CostModelApplyWarn
  256   = -- | See Note [Cost model parameters from the ledger's point of view]
  257     CMTooManyParamsWarn {cmExpected :: !Int, cmActual :: !Int}
  258   | -- | See Note [Cost model parameters from the ledger's point of view]
  259     CMTooFewParamsWarn {cmExpected :: !Int, cmActual :: !Int}
  260 
  261 instance Pretty CostModelApplyError where
  262   pretty =
  263     (preamble <+>) . \case
  264       CMUnknownParamError k -> "No such parameter in target cost model:" <+> pretty k
  265       CMInternalReadError -> "Internal problem occurred upon reading the given cost model parameters"
  266       CMInternalWriteError str -> "Internal problem occurred upon generating the applied cost model parameters with JSON error:" <+> pretty str
  267     where
  268       preamble = "applyParams error:"
  269 
  270 instance Pretty CostModelApplyWarn where
  271   pretty =
  272     (preamble <+>) . \case
  273       CMTooManyParamsWarn {..} -> "Too many cost model parameters passed, expected" <+> pretty cmExpected <+> "but got" <+> pretty cmActual
  274       CMTooFewParamsWarn {..} -> "Too few cost model parameters passed, expected" <+> pretty cmExpected <+> "but got" <+> pretty cmActual
  275     where
  276       preamble = "applyParams warn:"

  278 -- See Note [Cost model parameters]
  279 -- | Update a model by overwriting the parameters with the given ones.
  280 applyParams
  281   :: (FromJSON a, ToJSON a, MonadError CostModelApplyError m)
  282   => a
  283   -> CostModelParams
  284   -> m a
  285 applyParams cm params = case toJSON cm of
  286   Object o ->
  287     let
  288       usingScientific = fmap (Number . fromIntegral) params
  289       flattened = fromHash $ objToHm $ flattenObject "-" o
  290      in
  291       do
  292         -- this is where the overwriting happens
  293         -- fail when key is in params (left) but not in the model (right)
  294         merged <- Map.mergeA failMissing Map.preserveMissing (Map.zipWithMatched leftBiased) usingScientific flattened
  295         let unflattened = unflattenObject "-" $ hmToObj $ toHash merged
  296         case fromJSON (Object unflattened) of
  297           Success a -> pure a
  298           Error str -> throwError $ CMInternalWriteError str
  299   _ -> throwError CMInternalReadError
  300   where
  301     toHash = HM.fromList . Map.toList
  302     fromHash = Map.fromList . HM.toList
  303     -- fail when field missing
  304     failMissing = Map.traverseMissing $ \k _v -> throwError $ CMUnknownParamError k
  305     -- left-biased merging when key found in both maps
  306     leftBiased _k l _r = l

  308 -- | Parameters for a machine step model and a builtin evaluation model bundled together.
  309 data SplitCostModelParams
  310   = SplitCostModelParams
  311   { _machineParams :: CostModelParams
  312   , _builtinParams :: CostModelParams
  313   }
  314 
  315 {-| Split a CostModelParams object into two subobjects according to some prefix:
  316 see item 5 of Note [Cost model parameters]. -}
  317 splitParams :: Text.Text -> CostModelParams -> SplitCostModelParams
  318 splitParams prefix params =
  319   let (machineparams, builtinparams) = Map.partitionWithKey (\k _ -> Text.isPrefixOf prefix k) params
  320    in SplitCostModelParams machineparams builtinparams

  322 -- | Given a CostModel, produce a single map containing the parameters from both components
  323 extractCostModelParams
  324   :: (ToJSON machinecosts, ToJSON builtincosts)
  325   => CostModel machinecosts builtincosts -> Maybe CostModelParams
  326 extractCostModelParams model =
  327   -- this is using the applicative instance of Maybe
  328   Map.union <$> extractParams (_machineCostModel model) <*> extractParams (_builtinCostModel model)

  330 {-| Given a set of cost model parameters, split it into two parts according to
  331 some prefix and use those parts to update the components of a cost model. -}
  332 
  333 {- Strictly we don't need to do the splitting: when we call fromJSON in
  334    applyParams any superfluous objects in the map being decoded will be
  335    discarded, so we could update both components of the cost model with the
  336    entire set of parameters without having to worry about splitting the
  337    parameters on a prefix of the key.  This relies on what appears to be an
  338    undocumented implementation choice in Aeson though (other JSON decoders (for
  339    other languages) seem to vary in how unknown fields are handled), so let's be
  340    explicit. -}
  341 applySplitCostModelParams
  342   :: (FromJSON evaluatorcosts, FromJSON builtincosts, ToJSON evaluatorcosts, ToJSON builtincosts, MonadError CostModelApplyError m)
  343   => Text.Text
  344   -> CostModel evaluatorcosts builtincosts
  345   -> CostModelParams
  346   -> m (CostModel evaluatorcosts builtincosts)
  347 applySplitCostModelParams prefix model params =
  348   let SplitCostModelParams machineparams builtinparams = splitParams prefix params
  349    in CostModel
  350         <$> applyParams (_machineCostModel model) machineparams
  351         <*> applyParams (_builtinCostModel model) builtinparams
