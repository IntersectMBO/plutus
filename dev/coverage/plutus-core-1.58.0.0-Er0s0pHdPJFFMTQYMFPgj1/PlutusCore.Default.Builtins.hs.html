  236 {-| Turn a function into another function that 'fail's when its second argument is @0@ or calls the
  237 original function otherwise and wraps the result in 'pure'. Useful for correctly handling `div`,
  238 `mod`, etc. -}
  239 nonZeroSecondArg
  240   :: (Integer -> Integer -> Integer) -> Integer -> Integer -> BuiltinResult Integer
  241 -- If we match against @IS 0#@ instead of @0@, GHC will generate tidier Core for some reason. It
  242 -- probably doesn't really matter performance-wise, but would be easier to read. We don't do it out
  243 -- of paranoia and because it requires importing the 'IS' constructor, which is in different
  244 -- packages depending on the GHC version, so requires a bunch of irritating CPP.
  245 --
  246 -- We could also replace 'div' with 'integerDiv' (and do the same for other division builtins) at
  247 -- the call site of this function in order to avoid double matching against @0@, but that also
  248 -- requires CPP. Perhaps we can afford one additional pattern match for division builtins for the
  249 -- time being, since those aren't particularly fast anyway.
  250 --
  251 -- The bang is to communicate to GHC that the function is strict in both the arguments just in case
  252 -- it'd want to allocate a thunk for the first argument otherwise.
  253 nonZeroSecondArg _ !_ 0 =
  254   -- See Note [Structural vs operational errors within builtins].
  255   fail "Cannot divide by zero"
  256 nonZeroSecondArg f x y = pure $ f x y
  257 {-# INLINE nonZeroSecondArg #-}

  265 {- Note [Constants vs built-in functions]
  266 A constant is any value of a built-in type. For example, 'Integer' is a built-in type, so anything
  267 of type 'Integer' is a constant.
  268 
  269 On the contrary a built-in function can't be of a built-in type, because the type of a built-in
  270 function is always of either the @all a. b@ form or the @a -> b@ one, none of which is a built-in
  271 type. This is checked by the machinery, so if the user tries to add a built-in function that is not
  272 of one of these forms, they'll get a nice custom type error.
  273 
  274 A built-in function is associated with its Haskell implementation: there can be many built-in
  275 functions of the same type, all doing different things, and there can be infinitely more _definable_
  276 built-in functions of the same type that are not built-in functions nonetheless, because we didn't
  277 register them as such by providing a Haskell implementation for each of them. This is the difference
  278 between constants and built-in functions: the set of constants (infinite in our case) depends solely
  279 on the set of available built-in types (also infinite in our case, because we have @Integer@,
  280 @[Integer]@, @[[Integer]]@ etc), while the set of built-in functions is defined by explicitly
  281 assigning each member a specific name and an associated with it Haskell implementation. It is
  282 theoretically possible to have an infinite set of built-in functions, but we neither do that nor
  283 need it, hence our set of built-in functions is finite.
  284 
  285 The rule of thumb is: constants are raw data and built-in functions are, well, functions.
  286 
  287 @(:)@ works as follows: it takes two constants wrapped as values, extracts an integer from the first
  288 constant and a list of integers from the second one, prepends the former to the latter and wraps the
  289 resulting list back into a constant, which gets wrapped into a value.
  290 
  291 Why does @(:)@ have to be a built-in function? Because its type is
  292 
  293     all a. a -> list a -> list a
  294 
  295 and if we tried to make @(:)@ a constant we'd have to somehow make this type a built-in type and
  296 promise that every value (i.e. every definable function) of this type can be used as a Plutus term,
  297 which doesn't make any sense. Only the particular Haskell implementation that prepends an element to
  298 a list is what we're interested in.
  299 
  300 Why may @[]@ not be a built-in function? If its type is hardcoded to @[Integer]@, then that's a
  301 built-in type and we know that anything of a built-in type can be embedded into a term as a
  302 constant. I.e. @[] :: [Integer]@ is perfectly fine as a constant and does not need to be a built-in
  303 function.
  304 
  305 Why may @[]@ be a built-in function? If it's polymorphic over the type of the elements, then its
  306 Plutus Core type is @all a. list a@ and that is not a built-in type, hence we have to make that a
  307 built-in function.
  308 -}

  310 {- Note [How to add a built-in function: simple cases]
  311 This Note explains how to add a built-in function and how to read definitions of existing built-in
  312 functions. It does not attempt to explain why things the way they are, that is explained in comments
  313 in relevant modules, check out the following for an overview of the module structure:
  314 https://github.com/IntersectMBO/plutus/blob/97c2b2c6975e41ce25ee5efa1dff0f1bd891a589/plutus-core/docs/BuiltinsOverview.md
  315 
  316 In order to add a new built-in function one needs to add a constructor to 'DefaultFun' and handle
  317 it within the @ToBuiltinMeaning uni DefaultFun@ instance. The general pattern is
  318 
  319     toBuiltinMeaning semvar <BuiltinName> =
  320         let <builtinNameDenotation> :: BS.ByteString -> BS.ByteString
  321             <builtinNameDenotation> = <denotation>
  322             {-# INLINE <builtinNameDenotation> #-}
  323         in makeBuiltinMeaning
  324             <builtinNameDenotation>
  325             <costingFunction>
  326 
  327 Here's a specific example:
  328 
  329     toBuiltinMeaning _ AddInteger =
  330         let addIntegerDenotation :: Integer -> Integer -> Integer
  331             addIntegerDenotation = (+)
  332             {-# INLINE addIntegerDenotation #-}
  333         in makeBuiltinMeaning
  334             addIntegerDenotation
  335             (runCostingFunTwoArguments . paramAddInteger)
  336 
  337 'makeBuiltinMeaning' creates a Plutus builtin out of its denotation (i.e. Haskell implementation)
  338 and a costing function for it. Once a builtin is added, its Plutus type is kind-checked and printed
  339 to a golden file automatically (consult @git status@). 'toBuiltinMeaning' also takes a
  340 'BuiltinSemanticsVariant' argument which allows a particular builtin name to have multiple
  341 associated denotations (see Note [Builtin semantics variants]), but for simplicity we assume in the
  342 examples below that all builtins have only one variant rendering the @semvar@ argument irrelevant.
  343 
  344 See Note [Builtin semantics variants] for how @semvar@ enables us to customize the behavior of a
  345 built-in function. For the purpose of these docs we're going to ignore that and use @_@ instead of
  346 @semvar@.
  347 
  348 Note that it's very important for the denotation to have an explicit type signature for several
  349 reasons:
  350 
  351 1. makes it easier to review the code and make sure it makes sense
  352 2. makes it easier to search for builtins associated with certain types -- just @grep@ for the type
  353 3. most importantly, if we let GHC infer the types, there's a small but very real chance that
  354    updating a library to a newer version will change the type of some definition used within the
  355    denotation of a builtin and that may get reflected in the type signature of the builtin without
  356    us noticing, since the builtins machinery will gladly swallow the change. And since the type
  357    signature of a builtin determines its behavior via ad hoc polymorphism a change in the type
  358    signature can cause a sudden hardfork, which would be very bad
  359 
  360 hence we specify the type signature for the denotation of each builtin explicitly and always create
  361 a @let@ binding for consistency. We add an @INLINE@ pragma to the @let@ binding to make sure that
  362 the binding doesn't get in the way of performance.
  363 
  364 Below we will enumerate what kind of denotations are accepted by 'makeBuiltinMeaning' without
  365 touching any costing stuff.
  366 
  367 1. The simplest example of an accepted denotation is a monomorphic function that takes values of
  368 built-in types and returns a value of a built-in type as well. For example
  369 
  370     encodeUtf8 :: Text -> BS.ByteString
  371 
  372 You can feed 'encodeUtf8' directly to 'makeBuiltinMeaning' without specifying any types:
  373 
  374     toBuiltinMeaning _ EncodeUtf8 =
  375         let encodeUtf8Denotation :: Text -> BS.ByteString
  376             encodeUtf8Denotation = encodeUtf8
  377             {-# INLINE encodeUtf8Denotation #-}
  378         in makeBuiltinMeaning
  379             encodeUtf8Denotation
  380             <costingFunction>
  381 
  382 This will add the builtin, the only two things that remain are implementing costing for this
  383 builtin (out of the scope of this Note) and handling it within the @Flat DefaultFun@ instance
  384 (see Note [Stable encoding of TPLC]).
  385 
  386 2. Unconstrained type variables are fine, you don't need to instantiate them. For example
  387 
  388     toBuiltinMeaning _ IfThenElse =
  389         let ifThenElseDenotation :: Bool -> a -> a -> a
  390             ifThenElseDenotation b x y = if b then x else y
  391             {-# INLINE ifThenElseDenotation #-}
  392         in makeBuiltinMeaning
  393             ifThenElseDenotation
  394             <costingFunction>
  395 
  396 works alright. The Haskell type of the denotation is
  397 
  398     forall a. Bool -> a -> a -> a
  399 
  400 whose counterpart in Plutus is
  401 
  402     all a. bool -> a -> a -> a
  403 
  404 and unsurprisingly it's the exact Plutus type of the added builtin.
  405 
  406 It may seem like getting the latter from the former is entirely trivial, however
  407 'makeBuiltinMeaning' jumps through quite a few hoops to achieve that and below we'll consider those
  408 of them that are important to know to be able to use 'makeBuiltinMeaning' in cases that are more
  409 complicated than a simple monomorphic or polymorphic function. But for now let's talk about a few
  410 more simple cases.
  411 
  412 3. Certain types are not built-in, but can be represented via built-in ones. For example, we don't
  413 have 'Int' built-in, but we have 'Integer' and we can represent the former in terms of the
  414 latter. The conversions between the two types are handled by 'makeBuiltinMeaning', so that the user
  415 doesn't need to write them themselves and can just write
  416 
  417     toBuiltinMeaning _ LengthOfByteString =
  418         let lengthOfByteStringDenotation :: BS.ByteString -> Int
  419             lengthOfByteStringDenotation = BS.length
  420             {-# INLINE lengthOfByteStringDenotation #-}
  421         in makeBuiltinMeaning
  422             lengthOfByteStringDenotation
  423             <costingFunction>
  424 
  425 directly (where @BS.length :: BS.ByteString -> Int@).
  426 
  427 Note however that while it's always safe to convert an 'Int' to an 'Integer', doing the opposite is
  428 not safe in general, because an 'Integer' may not fit into the range of 'Int'. For this reason
  429 
  430     YOU MUST NEVER USE 'fromIntegral' AND SIMILAR FUNCTIONS THAT CAN SILENTLY UNDER- OR OVERFLOW
  431     WHEN DEFINING A BUILT-IN FUNCTION
  432 
  433 For example defining a builtin that takes an 'Integer' and converts it to an 'Int' using
  434 'fromIntegral' is not allowed under any circumstances and can be a huge vulnerability.
  435 
  436 It's completely fine to define a builtin that takes an 'Int' directly, though. How so? That's due
  437 to the fact that the builtin application machinery checks that an 'Integer' is in the bounds of
  438 'Int' before doing the conversion. If the bounds check succeeds, then the 'Integer' gets converted
  439 to the corresponding 'Int', and if it doesn't, then the builtin application fails.
  440 
  441 For the list of types that can be converted to/from built-in ones look into the file with the
  442 default universe. If you need to add a new such type, just copy-paste what's done for an existing
  443 one and adjust.
  444 
  445 Speaking of builtin application failing:
  446 
  447 4. A built-in function can fail. Whenever a builtin fails, evaluation of the whole program fails.
  448 There's a number of ways a builtin can fail:
  449 
  450 - as we've just seen a type conversion can fail due to an unsuccessful bounds check
  451 - if the builtin expects, say, a 'Text' argument, but gets fed an 'Integer' argument
  452 - if the builtin expects any constant, but gets fed a non-constant
  453 - if its denotation runs in the 'BuiltinResult' monad and an 'evaluationFailure' gets returned
  454 
  455 Most of these are not a concern to the user defining a built-in function (conversions are handled
  456 within the builtin application machinery, type mismatches are on the type checker and the person
  457 writing the program etc), however explicitly returning 'evaluationFailure' from a builtin is
  458 something that happens commonly.
  459 
  460 One simple example is a monomorphic function matching on a certain constructor and failing in all
  461 other cases:
  462 
  463     toBuiltinMeaning _ UnIData =
  464         let unIDataDenotation :: Data -> BuiltinResult Integer
  465             unIDataDenotation = \case
  466                 I i -> pure i
  467                 _   -> evaluationFailure
  468             {-# INLINE unIDataDenotation #-}
  469         in makeBuiltinMeaning
  470             unIDataDenotation
  471             <costingFunction>
  472 
  473 The type of the denotation is
  474 
  475     Data -> BuiltinResult Integer
  476 
  477 and the Plutus type of the builtin is
  478 
  479     data -> integer
  480 
  481 because the error effect is implicit in Plutus.
  482 
  483 Returning @BuiltinResult a@ for a type variable @a@ is also fine, i.e. it doesn't matter whether
  484 the denotation is monomorphic or polymorphic w.r.t. failing.
  485 
  486 But note that
  487 
  488     'BuiltinResult' MUST BE EXPLICITLY USED FOR ANY FAILING BUILTIN AND THROWING AN EXCEPTION
  489     VIA 'error' OR 'throw' OR ELSE IS NOT ALLOWED AND CAN BE A HUGE VULNERABILITY. MAKE SURE THAT
  490     NONE OF THE FUNCTIONS THAT YOU USE TO DEFINE A BUILTIN THROW EXCEPTIONS
  491 
  492 An argument of a builtin can't have 'BuiltinResult' in its type -- only the result.
  493 
  494 5. A builtin can emit log messages. For that its denotation needs to run in the 'BuiltinResult' as
  495 in case of failing. The ergonomics are the same. For example:
  496 
  497     toBuiltinMeaning _ Trace =
  498         let traceDenotation :: Text -> a -> BuiltinResult a
  499             traceDenotation text a = a <$ emit text
  500             {-# INLINE traceDenotation #-}
  501         in makeBuiltinMeaning
  502             traceDenotation
  503             <costingFunction>
  504 
  505 The type of the denotation is
  506 
  507     forall a. Text -> a -> Builtin a
  508 
  509 and the Plutus type of the builtin is
  510 
  511     all a. text -> a -> a
  512 
  513 because just like with the error effect, whether a function logs anything or not is not reflected
  514 in its type.
  515 
  516 This concludes the list of simple cases. Before we jump to the hard ones, we need to talk about how
  517 polymorphism gets elaborated, so read Note [Elaboration of polymorphism] next.
  518 -}

  520 {- Note [Elaboration of polymorphism]
  521 In Note [How to add a built-in function: simple cases] we defined the following builtin:
  522 
  523     toBuiltinMeaning _ IfThenElse =
  524         let ifThenElseDenotation :: Bool -> a -> a -> a
  525             ifThenElseDenotation b x y = if b then x else y
  526             {-# INLINE ifThenElseDenotation #-}
  527         in makeBuiltinMeaning
  528             ifThenElseDenotation
  529             <costingFunction>
  530 
  531 whose Haskell type is
  532 
  533     forall a. Bool -> a -> a -> a
  534 
  535 The way 'makeBuiltinMeaning' handles such a type is by traversing it and instantiating every type
  536 variable. What a type variable gets instantiated to depends on where it appears. When the entire
  537 type of an argument is a single type variable, it gets instantiated to @Opaque val VarN@ where
  538 @VarN@ is pseudocode for "a Haskell type representing a Plutus type variable with 'Unique' N"
  539 For the purpose of this explanation it doesn't matter what @VarN@ actually is and the representation
  540 is subject to change anyway (see Note [Implementation of polymorphic built-in functions] if you want
  541 to know the details). 'Opaque' however is more fundamental and so we need to talk about it.
  542 Here's how it's defined:
  543 
  544     newtype Opaque val (rep :: GHC.Type) = Opaque
  545         { unOpaque :: val
  546         }
  547 
  548 I.e. @Opaque val rep@ is a wrapper around @val@, which stands for the type of value that an
  549 evaluator uses (the builtins machinery is designed to work with any evaluator and different
  550 evaluators define their type of values differently, for example 'CkValue' if the type of value for
  551 the CK machine). The idea is simple: in order to apply the denotation of a builtin expecting, say,
  552 an 'Integer' constant we need to actually extract that 'Integer' from the AST of the given value,
  553 but if the denotation is polymorphic over the type of its argument, then we don't need to extract
  554 anything, we can just pass the AST of the value directly to the denotation (which means the value
  555 doesn't have to be a 'Constant', it can be completely arbitrary). I.e. in order for a polymorphic
  556 function to become a monomorphic denotation (denotations are always monomorpic) all type variables
  557 in the type of that function need to be instantiated at the type of value that a given evaluator
  558 uses.
  559 
  560 If we used just @val@ rather than @Opaque val rep@, we'd specialize
  561 
  562     forall a. Bool -> a -> a -> a
  563 
  564 to
  565 
  566     Bool -> val -> val -> val
  567 
  568 however then we'd need to separately specify the Plutus type of this builtin, since we can't infer
  569 it from all these @val@s in the general case, for example does
  570 
  571     val -> val -> val
  572 
  573 stand for
  574 
  575     all a. a -> a -> a
  576 
  577 or
  578 
  579     all a b. a -> b -> a
  580 
  581 or something else?
  582 
  583 So we use the @Opaque val rep@ wrapper, which is basically a @val@ with a @rep@ attached to it where
  584 @rep@ represents the Plutus type of the argument/result, which is how we arrive at
  585 
  586     Bool -> Opaque val Var0 -> Opaque val Var0 -> Opaque val Var0
  587 
  588 This encoding allows us to specify both the Haskell and the Plutus types of the builtin
  589 simultaneously.
  590 
  591 If we wanted to we could add explicit 'Opaque' while still having explicit polymorphism (leaving out
  592 the @Var0@ thing for the elaboration machinery to figure out):
  593 
  594     toBuiltinMeaning _ IfThenElse =
  595         let ifThenElseDenotation :: Bool -> Opaque val a -> Opaque val a -> Opaque val a
  596             ifThenElseDenotation b x y = if b then x else y
  597             {-# INLINE ifThenElseDenotation #-}
  598         in makeBuiltinMeaning
  599             ifThenElseDenotation
  600             <costingFunction>
  601 
  602 and it would be equivalent to the original definition, but note how @a@ is now an argument to
  603 'Opaque' rather than the entire type of an argument. In order for this definition to elaborate to
  604 the same type as before @a@ needs to be instantiated to just @Var0@, as opposed to @Opaque val
  605 Var0@, because the 'Opaque' part is already there, so this is what the elaboration machinery does.
  606 
  607 So regardless of which method of defining 'IfThenElse' we choose, the type of its denotation gets
  608 elaborated to the same
  609 
  610     Bool -> Opaque val Var0 -> Opaque val Var0 -> Opaque val Var0
  611 
  612 which then gets digested, so that we can compute what Plutus type it corresponds to. The procedure
  613 is simple: collect all distinct type variables, @all@-bind them and replace the usages with the
  614 bound variables. This turns the type above into
  615 
  616     all a. bool -> a -> a -> a
  617 
  618 which is the Plutus type of the 'IfThenElse' builtin.
  619 
  620 It's of course allowed to have multiple type variables, e.g. in the following snippet:
  621 
  622     toBuiltinMeaning _ Const =
  623         let constDenotation :: a -> b -> a
  624             constDenotation = Prelude.const
  625             {-# INLINE constDenotation #-}
  626         in makeBuiltinMeaning
  627             constDenotation
  628             <costingFunction>
  629 
  630 the Haskell type of 'const' is
  631 
  632     forall a b. a -> b -> a
  633 
  634 which the elaboration machinery turns into
  635 
  636     Opaque val Var0 -> Opaque val Var1 -> Opaque val Var0
  637 
  638 The elaboration machinery respects the explicitly specified parts of the type and does not attempt
  639 to argue with them. For example if the user insisted that the instantiated type of 'const' had
  640 @Var0@ and @Var1@ swapped:
  641 
  642     Opaque val Var1 -> Opaque val Var0 -> Opaque val Var1
  643 
  644 the elaboration machinery wouldn't make a fuss about that.
  645 
  646 As a final simple example, consider
  647 
  648     toBuiltinMeaning _ Trace =
  649         let traceDenotation :: Text -> a -> BuiltinResult a
  650             traceDenotation text a = a <$ emit text
  651             {-# INLINE traceDenotation #-}
  652         in makeBuiltinMeaning
  653             traceDenotation
  654             <costingFunction>
  655 
  656 from [How to add a built-in function: simple cases]. The type of the denotation is
  657 
  658     forall a. Text -> a -> BuiltinResult a
  659 
  660 which elaborates to
  661 
  662     Text -> Opaque val Var0 -> BuiltinResult (Opaque val Var0)
  663 
  664 Elaboration machinery is able to look under 'BuiltinResult' even if there's a type variable inside
  665 that does not appear anywhere else in the type signature, for example the type of the denotation in
  666 
  667     toBuiltinMeaning _ ErrorPrime =
  668         let errorPrimeDenotation :: BuiltinResult a
  669             errorPrimeDenotation = evaluationFailure
  670             {-# INLINE errorPrimeDenotation #-}
  671         in makeBuiltinMeaning
  672             errorPrimeDenotation
  673             <costingFunction>
  674 
  675 is
  676 
  677     forall a. BuiltinResult a
  678 
  679 which gets elaborated to
  680 
  681     BuiltinResult (Opaque val Var0)
  682 
  683 from which the final Plutus type of the builtin is computed:
  684 
  685     all a. a
  686 
  687 Read Note [How to add a built-in function: complicated cases] next.
  688 -}

  690 {- Note [How to add a built-in function: complicated cases]
  691 Now let's talk about more complicated built-in functions.
  692 
  693 1. In Note [Elaboration of polymorphism] we saw how a Haskell type variable gets elaborated to an
  694 @Opaque val VarN@ and we learned that this type can be used directly as opposed to being inferred.
  695 However there exist more ways to use 'Opaque' explicitly. Here's a simple example:
  696 
  697     toBuiltinMeaning _ IdAssumeBool =
  698         let idAssumeBoolDenotation :: Opaque val Bool -> Opaque val Bool
  699             idAssumeBoolDenotation = Prelude.id
  700             {-# INLINE idAssumeBoolDenotation #-}
  701         in makeBuiltinMeaning
  702             idAssumeBoolDenotation
  703             <costingFunction>
  704 
  705 This creates a built-in function whose Plutus type is
  706 
  707     id : bool -> bool
  708 
  709 i.e. the Plutus type signature of the built-in function is the same as with
  710 
  711     toBuiltinMeaning _ IdBool =
  712         let idBoolDenotation :: Bool -> Bool
  713             idBoolDenotation = Prelude.id
  714             {-# INLINE idBoolDenotation #-}
  715         in makeBuiltinMeaning
  716             idBoolDenotation
  717             <costingFunction>
  718 
  719 but the two evaluate differently: the former takes a value and returns it right away while the
  720 latter takes a value, extracts a 'Bool' constant out of it and then lifts that constant back into
  721 @val@. The difference is not only in performance (obviously returning something right away is
  722 cheaper than unlifting-then-lifting-back), but also in semantics: the former returns its argument
  723 during evaluation regardless of what that argument is, so if someone generates Untyped Plutus Core
  724 directly, they can apply @IdAssumeBool@ to a term that doesn't evaluate to a 'Bool' constant or
  725 even a constant at all and that won't be a runtime error, while the latter has to be applied to
  726 a term evaluating to a 'Bool' constant in order not to fail at runtime.
  727 
  728 2. @val@ in @Opaque val rep@ is not completely arbitrary, it has to implement 'HasConstant', which
  729 makes it possible to unlift @val@ as a constant or lift a constant back into @val@. There's a
  730 'HasConstant' instance for @Opaque val rep@ whenever there's one for @val@, so if we, for some
  731 reason, wanted to have 'Opaque' in the type signature of the denotation, but still unlift the
  732 argument as a 'Bool', we could do that:
  733 
  734     toBuiltinMeaning _ IdAssumeCheckBool =
  735         let idAssumeCheckBoolDenotation :: Opaque val Bool -> BuiltinResult Bool
  736             idAssumeCheckBoolDenotation val = asConstant val of
  737                 Right (Some (ValueOf DefaultUniBool b)) -> pure b
  738                 _                                       -> evaluationFailure
  739             {-# INLINE idAssumeCheckBoolDenotation #-}
  740         in makeBuiltinMeaning
  741             idAssumeCheckBoolDenotation
  742             <costingFunction>
  743 
  744 Here in the denotation we unlift the given value as a constant, check that its type tag is
  745 'DefaultUniBool' and return the unlifted 'Bool'. If any of that fails, we return an explicit
  746 'evaluationFailure'.
  747 
  748 This achieves almost the same as 'IdBool', which keeps all the bookkeeping behind the scenes, but
  749 there is a minor difference: in case of error its message is ignored. It would be easy to allow for
  750 returning an unlifting error from a builtin explicitly, but we don't need that for anything, hence
  751 it's not implemented.
  752 
  753 We call this style of manually calling 'asConstant' and matching on the type tag "manual unlifting".
  754 As opposed to "automatic unlifting" that we were using before where 'Bool' in the type of the
  755 denotation of a builtin causes the builtins machinery to convert the given argument to a 'Bool'
  756 constant automatically behind the scenes.
  757 
  758 3. There's a middle ground between automatic and manual unlifting to 'Bool', one can unlift a value
  759 automatically as a constant and then unlift the result manually to 'Bool' using the 'SomeConstant'
  760 wrapper:
  761 
  762     newtype SomeConstant uni (rep :: GHC.Type) = SomeConstant
  763         { unSomeConstant :: Some (ValueOf uni)
  764         }
  765 
  766 'SomeConstant' is similar to 'Opaque' in that it has a @rep@ representing a Plutus type.
  767 The difference is that 'Opaque' is a wrapper around an arbitrary value and 'SomeConstant' is a
  768 wrapper around a constant. 'SomeConstant' allows one to automatically unlift an argument of a
  769 built-in function as a constant with all 'asConstant' business kept behind the scenes, for example:
  770 
  771     toBuiltinMeaning _ IdSomeConstantBool =
  772         let idSomeConstantBoolDenotation :: SomeConstant uni Bool -> BuiltinResult Bool
  773             idSomeConstantBoolDenotation = \case
  774                 SomeConstant (Some (ValueOf DefaultUniBool b)) -> pure b
  775                 _                                              -> evaluationFailure
  776             {-# INLINE idSomeConstantBoolDenotation #-}
  777         in makeBuiltinMeaning
  778             idSomeConstantBoolDenotation
  779             <costingFunction>
  780 
  781 Note how we no longer call 'asConstant' manually, but still manually match on 'DefaultUniBool'.
  782 
  783 So there's a whole range of how "low-level" one can choose to be when defining a built-in function.
  784 However it's not always possible to use automatic unlifting, see next.
  785 
  786 4. If we try to define the following built-in function:
  787 
  788     toBuiltinMeaning _ NullList =
  789         let nullListDenotation :: [a] -> Bool
  790             nullListDenotation = null
  791             {-# INLINE nullListDenotation #-}
  792         in makeBuiltinMeaning
  793             nullListDenotation
  794             <costingFunction>
  795 
  796 we'll get an error, saying that a polymorphic built-in type can't be applied to a type variable.
  797 It's not impossible to make it work, see Note [Unlifting a term as a value of a built-in type], but
  798 not in the general case, plus it has to be very inefficient.
  799 
  800 Instead we have to use 'SomeConstant' to automatically unlift the argument as a constant and then
  801 check that the value inside of it is a list (by matching on the type tag):
  802 
  803     toBuiltinMeaning _ NullList =
  804         let nullListDenotation :: SomeConstant uni [a] -> BuiltinResult Bool
  805             nullListDenotation (SomeConstant (Some (ValueOf uniListA xs))) = do
  806                 case uniListA of
  807                     DefaultUniList _ -> pure $ null xs
  808                     _ -> throwError $ structuralUnliftingError "Expected a list but got something else"
  809             {-# INLINE nullListDenotation #-}
  810         in makeBuiltinMeaning
  811             nullListDenotation
  812             <costingFunction>
  813 
  814 As before, we have to match on the type tag, because there's no relation between @rep@ from
  815 @SomeConstant uni rep@ and the constant that the built-in function actually receives at runtime
  816 (someone could generate Untyped Plutus Core directly and apply 'nullPlc' to an 'Integer' or
  817 whatever). @rep@ is only for the Plutus type checker to look at, it doesn't influence evaluation
  818 in any way.
  819 
  820 Here's a similar built-in function:
  821 
  822     toBuiltinMeaning _ FstPair =
  823         let fstPairDenotation :: SomeConstant uni (a, b) -> BuiltinResult (Opaque val a)
  824             fstPairDenotation (SomeConstant (Some (ValueOf uniPairAB xy))) = do
  825                 case uniPairAB of
  826                     DefaultUniPair uniA _ ->              -- [1]
  827                         pure . fromValueOf uniA $ fst xy  -- [2]
  828                     _ ->
  829                         throwError $ structuralUnliftingError "Expected a pair but got something else"
  830             {-# INLINE fstPairDenotation #-}
  831         in makeBuiltinMeaning
  832             fstPairDenotation
  833             <costingFunction>
  834 
  835 In this definition we extract the first element of a pair by checking that the given constant is
  836 indeed a pair [1] and lifting its first element into @val@ using the type tag for the first
  837 element [2] (extracted from the type tag for the whole pair constant [1]).
  838 
  839 Note that it's fine to mix automatic unlifting for polymorphism not related to built-in types and
  840 manual unlifting for arguments having non-monomorphized polymorphic built-in types, for example:
  841 
  842     toBuiltinMeaning _ ChooseList =
  843         let chooseListDenotation :: SomeConstant uni [a] -> b -> b -> BuiltinResult b
  844             chooseListDenotation (SomeConstant (Some (ValueOf uniListA xs))) a b = do
  845                 case uniListA of
  846                     DefaultUniList _ -> pure $ case xs of
  847                         []    -> a
  848                         _ : _ -> b
  849                     _ -> throwError $ structuralUnliftingError "Expected a list but got something else"
  850             {-# INLINE chooseListDenotation #-}
  851         in makeBuiltinMeaning
  852             chooseListDenotation
  853             (runCostingFunThreeArguments . paramChooseList)
  854             <costingFunction>
  855 
  856 Here @a@ appears inside @[]@, which is a polymorphic built-in type, and so we have to use
  857 'SomeConstant' and check that the given constant is indeed a list, while @b@ doesn't appear inside
  858 of any built-in type and so we don't need to instantiate it to 'Opaque' manually, the elaboration
  859 machinery will do it for us.
  860 
  861 Our final example is this:
  862 
  863     toBuiltinMeaning _ MkCons =
  864         let mkConsDenotation
  865                 :: SomeConstant uni a -> SomeConstant uni [a] -> BuiltinResult (Opaque val [a])
  866             mkConsDenotation
  867               (SomeConstant (Some (ValueOf uniA x)))
  868               (SomeConstant (Some (ValueOf uniListA xs))) =
  869                 case uniListA of
  870                     DefaultUniList uniA' -> case uniA `geq` uniA' of       -- [1]
  871                         Just Refl ->                                       -- [2]
  872                             pure . fromValueOf uniListA $ x : xs           -- [3]
  873                         Nothing -> throwError $ structuralUnliftingError
  874                             "The type of the value does not match the type of elements in the list"
  875                     _ -> throwError $ structuralUnliftingError "Expected a list but got something else"
  876             {-# INLINE mkConsDenotation #-}
  877         in makeBuiltinMeaning
  878             mkConsDenotation
  879             <costingFunction>
  880 
  881 Here we prepend an element to a list [3] after checking that the second argument is indeed a
  882 list [1] and that the type tag of the element being prepended equals the type tag for elements of
  883 the list [2] (extracted from the type tag for the whole list constant [1]).
  884 -}

  886 {- Note [Builtins and Plutus type checking]
  887 There's a direct correspondence between the Haskell type of the denotation of a builtin and the
  888 Plutus type of the builtin:
  889 
  890 1. elaboration turns a Haskell type variable into a concrete Haskell type representing a Plutus type
  891    variable, which later becomes demoted (in the regular @singletons@ sense via 'KnownSymbol' etc)
  892    to a regular Haskell value representing a Plutus type variable (as a part of the AST)
  893 2. a builtin head (i.e. a completely uninstantiated built-in type such as @Bool@ and @[]@) is
  894    considered abstract by the Plutus type checker. All the type checker cares about is being able to
  895    get the (Plutus) kind of a builtin head and check two builtin heads for equality
  896 3. Plutus type normalization tears partially or fully instantiated built-in types (such as
  897    @[Integer]@) apart and creates a Plutus type application for each Haskell type application
  898 4. 'BuiltinResult' does not appear on the Plutus side, since the logging and failure effects are
  899    implicit in Plutus as was discussed above
  900 5. 'Opaque' and 'SomeConstant' both carry a Haskell @rep@ type argument representing some Plutus
  901    type to be used for Plutus type checking
  902 
  903 This last part means that one can attach any (legal) @rep@ to an 'Opaque' or 'SomeConstant' and
  904 it'll be used by the Plutus type checker completely regardless of what the built-in function
  905 actually does. Let's look at some examples.
  906 
  907 1. The following built-in function unlifts to 'Bool' and lifts the result back:
  908 
  909     toBuiltinMeaning _ IdIntegerAsBool =
  910         let idIntegerAsBoolDenotation
  911                 :: SomeConstant uni Integer -> BuiltinResult (SomeConstant uni Integer)
  912             idIntegerAsBoolDenotation = \case
  913                 con@(SomeConstant (Some (ValueOf DefaultUniBool _))) -> pure con
  914                 _                                                    -> evaluationFailure
  915             {-# INLINE idIntegerAsBoolDenotation #-}
  916         in makeBuiltinMeaning
  917             idIntegerAsBoolDenotation
  918             <costingFunction>
  919 
  920 but on the Plutus side its type is
  921 
  922     integer -> integer
  923 
  924 because the @rep@ that 'SomeConstant' carries is 'Integer' in both the cases (in the type of the
  925 argument, as well as in the type of the result).
  926 
  927 This means that for this built-in function the Plutus type checker will accept a program that fails
  928 at runtime due to a type mismatch and will reject a program that runs successfully. Other built-in
  929 functions also can fail, e.g. the type of @ifThenElse@ says that the builtin expects a @Bool@ and
  930 feeding it something else will result in evaluation failure, but 'idIntegerAsBool' is different:
  931 it's respecting its type signature is what causes a failure, not disrespecting it.
  932 
  933 2. Another example of an unsafe built-in function is this one that checks whether an argument is a
  934 constant or not:
  935 
  936     toBuiltinMeaning _ IsConstant =
  937         let isConstantDenotation :: Opaque val a -> Bool
  938             isConstantDenotation = isRight . asConstant
  939             {-# INLINE isConstantDenotation #-}
  940         in makeBuiltinMeaning
  941             isConstantDenotation
  942             <costingFunction>
  943 
  944 Its type on the Plutus side is
  945 
  946     all a. a -> bool
  947 
  948 By parametricity any inhabitant of this type has to be either bottom or a function ignoring its
  949 argument, but @IsConstant@ actually uses the argument and so we break parametricity with this
  950 built-in function.
  951 
  952 3. Finally, we can have a Plutus version of @unsafeCoerce@:
  953 
  954     toBuiltinMeaning _ UnsafeCoerce =
  955         let unsafeCoerceDenotation :: Opaque val a -> Opaque val b
  956             unsafeCoerceDenotation = Opaque . unOpaque
  957             {-# INLINE unsafeCoerceDenotation #-}
  958         in makeBuiltinMeaning
  959             unsafeCoerceDenotation
  960             <costingFunction>
  961 
  962 Its type on the Plutus side is
  963 
  964     all a b. a -> b
  965 
  966 and thus this built-in function allows for viewing any Plutus expression as having an arbitrary
  967 type. Which is of course not nearly as bad as @unsafeCoerce@ in Haskell, because in Plutus a
  968 blob of memory representing an @Integer@ is not going to be viewed as a @[Bool]@ and an attempt to
  969 actually extract that @[Bool]@ will result in evaluation failure, but this built-in function is
  970 still not a good citizen of the Plutus type system.
  971 
  972 One could of course simply wrap Haskell's @unsafeCoerce@ as a built-in function in Plutus, but it
  973 goes without saying that this is not supposed to be done.
  974 
  975 So overall one needs to be very careful when defining built-in functions that have explicit
  976 'Opaque' and 'SomeConstant' arguments. Expressiveness doesn't come for free.
  977 -}

  979 {- Note [Representable built-in functions over polymorphic built-in types]
  980 Note [Legacy pattern matching on built-in types] discusses how general higher-order polymorphic
  981 built-in functions are troubling, but polymorphic built-in functions can be troubling even in
  982 the first-order case. In a Plutus program we always pair constants of built-in types with their
  983 tags from the universe, which means that in order to produce a constant embedded into a program
  984 we need the tag of the type of that constant. We can't get that tag from a Plutus type -- those
  985 are gone at runtime, so the only place we can get a type tag from during evaluation is some already
  986 existing constant. I.e. the following built-in function is representable:
  987 
  988     tail : all a. [a] -> [a]
  989 
  990 because for constructing the result we need a type tag for @[a]@, but we have a value of that type
  991 as an argument and so we can extract the type tag from it. Same applies to
  992 
  993     swap : all a b. (a, b) -> (b, a)
  994 
  995 since 'SomeConstantOf' always contains a type tag for each type that a polymorphic built-in type is
  996 instantiated with and so constructing a type tag for @(b, a)@ given type tags for @a@ and @b@ is
  997 unproblematic.
  998 
  999 And so neither
 1000 
 1001     cons : all a. a -> [a] -> [a]
 1002 
 1003 is troubling (even though that ones requires checking at runtime that the element to be prepended
 1004 is of the same type as the type of the elements of the list as it's impossible to enforce this kind
 1005 of type safety in Haskell over possibly untyped PLC).
 1006 
 1007 However consider the following imaginary builtin:
 1008 
 1009     nil : all a. [a]
 1010 
 1011 we can't represent it for two reasons:
 1012 
 1013 1. we don't have any argument providing us a type tag for @a@ and hence we can't construct a type
 1014    tag for @[a]@
 1015 2. it would be a very unsound builtin to have. We can only instantiate built-in types with other
 1016    built-in types and so allowing @nil {some_non_built_in_type}@ would be a lie that couldn't reduce
 1017    to anything since it's not even possible to represent a built-in list with non-built-in elements
 1018    (even if there's zero of them)
 1019 
 1020 "Wait, but wouldn't @cons {some_non_built_in_type}@ be a lie as well?" -- No! Since @cons@ does not
 1021 just construct a list filled with elements of a non-built-in type but also expects one as an
 1022 argument and providing such an argument is impossible, 'cause it's pretty much the same thing as
 1023 populating 'Void' -- both values are equally unrepresentable. And so @cons {some_non_built_in_type}@
 1024 is a way to say @absurd@, which is perfectly fine to have.
 1025 
 1026 Finally,
 1027 
 1028     comma :: all a b. a -> b -> (a, b)
 1029 
 1030 is representable (because we can require arguments to be constants carrying universes with them,
 1031 which we can use to construct the resulting universe), but is still a lie, because instantiating
 1032 that builtin with non-built-in types is possible and so the PLC type checker won't throw on such
 1033 an instantiation, which will become 'evalutionFailure' at runtime the moment unlifting of a
 1034 non-constant is attempted when a constant is expected.
 1035 
 1036 So could we still get @nil@ or a safe version of @comma@ somehow? Well, we could have this
 1037 weirdness:
 1038 
 1039     nilOfTypeOf : all a. [a] -> [a]
 1040 
 1041 i.e. ask for an already existing list, but ignore the actual list and only use the type tag.
 1042 
 1043 But since we're ignoring the actual list, can't we just not pass it in the first place? And instead
 1044 pass around our good old friends, singletons. We should be able to do that, but it hasn't been
 1045 investigated. Perhaps something along the lines of adding the following constructor to 'DefaultUni':
 1046 
 1047     DefaultUniProtoSing :: DefaultUni (Esc (Proxy @GHC.Type))
 1048 
 1049 and then defining
 1050 
 1051     nil : all a. sing a -> [a]
 1052 
 1053 and then the Plutus Tx compiler can provide a type class or something for constructing singletons
 1054 for built-in types.
 1055 
 1056 This was investigated in https://github.com/IntersectMBO/plutus/pull/4337 but we decided not to
 1057 do it quite yet, even though it worked (the Plutus Tx part wasn't implemented).
 1058 -}

 1889   toBuiltinMeaning _semvar DropList =
 1890     let dropListDenotation
 1891           :: IntegerCostedLiterally -> SomeConstant uni [a] -> BuiltinResult (Opaque val [a])
 1892         dropListDenotation i (SomeConstant (Some (ValueOf uniListA xs))) = do
 1893           -- See Note [Operational vs structural errors within builtins].
 1894           case uniListA of
 1895             DefaultUniList _ ->
 1896               -- The fastest way of dropping elements from a list is by operating on
 1897               -- an unboxed int (i.e. an 'Int#'). We could implement that manually, but
 1898               -- 'drop' in @Prelude@ already does that under the hood, so we just need to
 1899               -- convert the given 'Integer' to an 'Int' and call 'drop' over that.
 1900               fromValueOf uniListA <$> case unIntegerCostedLiterally i of
 1901                 IS i# -> pure $ drop (I# i#) xs
 1902                 IN _ -> pure xs
 1903                 -- If the given 'Integer' is higher than @maxBound :: Int@, then we
 1904                 -- call 'drop' over the latter instead to get the same performance as in
 1905                 -- the previous case. This will produce a different result when the
 1906                 -- list is longer than @maxBound :: Int@, but in practice not only is
 1907                 -- the budget going to get exhausted long before a @maxBound@ number of
 1908                 -- elements is skipped, it's not even feasible to skip so many elements
 1909                 -- for 'Int64' (and we enforce @Int = Int64@ in the @Universe@ module)
 1910                 -- as that'll take ~3000 years assuming it takes a second to skip
 1911                 -- @10^8@ elements.
 1912                 --
 1913                 -- We could "optimistically" return '[]' directly without doing any
 1914                 -- skipping at all, but then we'd be occasionally returning the wrong
 1915                 -- answer immediately rather than in 3000 years and that doesn't sound
 1916                 -- like a good idea. Particularly given that costing for this builtin is
 1917                 -- oblivious to such implementation details anyway, so we wouldn't even
 1918                 -- be able to capitalize on the fast and loose approach.
 1919                 --
 1920                 -- Instead of using 'drop' we could've made it something like
 1921                 -- @foldl' const []@, which would be a tad faster while taking even more
 1922                 -- than 3000 years to return the wrong result, but with the current
 1923                 -- approach the wrong result is an error, while with the foldl-based one
 1924                 -- it'd be an actual incorrect value. Plus, it's best not to rely on
 1925                 -- GHC not optimizing such an expression away to just '[]' (it really
 1926                 -- shouldn't, but not taking chances with GHC is the best approach). And
 1927                 -- again, we wouldn't be able to capitalize on such a speedup anyway.
 1928                 IP _ -> case drop maxBound xs of
 1929                   [] -> pure []
 1930                   _ ->
 1931                     throwError $
 1932                       structuralUnliftingError
 1933                         "Panic: unreachable clause executed"
 1934             _ -> throwError $ structuralUnliftingError "Expected a list but got something else"
 1935         {-# INLINE dropListDenotation #-}
 1936      in makeBuiltinMeaning
 1937           dropListDenotation
 1938           (runCostingFunTwoArguments . paramDropList)

 2294 {- Note [Legacy pattern matching on built-in types]
 2295 We used to only support direct pattern matching on enumeration types: 'Void', 'Unit', 'Bool'
 2296 etc. This is because it was impossible to 'Case' on a value of a built-in type.
 2297 
 2298 So e.g. if we wanted to add the following data type:
 2299 
 2300     newtype AnInt = AnInt Int
 2301 
 2302 as a built-in type, we wouldn't be able to add the following function as its pattern matcher:
 2303 
 2304     matchAnInt :: AnInt -> (Int -> r) -> r
 2305     matchAnInt (AnInt i) f = f i
 2306 
 2307 because we could not express the @f i@ part using the builtins machinery.
 2308 
 2309 But it still was possible to have @AnInt@ as a built-in type, it's just that instead of trying to
 2310 make its pattern matcher into a builtin we could have the following builtin:
 2311 
 2312     anIntToInt :: AnInt -> Int
 2313     anIntToInt (AnInt i) = i
 2314 
 2315 Although that was an annoyance for more complex data types. For tuples we needed to provide two
 2316 projection functions ('fst' and 'snd') instead of a single pattern matcher, which is not too bad,
 2317 but to get pattern matching on lists we needed a more complicated setup. For example originally we
 2318 would define three built-in functions: @null@, @head@ and @tail@, plus require `Bool` to be in the
 2319 universe, so that we can define an equivalent of
 2320 
 2321     matchList :: [a] -> r -> (a -> [a] -> r) -> r
 2322     matchList xs z f = if null xs then z else f (head xs) (tail xs)
 2323 
 2324 If a constructor stores more than one value, the corresponding projection function would pack them
 2325 into a (possibly nested) pair, for example for
 2326 
 2327     data Data
 2328         = Constr Integer [Data]
 2329         | <...>
 2330 
 2331 we had (pseudocode):
 2332 
 2333     unConstrData (Constr i ds) = (i, ds)
 2334 
 2335 and we still have that built-in function for backwards compatibility reasons.
 2336 
 2337 In order to get pattern matching over 'Data' we needed a projection function per constructor as well
 2338 as with lists, but writing (where the @Data@ suffix indicates that a function is a builtin that
 2339 somehow corresponds to a constructor of 'Data')
 2340 
 2341     if isConstrData d
 2342         then uncurry fConstr $ unConstrData d
 2343         else if isMapData d
 2344             then fMap $ unMapData d
 2345             else if isListData d
 2346                 then fList $ unListData d
 2347                 else <...>
 2348 
 2349 is tedious and inefficient and so instead we introduced a single @chooseData@ builtin that matches
 2350 on its @Data@ argument and chooses the appropriate branch (type instantiations and strictness
 2351 concerns are omitted for clarity):
 2352 
 2353      chooseData
 2354         (uncurry fConstr $ unConstrData d)
 2355         (fMap $ unMapData d)
 2356         (fList $ unListData d)
 2357         <...>
 2358         d
 2359 
 2360 which, for example, evaluates to @fMap es@ when @d@ is @Map es@
 2361 
 2362 We decided to handle lists the same way by using @chooseList@ rather than @null@ for consistency,
 2363 before introduction of casing on values of built-in types.
