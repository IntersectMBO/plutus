  204 -- | Turn a function into another function that 'fail's when its second argument is @0@ or calls the
  205 -- original function otherwise and wraps the result in 'pure'. Useful for correctly handling `div`,
  206 -- `mod`, etc.
  207 nonZeroSecondArg
  208     :: (Integer -> Integer -> Integer) -> Integer -> Integer -> BuiltinResult Integer
  209 -- If we match against @IS 0#@ instead of @0@, GHC will generate tidier Core for some reason. It
  210 -- probably doesn't really matter performance-wise, but would be easier to read. We don't do it out
  211 -- of paranoia and because it requires importing the 'IS' constructor, which is in different
  212 -- packages depending on the GHC version, so requires a bunch of irritating CPP.
  213 --
  214 -- We could also replace 'div' with 'integerDiv' (and do the same for other division builtins) at
  215 -- the call site of this function in order to avoid double matching against @0@, but that also
  216 -- requires CPP. Perhaps we can afford one additional pattern match for division builtins for the
  217 -- time being, since those aren't particularly fast anyway.
  218 --
  219 -- The bang is to communicate to GHC that the function is strict in both the arguments just in case
  220 -- it'd want to allocate a thunk for the first argument otherwise.
  221 nonZeroSecondArg _ !_ 0 =
  222     -- See Note [Structural vs operational errors within builtins].
  223     fail "Cannot divide by zero"
  224 nonZeroSecondArg f  x y = pure $ f x y
  225 {-# INLINE nonZeroSecondArg #-}

  233 {- Note [Constants vs built-in functions]
  234 A constant is any value of a built-in type. For example, 'Integer' is a built-in type, so anything
  235 of type 'Integer' is a constant.
  236 
  237 On the contrary a built-in function can't be of a built-in type, because the type of a built-in
  238 function is always of either the @all a. b@ form or the @a -> b@ one, none of which is a built-in
  239 type. This is checked by the machinery, so if the user tries to add a built-in function that is not
  240 of one of these forms, they'll get a nice custom type error.
  241 
  242 A built-in function is associated with its Haskell implementation: there can be many built-in
  243 functions of the same type, all doing different things, and there can be infinitely more _definable_
  244 built-in functions of the same type that are not built-in functions nonetheless, because we didn't
  245 register them as such by providing a Haskell implementation for each of them. This is the difference
  246 between constants and built-in functions: the set of constants (infinite in our case) depends solely
  247 on the set of available built-in types (also infinite in our case, because we have @Integer@,
  248 @[Integer]@, @[[Integer]]@ etc), while the set of built-in functions is defined by explicitly
  249 assigning each member a specific name and an associated with it Haskell implementation. It is
  250 theoretically possible to have an infinite set of built-in functions, but we neither do that nor
  251 need it, hence our set of built-in functions is finite.
  252 
  253 The rule of thumb is: constants are raw data and built-in functions are, well, functions.
  254 
  255 @(:)@ works as follows: it takes two constants wrapped as values, extracts an integer from the first
  256 constant and a list of integers from the second one, prepends the former to the latter and wraps the
  257 resulting list back into a constant, which gets wrapped into a value.
  258 
  259 Why does @(:)@ have to be a built-in function? Because its type is
  260 
  261     all a. a -> list a -> list a
  262 
  263 and if we tried to make @(:)@ a constant we'd have to somehow make this type a built-in type and
  264 promise that every value (i.e. every definable function) of this type can be used as a Plutus term,
  265 which doesn't make any sense. Only the particular Haskell implementation that prepends an element to
  266 a list is what we're interested in.
  267 
  268 Why may @[]@ not be a built-in function? If its type is hardcoded to @[Integer]@, then that's a
  269 built-in type and we know that anything of a built-in type can be embedded into a term as a
  270 constant. I.e. @[] :: [Integer]@ is perfectly fine as a constant and does not need to be a built-in
  271 function.
  272 
  273 Why may @[]@ be a built-in function? If it's polymorphic over the type of the elements, then its
  274 Plutus Core type is @all a. list a@ and that is not a built-in type, hence we have to make that a
  275 built-in function.
  276 -}

  278 {- Note [How to add a built-in function: simple cases]
  279 This Note explains how to add a built-in function and how to read definitions of existing built-in
  280 functions. It does not attempt to explain why things the way they are, that is explained in comments
  281 in relevant modules, check out the following for an overview of the module structure:
  282 https://github.com/IntersectMBO/plutus/blob/97c2b2c6975e41ce25ee5efa1dff0f1bd891a589/plutus-core/docs/BuiltinsOverview.md
  283 
  284 In order to add a new built-in function one needs to add a constructor to 'DefaultFun' and handle
  285 it within the @ToBuiltinMeaning uni DefaultFun@ instance. The general pattern is
  286 
  287     toBuiltinMeaning semvar <BuiltinName> =
  288         let <builtinNameDenotation> :: BS.ByteString -> BS.ByteString
  289             <builtinNameDenotation> = <denotation>
  290             {-# INLINE <builtinNameDenotation> #-}
  291         in makeBuiltinMeaning
  292             <builtinNameDenotation>
  293             <costingFunction>
  294 
  295 Here's a specific example:
  296 
  297     toBuiltinMeaning _ AddInteger =
  298         let addIntegerDenotation :: Integer -> Integer -> Integer
  299             addIntegerDenotation = (+)
  300             {-# INLINE addIntegerDenotation #-}
  301         in makeBuiltinMeaning
  302             addIntegerDenotation
  303             (runCostingFunTwoArguments . paramAddInteger)
  304 
  305 'makeBuiltinMeaning' creates a Plutus builtin out of its denotation (i.e. Haskell implementation)
  306 and a costing function for it. Once a builtin is added, its Plutus type is kind-checked and printed
  307 to a golden file automatically (consult @git status@). 'toBuiltinMeaning' also takes a
  308 'BuiltinSemanticsVariant' argument which allows a particular builtin name to have multiple
  309 associated denotations (see Note [Builtin semantics variants]), but for simplicity we assume in the
  310 examples below that all builtins have only one variant rendering the @semvar@ argument irrelevant.
  311 
  312 See Note [Builtin semantics variants] for how @semvar@ enables us to customize the behavior of a
  313 built-in function. For the purpose of these docs we're going to ignore that and use @_@ instead of
  314 @semvar@.
  315 
  316 Note that it's very important for the denotation to have an explicit type signature for several
  317 reasons:
  318 
  319 1. makes it easier to review the code and make sure it makes sense
  320 2. makes it easier to search for builtins associated with certain types -- just @grep@ for the type
  321 3. most importantly, if we let GHC infer the types, there's a small but very real chance that
  322    updating a library to a newer version will change the type of some definition used within the
  323    denotation of a builtin and that may get reflected in the type signature of the builtin without
  324    us noticing, since the builtins machinery will gladly swallow the change. And since the type
  325    signature of a builtin determines its behavior via ad hoc polymorphism a change in the type
  326    signature can cause a sudden hardfork, which would be very bad
  327 
  328 hence we specify the type signature for the denotation of each builtin explicitly and always create
  329 a @let@ binding for consistency. We add an @INLINE@ pragma to the @let@ binding to make sure that
  330 the binding doesn't get in the way of performance.
  331 
  332 Below we will enumerate what kind of denotations are accepted by 'makeBuiltinMeaning' without
  333 touching any costing stuff.
  334 
  335 1. The simplest example of an accepted denotation is a monomorphic function that takes values of
  336 built-in types and returns a value of a built-in type as well. For example
  337 
  338     encodeUtf8 :: Text -> BS.ByteString
  339 
  340 You can feed 'encodeUtf8' directly to 'makeBuiltinMeaning' without specifying any types:
  341 
  342     toBuiltinMeaning _ EncodeUtf8 =
  343         let encodeUtf8Denotation :: Text -> BS.ByteString
  344             encodeUtf8Denotation = encodeUtf8
  345             {-# INLINE encodeUtf8Denotation #-}
  346         in makeBuiltinMeaning
  347             encodeUtf8Denotation
  348             <costingFunction>
  349 
  350 This will add the builtin, the only two things that remain are implementing costing for this
  351 builtin (out of the scope of this Note) and handling it within the @Flat DefaultFun@ instance
  352 (see Note [Stable encoding of TPLC]).
  353 
  354 2. Unconstrained type variables are fine, you don't need to instantiate them. For example
  355 
  356     toBuiltinMeaning _ IfThenElse =
  357         let ifThenElseDenotation :: Bool -> a -> a -> a
  358             ifThenElseDenotation b x y = if b then x else y
  359             {-# INLINE ifThenElseDenotation #-}
  360         in makeBuiltinMeaning
  361             ifThenElseDenotation
  362             <costingFunction>
  363 
  364 works alright. The Haskell type of the denotation is
  365 
  366     forall a. Bool -> a -> a -> a
  367 
  368 whose counterpart in Plutus is
  369 
  370     all a. bool -> a -> a -> a
  371 
  372 and unsurprisingly it's the exact Plutus type of the added builtin.
  373 
  374 It may seem like getting the latter from the former is entirely trivial, however
  375 'makeBuiltinMeaning' jumps through quite a few hoops to achieve that and below we'll consider those
  376 of them that are important to know to be able to use 'makeBuiltinMeaning' in cases that are more
  377 complicated than a simple monomorphic or polymorphic function. But for now let's talk about a few
  378 more simple cases.
  379 
  380 3. Certain types are not built-in, but can be represented via built-in ones. For example, we don't
  381 have 'Int' built-in, but we have 'Integer' and we can represent the former in terms of the
  382 latter. The conversions between the two types are handled by 'makeBuiltinMeaning', so that the user
  383 doesn't need to write them themselves and can just write
  384 
  385     toBuiltinMeaning _ LengthOfByteString =
  386         let lengthOfByteStringDenotation :: BS.ByteString -> Int
  387             lengthOfByteStringDenotation = BS.length
  388             {-# INLINE lengthOfByteStringDenotation #-}
  389         in makeBuiltinMeaning
  390             lengthOfByteStringDenotation
  391             <costingFunction>
  392 
  393 directly (where @BS.length :: BS.ByteString -> Int@).
  394 
  395 Note however that while it's always safe to convert an 'Int' to an 'Integer', doing the opposite is
  396 not safe in general, because an 'Integer' may not fit into the range of 'Int'. For this reason
  397 
  398     YOU MUST NEVER USE 'fromIntegral' AND SIMILAR FUNCTIONS THAT CAN SILENTLY UNDER- OR OVERFLOW
  399     WHEN DEFINING A BUILT-IN FUNCTION
  400 
  401 For example defining a builtin that takes an 'Integer' and converts it to an 'Int' using
  402 'fromIntegral' is not allowed under any circumstances and can be a huge vulnerability.
  403 
  404 It's completely fine to define a builtin that takes an 'Int' directly, though. How so? That's due
  405 to the fact that the builtin application machinery checks that an 'Integer' is in the bounds of
  406 'Int' before doing the conversion. If the bounds check succeeds, then the 'Integer' gets converted
  407 to the corresponding 'Int', and if it doesn't, then the builtin application fails.
  408 
  409 For the list of types that can be converted to/from built-in ones look into the file with the
  410 default universe. If you need to add a new such type, just copy-paste what's done for an existing
  411 one and adjust.
  412 
  413 Speaking of builtin application failing:
  414 
  415 4. A built-in function can fail. Whenever a builtin fails, evaluation of the whole program fails.
  416 There's a number of ways a builtin can fail:
  417 
  418 - as we've just seen a type conversion can fail due to an unsuccessful bounds check
  419 - if the builtin expects, say, a 'Text' argument, but gets fed an 'Integer' argument
  420 - if the builtin expects any constant, but gets fed a non-constant
  421 - if its denotation runs in the 'BuiltinResult' monad and an 'evaluationFailure' gets returned
  422 
  423 Most of these are not a concern to the user defining a built-in function (conversions are handled
  424 within the builtin application machinery, type mismatches are on the type checker and the person
  425 writing the program etc), however explicitly returning 'evaluationFailure' from a builtin is
  426 something that happens commonly.
  427 
  428 One simple example is a monomorphic function matching on a certain constructor and failing in all
  429 other cases:
  430 
  431     toBuiltinMeaning _ UnIData =
  432         let unIDataDenotation :: Data -> BuiltinResult Integer
  433             unIDataDenotation = \case
  434                 I i -> pure i
  435                 _   -> evaluationFailure
  436             {-# INLINE unIDataDenotation #-}
  437         in makeBuiltinMeaning
  438             unIDataDenotation
  439             <costingFunction>
  440 
  441 The type of the denotation is
  442 
  443     Data -> BuiltinResult Integer
  444 
  445 and the Plutus type of the builtin is
  446 
  447     data -> integer
  448 
  449 because the error effect is implicit in Plutus.
  450 
  451 Returning @BuiltinResult a@ for a type variable @a@ is also fine, i.e. it doesn't matter whether
  452 the denotation is monomorphic or polymorphic w.r.t. failing.
  453 
  454 But note that
  455 
  456     'BuiltinResult' MUST BE EXPLICITLY USED FOR ANY FAILING BUILTIN AND THROWING AN EXCEPTION
  457     VIA 'error' OR 'throw' OR ELSE IS NOT ALLOWED AND CAN BE A HUGE VULNERABILITY. MAKE SURE THAT
  458     NONE OF THE FUNCTIONS THAT YOU USE TO DEFINE A BUILTIN THROW EXCEPTIONS
  459 
  460 An argument of a builtin can't have 'BuiltinResult' in its type -- only the result.
  461 
  462 5. A builtin can emit log messages. For that its denotation needs to run in the 'BuiltinResult' as
  463 in case of failing. The ergonomics are the same. For example:
  464 
  465     toBuiltinMeaning _ Trace =
  466         let traceDenotation :: Text -> a -> BuiltinResult a
  467             traceDenotation text a = a <$ emit text
  468             {-# INLINE traceDenotation #-}
  469         in makeBuiltinMeaning
  470             traceDenotation
  471             <costingFunction>
  472 
  473 The type of the denotation is
  474 
  475     forall a. Text -> a -> Builtin a
  476 
  477 and the Plutus type of the builtin is
  478 
  479     all a. text -> a -> a
  480 
  481 because just like with the error effect, whether a function logs anything or not is not reflected
  482 in its type.
  483 
  484 This concludes the list of simple cases. Before we jump to the hard ones, we need to talk about how
  485 polymorphism gets elaborated, so read Note [Elaboration of polymorphism] next.
  486 -}

  488 {- Note [Elaboration of polymorphism]
  489 In Note [How to add a built-in function: simple cases] we defined the following builtin:
  490 
  491     toBuiltinMeaning _ IfThenElse =
  492         let ifThenElseDenotation :: Bool -> a -> a -> a
  493             ifThenElseDenotation b x y = if b then x else y
  494             {-# INLINE ifThenElseDenotation #-}
  495         in makeBuiltinMeaning
  496             ifThenElseDenotation
  497             <costingFunction>
  498 
  499 whose Haskell type is
  500 
  501     forall a. Bool -> a -> a -> a
  502 
  503 The way 'makeBuiltinMeaning' handles such a type is by traversing it and instantiating every type
  504 variable. What a type variable gets instantiated to depends on where it appears. When the entire
  505 type of an argument is a single type variable, it gets instantiated to @Opaque val VarN@ where
  506 @VarN@ is pseudocode for "a Haskell type representing a Plutus type variable with 'Unique' N"
  507 For the purpose of this explanation it doesn't matter what @VarN@ actually is and the representation
  508 is subject to change anyway (see Note [Implementation of polymorphic built-in functions] if you want
  509 to know the details). 'Opaque' however is more fundamental and so we need to talk about it.
  510 Here's how it's defined:
  511 
  512     newtype Opaque val (rep :: GHC.Type) = Opaque
  513         { unOpaque :: val
  514         }
  515 
  516 I.e. @Opaque val rep@ is a wrapper around @val@, which stands for the type of value that an
  517 evaluator uses (the builtins machinery is designed to work with any evaluator and different
  518 evaluators define their type of values differently, for example 'CkValue' if the type of value for
  519 the CK machine). The idea is simple: in order to apply the denotation of a builtin expecting, say,
  520 an 'Integer' constant we need to actually extract that 'Integer' from the AST of the given value,
  521 but if the denotation is polymorphic over the type of its argument, then we don't need to extract
  522 anything, we can just pass the AST of the value directly to the denotation (which means the value
  523 doesn't have to be a 'Constant', it can be completely arbitrary). I.e. in order for a polymorphic
  524 function to become a monomorphic denotation (denotations are always monomorpic) all type variables
  525 in the type of that function need to be instantiated at the type of value that a given evaluator
  526 uses.
  527 
  528 If we used just @val@ rather than @Opaque val rep@, we'd specialize
  529 
  530     forall a. Bool -> a -> a -> a
  531 
  532 to
  533 
  534     Bool -> val -> val -> val
  535 
  536 however then we'd need to separately specify the Plutus type of this builtin, since we can't infer
  537 it from all these @val@s in the general case, for example does
  538 
  539     val -> val -> val
  540 
  541 stand for
  542 
  543     all a. a -> a -> a
  544 
  545 or
  546 
  547     all a b. a -> b -> a
  548 
  549 or something else?
  550 
  551 So we use the @Opaque val rep@ wrapper, which is basically a @val@ with a @rep@ attached to it where
  552 @rep@ represents the Plutus type of the argument/result, which is how we arrive at
  553 
  554     Bool -> Opaque val Var0 -> Opaque val Var0 -> Opaque val Var0
  555 
  556 This encoding allows us to specify both the Haskell and the Plutus types of the builtin
  557 simultaneously.
  558 
  559 If we wanted to we could add explicit 'Opaque' while still having explicit polymorphism (leaving out
  560 the @Var0@ thing for the elaboration machinery to figure out):
  561 
  562     toBuiltinMeaning _ IfThenElse =
  563         let ifThenElseDenotation :: Bool -> Opaque val a -> Opaque val a -> Opaque val a
  564             ifThenElseDenotation b x y = if b then x else y
  565             {-# INLINE ifThenElseDenotation #-}
  566         in makeBuiltinMeaning
  567             ifThenElseDenotation
  568             <costingFunction>
  569 
  570 and it would be equivalent to the original definition, but note how @a@ is now an argument to
  571 'Opaque' rather than the entire type of an argument. In order for this definition to elaborate to
  572 the same type as before @a@ needs to be instantiated to just @Var0@, as opposed to @Opaque val
  573 Var0@, because the 'Opaque' part is already there, so this is what the elaboration machinery does.
  574 
  575 So regardless of which method of defining 'IfThenElse' we choose, the type of its denotation gets
  576 elaborated to the same
  577 
  578     Bool -> Opaque val Var0 -> Opaque val Var0 -> Opaque val Var0
  579 
  580 which then gets digested, so that we can compute what Plutus type it corresponds to. The procedure
  581 is simple: collect all distinct type variables, @all@-bind them and replace the usages with the
  582 bound variables. This turns the type above into
  583 
  584     all a. bool -> a -> a -> a
  585 
  586 which is the Plutus type of the 'IfThenElse' builtin.
  587 
  588 It's of course allowed to have multiple type variables, e.g. in the following snippet:
  589 
  590     toBuiltinMeaning _ Const =
  591         let constDenotation :: a -> b -> a
  592             constDenotation = Prelude.const
  593             {-# INLINE constDenotation #-}
  594         in makeBuiltinMeaning
  595             constDenotation
  596             <costingFunction>
  597 
  598 the Haskell type of 'const' is
  599 
  600     forall a b. a -> b -> a
  601 
  602 which the elaboration machinery turns into
  603 
  604     Opaque val Var0 -> Opaque val Var1 -> Opaque val Var0
  605 
  606 The elaboration machinery respects the explicitly specified parts of the type and does not attempt
  607 to argue with them. For example if the user insisted that the instantiated type of 'const' had
  608 @Var0@ and @Var1@ swapped:
  609 
  610     Opaque val Var1 -> Opaque val Var0 -> Opaque val Var1
  611 
  612 the elaboration machinery wouldn't make a fuss about that.
  613 
  614 As a final simple example, consider
  615 
  616     toBuiltinMeaning _ Trace =
  617         let traceDenotation :: Text -> a -> BuiltinResult a
  618             traceDenotation text a = a <$ emit text
  619             {-# INLINE traceDenotation #-}
  620         in makeBuiltinMeaning
  621             traceDenotation
  622             <costingFunction>
  623 
  624 from [How to add a built-in function: simple cases]. The type of the denotation is
  625 
  626     forall a. Text -> a -> BuiltinResult a
  627 
  628 which elaborates to
  629 
  630     Text -> Opaque val Var0 -> BuiltinResult (Opaque val Var0)
  631 
  632 Elaboration machinery is able to look under 'BuiltinResult' even if there's a type variable inside
  633 that does not appear anywhere else in the type signature, for example the type of the denotation in
  634 
  635     toBuiltinMeaning _ ErrorPrime =
  636         let errorPrimeDenotation :: BuiltinResult a
  637             errorPrimeDenotation = evaluationFailure
  638             {-# INLINE errorPrimeDenotation #-}
  639         in makeBuiltinMeaning
  640             errorPrimeDenotation
  641             <costingFunction>
  642 
  643 is
  644 
  645     forall a. BuiltinResult a
  646 
  647 which gets elaborated to
  648 
  649     BuiltinResult (Opaque val Var0)
  650 
  651 from which the final Plutus type of the builtin is computed:
  652 
  653     all a. a
  654 
  655 Read Note [How to add a built-in function: complicated cases] next.
  656 -}

  658 {- Note [How to add a built-in function: complicated cases]
  659 Now let's talk about more complicated built-in functions.
  660 
  661 1. In Note [Elaboration of polymorphism] we saw how a Haskell type variable gets elaborated to an
  662 @Opaque val VarN@ and we learned that this type can be used directly as opposed to being inferred.
  663 However there exist more ways to use 'Opaque' explicitly. Here's a simple example:
  664 
  665     toBuiltinMeaning _ IdAssumeBool =
  666         let idAssumeBoolDenotation :: Opaque val Bool -> Opaque val Bool
  667             idAssumeBoolDenotation = Prelude.id
  668             {-# INLINE idAssumeBoolDenotation #-}
  669         in makeBuiltinMeaning
  670             idAssumeBoolDenotation
  671             <costingFunction>
  672 
  673 This creates a built-in function whose Plutus type is
  674 
  675     id : bool -> bool
  676 
  677 i.e. the Plutus type signature of the built-in function is the same as with
  678 
  679     toBuiltinMeaning _ IdBool =
  680         let idBoolDenotation :: Bool -> Bool
  681             idBoolDenotation = Prelude.id
  682             {-# INLINE idBoolDenotation #-}
  683         in makeBuiltinMeaning
  684             idBoolDenotation
  685             <costingFunction>
  686 
  687 but the two evaluate differently: the former takes a value and returns it right away while the
  688 latter takes a value, extracts a 'Bool' constant out of it and then lifts that constant back into
  689 @val@. The difference is not only in performance (obviously returning something right away is
  690 cheaper than unlifting-then-lifting-back), but also in semantics: the former returns its argument
  691 during evaluation regardless of what that argument is, so if someone generates Untyped Plutus Core
  692 directly, they can apply @IdAssumeBool@ to a term that doesn't evaluate to a 'Bool' constant or
  693 even a constant at all and that won't be a runtime error, while the latter has to be applied to
  694 a term evaluating to a 'Bool' constant in order not to fail at runtime.
  695 
  696 2. @val@ in @Opaque val rep@ is not completely arbitrary, it has to implement 'HasConstant', which
  697 makes it possible to unlift @val@ as a constant or lift a constant back into @val@. There's a
  698 'HasConstant' instance for @Opaque val rep@ whenever there's one for @val@, so if we, for some
  699 reason, wanted to have 'Opaque' in the type signature of the denotation, but still unlift the
  700 argument as a 'Bool', we could do that:
  701 
  702     toBuiltinMeaning _ IdAssumeCheckBool =
  703         let idAssumeCheckBoolDenotation :: Opaque val Bool -> BuiltinResult Bool
  704             idAssumeCheckBoolDenotation val = asConstant val of
  705                 Right (Some (ValueOf DefaultUniBool b)) -> pure b
  706                 _                                       -> evaluationFailure
  707             {-# INLINE idAssumeCheckBoolDenotation #-}
  708         in makeBuiltinMeaning
  709             idAssumeCheckBoolDenotation
  710             <costingFunction>
  711 
  712 Here in the denotation we unlift the given value as a constant, check that its type tag is
  713 'DefaultUniBool' and return the unlifted 'Bool'. If any of that fails, we return an explicit
  714 'evaluationFailure'.
  715 
  716 This achieves almost the same as 'IdBool', which keeps all the bookkeeping behind the scenes, but
  717 there is a minor difference: in case of error its message is ignored. It would be easy to allow for
  718 returning an unlifting error from a builtin explicitly, but we don't need that for anything, hence
  719 it's not implemented.
  720 
  721 We call this style of manually calling 'asConstant' and matching on the type tag "manual unlifting".
  722 As opposed to "automatic unlifting" that we were using before where 'Bool' in the type of the
  723 denotation of a builtin causes the builtins machinery to convert the given argument to a 'Bool'
  724 constant automatically behind the scenes.
  725 
  726 3. There's a middle ground between automatic and manual unlifting to 'Bool', one can unlift a value
  727 automatically as a constant and then unlift the result manually to 'Bool' using the 'SomeConstant'
  728 wrapper:
  729 
  730     newtype SomeConstant uni (rep :: GHC.Type) = SomeConstant
  731         { unSomeConstant :: Some (ValueOf uni)
  732         }
  733 
  734 'SomeConstant' is similar to 'Opaque' in that it has a @rep@ representing a Plutus type.
  735 The difference is that 'Opaque' is a wrapper around an arbitrary value and 'SomeConstant' is a
  736 wrapper around a constant. 'SomeConstant' allows one to automatically unlift an argument of a
  737 built-in function as a constant with all 'asConstant' business kept behind the scenes, for example:
  738 
  739     toBuiltinMeaning _ IdSomeConstantBool =
  740         let idSomeConstantBoolDenotation :: SomeConstant uni Bool -> BuiltinResult Bool
  741             idSomeConstantBoolDenotation = \case
  742                 SomeConstant (Some (ValueOf DefaultUniBool b)) -> pure b
  743                 _                                              -> evaluationFailure
  744             {-# INLINE idSomeConstantBoolDenotation #-}
  745         in makeBuiltinMeaning
  746             idSomeConstantBoolDenotation
  747             <costingFunction>
  748 
  749 Note how we no longer call 'asConstant' manually, but still manually match on 'DefaultUniBool'.
  750 
  751 So there's a whole range of how "low-level" one can choose to be when defining a built-in function.
  752 However it's not always possible to use automatic unlifting, see next.
  753 
  754 4. If we try to define the following built-in function:
  755 
  756     toBuiltinMeaning _ NullList =
  757         let nullListDenotation :: [a] -> Bool
  758             nullListDenotation = null
  759             {-# INLINE nullListDenotation #-}
  760         in makeBuiltinMeaning
  761             nullListDenotation
  762             <costingFunction>
  763 
  764 we'll get an error, saying that a polymorphic built-in type can't be applied to a type variable.
  765 It's not impossible to make it work, see Note [Unlifting a term as a value of a built-in type], but
  766 not in the general case, plus it has to be very inefficient.
  767 
  768 Instead we have to use 'SomeConstant' to automatically unlift the argument as a constant and then
  769 check that the value inside of it is a list (by matching on the type tag):
  770 
  771     toBuiltinMeaning _ NullList =
  772         let nullListDenotation :: SomeConstant uni [a] -> BuiltinResult Bool
  773             nullListDenotation (SomeConstant (Some (ValueOf uniListA xs))) = do
  774                 case uniListA of
  775                     DefaultUniList _ -> pure $ null xs
  776                     _ -> throwing _StructuralUnliftingError "Expected a list but got something else"
  777             {-# INLINE nullListDenotation #-}
  778         in makeBuiltinMeaning
  779             nullListDenotation
  780             <costingFunction>
  781 
  782 As before, we have to match on the type tag, because there's no relation between @rep@ from
  783 @SomeConstant uni rep@ and the constant that the built-in function actually receives at runtime
  784 (someone could generate Untyped Plutus Core directly and apply 'nullPlc' to an 'Integer' or
  785 whatever). @rep@ is only for the Plutus type checker to look at, it doesn't influence evaluation
  786 in any way.
  787 
  788 Here's a similar built-in function:
  789 
  790     toBuiltinMeaning _ FstPair =
  791         let fstPairDenotation :: SomeConstant uni (a, b) -> BuiltinResult (Opaque val a)
  792             fstPairDenotation (SomeConstant (Some (ValueOf uniPairAB xy))) = do
  793                 case uniPairAB of
  794                     DefaultUniPair uniA _ ->              -- [1]
  795                         pure . fromValueOf uniA $ fst xy  -- [2]
  796                     _ ->
  797                         throwing _StructuralUnliftingError "Expected a pair but got something else"
  798             {-# INLINE fstPairDenotation #-}
  799         in makeBuiltinMeaning
  800             fstPairDenotation
  801             <costingFunction>
  802 
  803 In this definition we extract the first element of a pair by checking that the given constant is
  804 indeed a pair [1] and lifting its first element into @val@ using the type tag for the first
  805 element [2] (extracted from the type tag for the whole pair constant [1]).
  806 
  807 Note that it's fine to mix automatic unlifting for polymorphism not related to built-in types and
  808 manual unlifting for arguments having non-monomorphized polymorphic built-in types, for example:
  809 
  810     toBuiltinMeaning _ ChooseList =
  811         let chooseListDenotation :: SomeConstant uni [a] -> b -> b -> BuiltinResult b
  812             chooseListDenotation (SomeConstant (Some (ValueOf uniListA xs))) a b = do
  813                 case uniListA of
  814                     DefaultUniList _ -> pure $ case xs of
  815                         []    -> a
  816                         _ : _ -> b
  817                     _ -> throwing _StructuralUnliftingError "Expected a list but got something else"
  818             {-# INLINE chooseListDenotation #-}
  819         in makeBuiltinMeaning
  820             chooseListDenotation
  821             (runCostingFunThreeArguments . paramChooseList)
  822             <costingFunction>
  823 
  824 Here @a@ appears inside @[]@, which is a polymorphic built-in type, and so we have to use
  825 'SomeConstant' and check that the given constant is indeed a list, while @b@ doesn't appear inside
  826 of any built-in type and so we don't need to instantiate it to 'Opaque' manually, the elaboration
  827 machinery will do it for us.
  828 
  829 Our final example is this:
  830 
  831     toBuiltinMeaning _ MkCons =
  832         let mkConsDenotation
  833                 :: SomeConstant uni a -> SomeConstant uni [a] -> BuiltinResult (Opaque val [a])
  834             mkConsDenotation
  835               (SomeConstant (Some (ValueOf uniA x)))
  836               (SomeConstant (Some (ValueOf uniListA xs))) = do
  837                 case uniListA of
  838                     DefaultUniList uniA' -> case uniA `geq` uniA' of       -- [1]
  839                         Just Refl ->                                       -- [2]
  840                             pure . fromValueOf uniListA $ x : xs           -- [3]
  841                         _ -> throwing _StructuralUnliftingError
  842                             "The type of the value does not match the type of elements in the list"
  843                     _ -> throwing _StructuralUnliftingError "Expected a list but got something else"
  844             {-# INLINE mkConsDenotation #-}
  845         in makeBuiltinMeaning
  846             mkConsDenotation
  847             <costingFunction>
  848 
  849 Here we prepend an element to a list [3] after checking that the second argument is indeed a
  850 list [1] and that the type tag of the element being prepended equals the type tag for elements of
  851 the list [2] (extracted from the type tag for the whole list constant [1]).
  852 -}

  854 {- Note [Builtins and Plutus type checking]
  855 There's a direct correspondence between the Haskell type of the denotation of a builtin and the
  856 Plutus type of the builtin:
  857 
  858 1. elaboration turns a Haskell type variable into a concrete Haskell type representing a Plutus type
  859    variable, which later becomes demoted (in the regular @singletons@ sense via 'KnownSymbol' etc)
  860    to a regular Haskell value representing a Plutus type variable (as a part of the AST)
  861 2. a builtin head (i.e. a completely uninstantiated built-in type such as @Bool@ and @[]@) is
  862    considered abstract by the Plutus type checker. All the type checker cares about is being able to
  863    get the (Plutus) kind of a builtin head and check two builtin heads for equality
  864 3. Plutus type normalization tears partially or fully instantiated built-in types (such as
  865    @[Integer]@) apart and creates a Plutus type application for each Haskell type application
  866 4. 'BuiltinResult' does not appear on the Plutus side, since the logging and failure effects are
  867    implicit in Plutus as was discussed above
  868 5. 'Opaque' and 'SomeConstant' both carry a Haskell @rep@ type argument representing some Plutus
  869    type to be used for Plutus type checking
  870 
  871 This last part means that one can attach any (legal) @rep@ to an 'Opaque' or 'SomeConstant' and
  872 it'll be used by the Plutus type checker completely regardless of what the built-in function
  873 actually does. Let's look at some examples.
  874 
  875 1. The following built-in function unlifts to 'Bool' and lifts the result back:
  876 
  877     toBuiltinMeaning _ IdIntegerAsBool =
  878         let idIntegerAsBoolDenotation
  879                 :: SomeConstant uni Integer -> BuiltinResult (SomeConstant uni Integer)
  880             idIntegerAsBoolDenotation = \case
  881                 con@(SomeConstant (Some (ValueOf DefaultUniBool _))) -> pure con
  882                 _                                                    -> evaluationFailure
  883             {-# INLINE idIntegerAsBoolDenotation #-}
  884         in makeBuiltinMeaning
  885             idIntegerAsBoolDenotation
  886             <costingFunction>
  887 
  888 but on the Plutus side its type is
  889 
  890     integer -> integer
  891 
  892 because the @rep@ that 'SomeConstant' carries is 'Integer' in both the cases (in the type of the
  893 argument, as well as in the type of the result).
  894 
  895 This means that for this built-in function the Plutus type checker will accept a program that fails
  896 at runtime due to a type mismatch and will reject a program that runs successfully. Other built-in
  897 functions also can fail, e.g. the type of @ifThenElse@ says that the builtin expects a @Bool@ and
  898 feeding it something else will result in evaluation failure, but 'idIntegerAsBool' is different:
  899 it's respecting its type signature is what causes a failure, not disrespecting it.
  900 
  901 2. Another example of an unsafe built-in function is this one that checks whether an argument is a
  902 constant or not:
  903 
  904     toBuiltinMeaning _ IsConstant =
  905         let isConstantDenotation :: Opaque val a -> Bool
  906             isConstantDenotation = isRight . asConstant
  907             {-# INLINE isConstantDenotation #-}
  908         in makeBuiltinMeaning
  909             isConstantDenotation
  910             <costingFunction>
  911 
  912 Its type on the Plutus side is
  913 
  914     all a. a -> bool
  915 
  916 By parametricity any inhabitant of this type has to be either bottom or a function ignoring its
  917 argument, but @IsConstant@ actually uses the argument and so we break parametricity with this
  918 built-in function.
  919 
  920 3. Finally, we can have a Plutus version of @unsafeCoerce@:
  921 
  922     toBuiltinMeaning _ UnsafeCoerce =
  923         let unsafeCoerceDenotation :: Opaque val a -> Opaque val b
  924             unsafeCoerceDenotation = Opaque . unOpaque
  925             {-# INLINE unsafeCoerceDenotation #-}
  926         in makeBuiltinMeaning
  927             unsafeCoerceDenotation
  928             <costingFunction>
  929 
  930 Its type on the Plutus side is
  931 
  932     all a b. a -> b
  933 
  934 and thus this built-in function allows for viewing any Plutus expression as having an arbitrary
  935 type. Which is of course not nearly as bad as @unsafeCoerce@ in Haskell, because in Plutus a
  936 blob of memory representing an @Integer@ is not going to be viewed as a @[Bool]@ and an attempt to
  937 actually extract that @[Bool]@ will result in evaluation failure, but this built-in function is
  938 still not a good citizen of the Plutus type system.
  939 
  940 One could of course simply wrap Haskell's @unsafeCoerce@ as a built-in function in Plutus, but it
  941 goes without saying that this is not supposed to be done.
  942 
  943 So overall one needs to be very careful when defining built-in functions that have explicit
  944 'Opaque' and 'SomeConstant' arguments. Expressiveness doesn't come for free.
  945 
  946 Read Note [Pattern matching on built-in types] next.
  947 -}

  949 {- Note [Pattern matching on built-in types]
  950 Pattern matching over an enumeration built-in type ('Void', 'Unit', 'Bool' etc) is trivially
  951 implementable, see the 'IfThenElse' example in Note [How to add a built-in function: simple cases].
  952 Not so much for algebraic data types with at least one constructor carrying some kind of content.
  953 For example the @(:)@ constructor of @[a]@ has two arguments (an @a@ and a @[a]@) and all
  954 constructors of 'Data' carry something (e.g. 'I' carries an 'Integer' and 'Constr' carries an
  955 @Integer@ and a @[Data]@).
  956 
  957 In Haskell we'd represent the pattern matching function for lists as follows:
  958 
  959     caseList f z xs0 = case xs0 of
  960        []   -> z
  961        x:xs -> f x xs
  962 
  963 but in the denotation of a built-in function all those @f@, @z@ and @xs0@ are of type @val@, i.e.
  964 the type of values that the given evaluator uses (e.g. 'CkValue' for the CK machine or 'CekValue'
  965 for the CEK machine) and we don't know to apply one @val@ to another. We could try to constrain
  966 @val@ to implement some kind of "apply" function or try to somehow "parse" it into Haskell so that
  967 it becomes @val -> val@, but even if there was a way of doing either thing, it would be very
  968 complex and, more importantly, it's just not the job of the builtins machinery to evaluate such
  969 applications, it's what the actual evaluator is supposed to do.
  970 
  971 Hence we employ a very simple strategy: whenever we want to return an iterated application of a
  972 @val@ to a bunch of @val@s from a built-in function, we just construct it as is without trying to
  973 evaluate it and let the evaluator perform all the necessary reductions.
  974 
  975 So if you want to return an application from a built-in function, you need to use 'HeadSpine' at the
  976 type level and 'headSpine' at the term level, where the latter has the following signature:
  977 
  978     headSpine :: Opaque val asToB -> [val] -> Opaque (HeadSpine val) b
  979 
  980 'headSpine' takes the head of the application, i.e. a function from @a0@, @a1@ ... @an@ to @b@, and
  981 applies it to a list of values of respective types, returning a `b`. Whether types match or not is
  982 not checked, since that would be hard and largely pointless, so don't make mistakes.
  983 
  984 Back to the pattern matcher for lists:
  985 
  986     caseList f z xs0 = case xs0 of
  987        []   -> z
  988        x:xs -> f x xs
  989 
  990 Here's how we can define it as a built-in function using 'headSpine':
  991 
  992     toBuiltinMeaning _ver CaseList =
  993         let caseListDenotation
  994                 :: Opaque val b
  995                 -> Opaque val (a -> [a] -> b)
  996                 -> SomeConstant uni [a]
  997                 -> BuiltinResult (Opaque (HeadSpine val) b)
  998             caseListDenotation z f (SomeConstant (Some (ValueOf uniListA xs0))) = do
  999                 case uniListA of
 1000                     DefaultUniList uniA -> pure $ case xs0 of
 1001                         []     -> headSpine z []                                             -- [1]
 1002                         x : xs -> headSpine f [fromValueOf uniA x, fromValueOf uniListA xs]  -- [2]
 1003                     _ ->
 1004                         throwing _StructuralUnliftingError "Expected a list but got something else"
 1005             {-# INLINE caseListDenotation #-}
 1006         in makeBuiltinMeaning
 1007             caseListDenotation
 1008             <costingFunction>
 1009 
 1010 All the unlifting logic is the same as with, say, 'NullList' from
 1011 Note [How to add a built-in function: complicated cases], the only things that are different are [1]
 1012 and [2]. In [2] we have an iterated application of the given function @f@ to the head of the list
 1013 @x@ (lifted from a constant value to @val@ via 'fromValueOf') and the tail of the list @xs@ (lifted
 1014 to @val@ the same way). In [1] we return the given @z@, but since we need to return a 'HeadSpine'
 1015 (required by [2]), we have to use 'headSpine' just like in [2] except with an empty spine, since @z@
 1016 isn't applied to anything.
 1017 -}

 1019 {- Note [Representable built-in functions over polymorphic built-in types]
 1020 In Note [Pattern matching on built-in types] we discussed how general higher-order polymorphic
 1021 built-in functions are troubling, but polymorphic built-in functions can be troubling even in
 1022 the first-order case. In a Plutus program we always pair constants of built-in types with their
 1023 tags from the universe, which means that in order to produce a constant embedded into a program
 1024 we need the tag of the type of that constant. We can't get that tag from a Plutus type -- those
 1025 are gone at runtime, so the only place we can get a type tag from during evaluation is some already
 1026 existing constant. I.e. the following built-in function is representable:
 1027 
 1028     tail : all a. [a] -> [a]
 1029 
 1030 because for constructing the result we need a type tag for @[a]@, but we have a value of that type
 1031 as an argument and so we can extract the type tag from it. Same applies to
 1032 
 1033     swap : all a b. (a, b) -> (b, a)
 1034 
 1035 since 'SomeConstantOf' always contains a type tag for each type that a polymorphic built-in type is
 1036 instantiated with and so constructing a type tag for @(b, a)@ given type tags for @a@ and @b@ is
 1037 unproblematic.
 1038 
 1039 And so neither
 1040 
 1041     cons : all a. a -> [a] -> [a]
 1042 
 1043 is troubling (even though that ones requires checking at runtime that the element to be prepended
 1044 is of the same type as the type of the elements of the list as it's impossible to enforce this kind
 1045 of type safety in Haskell over possibly untyped PLC).
 1046 
 1047 However consider the following imaginary builtin:
 1048 
 1049     nil : all a. [a]
 1050 
 1051 we can't represent it for two reasons:
 1052 
 1053 1. we don't have any argument providing us a type tag for @a@ and hence we can't construct a type
 1054    tag for @[a]@
 1055 2. it would be a very unsound builtin to have. We can only instantiate built-in types with other
 1056    built-in types and so allowing @nil {some_non_built_in_type}@ would be a lie that couldn't reduce
 1057    to anything since it's not even possible to represent a built-in list with non-built-in elements
 1058    (even if there's zero of them)
 1059 
 1060 "Wait, but wouldn't @cons {some_non_built_in_type}@ be a lie as well?" -- No! Since @cons@ does not
 1061 just construct a list filled with elements of a non-built-in type but also expects one as an
 1062 argument and providing such an argument is impossible, 'cause it's pretty much the same thing as
 1063 populating 'Void' -- both values are equally unrepresentable. And so @cons {some_non_built_in_type}@
 1064 is a way to say @absurd@, which is perfectly fine to have.
 1065 
 1066 Finally,
 1067 
 1068     comma :: all a b. a -> b -> (a, b)
 1069 
 1070 is representable (because we can require arguments to be constants carrying universes with them,
 1071 which we can use to construct the resulting universe), but is still a lie, because instantiating
 1072 that builtin with non-built-in types is possible and so the PLC type checker won't throw on such
 1073 an instantiation, which will become 'evalutionFailure' at runtime the moment unlifting of a
 1074 non-constant is attempted when a constant is expected.
 1075 
 1076 So could we still get @nil@ or a safe version of @comma@ somehow? Well, we could have this
 1077 weirdness:
 1078 
 1079     nilOfTypeOf : all a. [a] -> [a]
 1080 
 1081 i.e. ask for an already existing list, but ignore the actual list and only use the type tag.
 1082 
 1083 But since we're ignoring the actual list, can't we just not pass it in the first place? And instead
 1084 pass around our good old friends, singletons. We should be able to do that, but it hasn't been
 1085 investigated. Perhaps something along the lines of adding the following constructor to 'DefaultUni':
 1086 
 1087     DefaultUniProtoSing :: DefaultUni (Esc (Proxy @GHC.Type))
 1088 
 1089 and then defining
 1090 
 1091     nil : all a. sing a -> [a]
 1092 
 1093 and then the Plutus Tx compiler can provide a type class or something for constructing singletons
 1094 for built-in types.
 1095 
 1096 This was investigated in https://github.com/IntersectMBO/plutus/pull/4337 but we decided not to
 1097 do it quite yet, even though it worked (the Plutus Tx part wasn't implemented).
 1098 -}

 2058     toBuiltinMeaning _semvar DropList =
 2059         let dropListDenotation
 2060                 :: IntegerCostedLiterally -> SomeConstant uni [a] -> BuiltinResult (Opaque val [a])
 2061             dropListDenotation i (SomeConstant (Some (ValueOf uniListA xs))) = do
 2062                 -- See Note [Operational vs structural errors within builtins].
 2063                 case uniListA of
 2064                     DefaultUniList _ ->
 2065                         -- The fastest way of dropping elements from a list is by operating on
 2066                         -- an unboxed int (i.e. an 'Int#'). We could implement that manually, but
 2067                         -- 'drop' in @Prelude@ already does that under the hood, so we just need to
 2068                         -- convert the given 'Integer' to an 'Int' and call 'drop' over that.
 2069                         fromValueOf uniListA <$> case unIntegerCostedLiterally i of
 2070                             IS i# -> pure $ drop (I# i#) xs
 2071                             IN _ -> pure xs
 2072                             -- If the given 'Integer' is higher than @maxBound :: Int@, then we
 2073                             -- call 'drop' over the latter instead to get the same performance as in
 2074                             -- the previous case. This will produce a different result when the
 2075                             -- list is longer than @maxBound :: Int@, but in practice not only is

 2392 {- Note [Legacy pattern matching on built-in types]
 2393 We used to only support direct pattern matching on enumeration types: 'Void', 'Unit', 'Bool'
 2394 etc. This is because it was impossible to return an iterated application from a built-in function.
 2395 
 2396 So e.g. if we wanted to add the following data type:
 2397 
 2398     newtype AnInt = AnInt Int
 2399 
 2400 as a built-in type, we wouldn't be able to add the following function as its pattern matcher:
 2401 
 2402     matchAnInt :: AnInt -> (Int -> r) -> r
 2403     matchAnInt (AnInt i) f = f i
 2404 
 2405 because we could not express the @f i@ part using the builtins machinery.
 2406 
 2407 But it still was possible to have @AnInt@ as a built-in type, it's just that instead of trying to
 2408 make its pattern matcher into a builtin we could have the following builtin:
 2409 
 2410     anIntToInt :: AnInt -> Int
 2411     anIntToInt (AnInt i) = i
 2412 
 2413 Although that was an annoyance for more complex data types. For tuples we needed to provide two
 2414 projection functions ('fst' and 'snd') instead of a single pattern matcher, which is not too bad,
 2415 but to get pattern matching on lists we needed a more complicated setup. For example originally we
 2416 would define three built-in functions: @null@, @head@ and @tail@, plus require `Bool` to be in the
 2417 universe, so that we can define an equivalent of
 2418 
 2419     matchList :: [a] -> r -> (a -> [a] -> r) -> r
 2420     matchList xs z f = if null xs then z else f (head xs) (tail xs)
 2421 
 2422 If a constructor stores more than one value, the corresponding projection function would pack them
 2423 into a (possibly nested) pair, for example for
 2424 
 2425     data Data
 2426         = Constr Integer [Data]
 2427         | <...>
 2428 
 2429 we had (pseudocode):
 2430 
 2431     unConstrData (Constr i ds) = (i, ds)
 2432 
 2433 and we still have that built-in function for backwards compatibility reasons.
 2434 
 2435 In order to get pattern matching over 'Data' we needed a projection function per constructor as well
 2436 as with lists, but writing (where the @Data@ suffix indicates that a function is a builtin that
 2437 somehow corresponds to a constructor of 'Data')
 2438 
 2439     if isConstrData d
 2440         then uncurry fConstr $ unConstrData d
 2441         else if isMapData d
 2442             then fMap $ unMapData d
 2443             else if isListData d
 2444                 then fList $ unListData d
 2445                 else <...>
 2446 
 2447 is tedious and inefficient and so instead we introduced a single @chooseData@ builtin that matches
 2448 on its @Data@ argument and chooses the appropriate branch (type instantiations and strictness
 2449 concerns are omitted for clarity):
 2450 
 2451      chooseData
 2452         (uncurry fConstr $ unConstrData d)
 2453         (fMap $ unMapData d)
 2454         (fList $ unListData d)
 2455         <...>
 2456         d
 2457 
 2458 which, for example, evaluates to @fMap es@ when @d@ is @Map es@
 2459 
 2460 We decided to handle lists the same way by using @chooseList@ rather than @null@ for consistency,
 2461 before introduction of pattern matching builtins.
 2462 -}
