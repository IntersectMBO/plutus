   54 {- Note [ExMemoryUsage instances for non-constants]
   55 In order to calculate the cost of a built-in function we need to feed the 'ExMemory' of each
   56 argument to the costing function associated with the builtin. For a polymorphic builtin this means
   57 that we need to be able to compute the 'ExMemory' of the AST provided as an argument to the builtin.
   58 How do we do that? Our strategy is:
   59 
   60 1. if the AST is a wrapped constant, then calculate the 'ExMemory' of the constant
   61 2. if the AST is something else, return 1
   62 
   63 This is pretty reasonable: a polymorphic builtin *is* allowed to check if the AST that it got as an
   64 argument is a constant or not, and if it happens to be a constant, the builtin *is* allowed to use
   65 it whatever way it wishes (see Note [Builtins and Plutus type checking] for details). Hence a
   66 builtin may in fact do something ad hoc for constants and we need to account for this possibility in
   67 the costing machinery.
   68 
   69 But if the given AST is not a constant, the builtin can't do anything else with it, hence we simply
   70 return 1, meaning "the costing function can't use this 'ExMemory' in any non-vacuous way".
   71 
   72 See 'HasMeaningIn' for a full list of constraints determining what a builtin can do with values.
   73 
   74 However for all types of values, except the one used by the production evaluator, we implement
   75 'ExMemoryUsage' as a call to 'error'. Not because other evaluators don't compute costs during
   76 evaluation -- the CK machine for example does in fact compute them (because we share the same
   77 builtins machinery between all the evaluators and we want it to be efficient on the production path,
   78 hence it's easier to optimize it for all evaluators than just for the single production evaluator).
   79 And not because the resulting 'ExBudget' is not forced by an evaluator that doesn't care about
   80 costing -- it still gets forced (for the same reason).
   81 
   82 The actual reason why we call 'error' is because at the moment no builtin is supposed to have a
   83 costing function that actually computes the 'ExMemory' of the given AST. Currently, if the builtin
   84 takes an 'Opaque', it's not supposed to actually look inside of it (unlike with 'SomeConstant') and
   85 hence the costing function is supposed to ignore that argument. It is possible that we'll eventually
   86 decide to add such a builtin, so the current approach of throwing an 'error' is a precaution
   87 ensuring that we won't add any weirdness by accident.
   88 
   89 We don't call 'error' on the production path, because we don't want this risk in there. A failing
   90 test is fine, a failing reasonable transaction is not and we don't want to risk it, even if it seems
   91 very unlikely that such a failure could slip in.
   92 
   93 The way we ignore arguments in costing functions is by computing the 'ExMemory' of each of those
   94 arguments lazily. I.e. a call to 'memoryUsage' can only be forced within a costing function and
   95 never outside of one. We have to do this regardless of all the reasoning above: if we compute
   96 the 'ExMemory' of, say, a list strictly, then a builtin prepending an element to a list will
   97 have the complexity of O(length_of_the_list) (because computing the 'ExMemory' of a list requires
   98 traversing the list), while we of course want it to be O(1).
   99 -}

  101 -- | A lazy tree of costs. Convenient for calculating the costs of values of built-in types, because
  102 -- they may have arbitrary branching (in particular a 'Data' object can contain a list of 'Data'
  103 -- objects inside of it).
  104 --
  105 -- 'CostRose' gets collapsed to a lazy linear structure down the pipeline, so that we can
  106 -- stream the costs to the outside where, say, the CEK machine picks them up one by one and handles
  107 -- somehow (in particular, subtracts from the remaining budget).
  108 data CostRose = CostRose {-# UNPACK #-} !CostingInteger ![CostRose]
  109     deriving stock (Show)
  110 
  111 -- | Create a 'CostRose' containing a single cost.
  112 singletonRose :: CostingInteger -> CostRose
  113 singletonRose cost = CostRose cost []
  114 {-# INLINE singletonRose #-}

  116 -- See Note [Global local functions].
  117 -- This is one way to define the worker. There are many more, see
  118 -- https://github.com/IntersectMBO/plutus/pull/5239#discussion_r1151197471
  119 -- We chose this one, because it's the simplest (no CPS shenanigans) among the safest (retrieving
  120 -- the next element takes O(1) time in the worst case).
  121 --
  122 -- The algorithm is a variation of the defunctionalization technique (see this post in particular:
  123 -- https://www.joachim-breitner.de/blog/778-Don%e2%80%99t_think,_just_defunctionalize), except we
  124 -- don't want a tail-recursive loop and instead emit costs lazily to the outside (as it's the whole
  125 -- point of the lazy costing approach)
  126 flattenCostRoseGo :: CostRose -> [CostRose] -> CostStream
  127 flattenCostRoseGo (CostRose cost1 forest1) forest2 =
  128     case forest1 of
  129         -- The current subtree doesn't have its own subtrees.
  130         [] -> case forest2 of
  131             -- No more elements in the entire tree, emit the last cost.
  132             []                -> CostLast cost1
  133             -- There's at least one unhandled subtree encountered earlier, emit the current cost
  134             -- and collapse the unhandled subtree.
  135             rose2' : forest2' -> CostCons cost1 $ flattenCostRoseGo rose2' forest2'
  136         -- The current subtree has at least one its own subtree.
  137         rose1' : forest1' ->
  138             -- Emit the current cost and handle the list of subtrees of the current subtree.
  139             CostCons cost1 $ case forest1' of
  140                 -- Same as the case below, except this one avoids creating a chain of
  141                 -- @[] ++ ([] ++ ([] ++ <...>))@, which would make retrieving the next element an
  142                 -- O(depth_of_the_tree) operation in the worst case.
  143                 [] -> flattenCostRoseGo rose1' forest2
  144                 -- Add the remaining subtrees of the current subtree (@forest1'@) to the stack of
  145                 -- all other subtrees (@forest2@) and handle the new current subtree (@rose1'@).
  146                 _  -> flattenCostRoseGo rose1' $ forest1' ++ forest2
  147 
  148 -- | Collapse a 'CostRose' to a lazy linear stream of costs. Retrieving the next element takes O(1)
  149 -- time in the worst case regardless of the recursion pattern of the given 'CostRose'.
  150 flattenCostRose :: CostRose -> CostStream
  151 flattenCostRose (CostRose cost [])              = CostLast cost
  152 flattenCostRose (CostRose cost (rose : forest)) = CostCons cost $ flattenCostRoseGo rose forest
  153 {-# INLINE flattenCostRose #-}

  159 {- Note [Alternative memory usage instances].  The `memoryUsage` function provides
  160  a measure of the size of an object for costing purposes, the idea being that
  161  the time taken to execute a builtin, and the memory used to contain its result,
  162  will depend on the size of the inputs.  The name `memoryUsage` is perhaps a
  163  misnomer: it was originally supposed to measure (in 64-bit words) the heap
  164  space required to store an object, but this is not always the correct measure
  165  to use.  For example, the time taken by `AddInteger` or `MultiplyInteger` will
  166  depend on the logarithms of the inputs (and the logarithm is proportional to
  167  the memory occupied by the inputs), and the memory occupied by the result will
  168  be some function of the memory occupied by the inputs, so for these functions
  169  the actual memory usage is a sensible size measure.  However, calling
  170  `replicateByte n b` function allocates a number of bytes which is equal to the
  171  actual value of `n`, which will be exponentially greater than the memory
  172  occupied by `n`, so this case the memory usage is not a sensible size measure.
  173  In most cases the default `memoryUsage` function returns the actual memory
  174  usage, but to deal with cases like `replicateByte` we occasionally use newtype
  175  wrappers which define a different size measure (see `IntegerCostedLiterally`
  176  below).  Polymorphic types require some care though: see Note [ExMemoryUsage
  177  for polymorphic types].
  178 -}
  179 
  180 {- Note [ExMemoryUsage for polymorphic types].  For polymorphic types such as
  181  `pair, `list`, and `array` DO NOT use newtype wrappers to define alternative
  182  size measures.  The denotations of functions which take polymorphic arguments
  183  use `SomeConstant` and this will ignore newtype wrappers and will only use the
  184  default `memoryUsage` function, which could lead to unexpected results.
  185  Furthermore, actual memory usage is typically not a good size measure for
  186  polymorphic arguments: the time taken to process a list, for example, will
  187  typically depend only on the length of the list and not the size of the
  188  contents.  Currently all such functions are parametrically polymorphic and only
  189  manipulate pointers without inspecting the contents of their polymorphic
  190  arguments, so it is reasonable to use size measures which depend only on the
  191  surface structure of polymorphic objects. -}
  192 
  193 {- We expect that all builtins which involve pairs will be constant cost and so
  194    their memory usage will never be involved in any computations.  The memory
  195    usage is set to maxBound so that we'll notice if this assumption is ever
  196    violated -}
  197 instance ExMemoryUsage (a, b) where
  198     memoryUsage _ = singletonRose maxBound
  199     {-# INLINE memoryUsage #-}

  201 {- Note the the `memoryUsage` of an empty list is zero.  This shouldn't cause any
  202  problems, but be sure to check that no costing function involving lists can
  203  return zero for an empty list (or any other input).
  204 -}
  205 {- Calculating the memory usage by processing the entire spine of the list eagerly
  206  is safe because there's no way to cheaply construct a long list: you either
  207  make one using repeated mkCons, which is expensive, or return one from a
  208  builtin, which has to be appropriately expensive too. -}
  209 instance ExMemoryUsage [a] where
  210   memoryUsage l = singletonRose . fromIntegral $ length l
  211   {-# INLINE memoryUsage #-}

  229 {- | Calculate a 'CostingInteger' for the size of the given 'Integer', measured in
  230 64-bit words.  This is the default size measure for `Integer`s.
  231 -}
  232 memoryUsageInteger :: Integer -> CostingInteger
  233 -- integerLog2# is unspecified for 0 (but in practice returns -1)
  234 -- ^ This changed with GHC 9.2: it now returns 0.  It's probably safest if we
  235 -- keep this special case for the time being though.
  236 memoryUsageInteger 0 = 1
  237 -- Assume 64 Int
  238 memoryUsageInteger i = fromIntegral $ I# (integerLog2# (abs i) `quotInt#` integerToInt 64) + 1
  239 -- So that the produced GHC Core doesn't explode in size, we don't win anything by inlining this
  240 -- function anyway.
  241 {-# OPAQUE memoryUsageInteger #-}

  244     memoryUsage i = singletonRose $ memoryUsageInteger i

  256 {- | When invoking a built-in function, a value of type `NumBytesCostedAsNumWords`
  257    can be used transparently as a built-in Integer but with a different size
  258    measure: see Note [Integral types as Integer].  This is required by the
  259    `integerToByteString` builtin, which takes an argument `w` specifying the
  260    width (in bytes) of the output bytestring (zero-padded to the desired size).
  261    The memory consumed by the function is given by `w`, *not* the size of `w`.
  262    The `NumBytesCostedAsNumWords` type wraps an Int `w` in a newtype whose
  263    `ExMemoryUsage` is equal to the number of eight-byte words required to
  264    contain `w` bytes, allowing its costing function to work properly.  We also
  265    use this for `replicateByte`.  If this is used to wrap an argument in the
  266    denotation of a builtin then it *MUST* also be used to wrap the same argument
  267    in the relevant budgeting benchmark.
  268 -}
  269 newtype NumBytesCostedAsNumWords = NumBytesCostedAsNumWords { unNumBytesCostedAsNumWords :: Integer }
  270 instance ExMemoryUsage NumBytesCostedAsNumWords where
  271     memoryUsage (NumBytesCostedAsNumWords n) = singletonRose . fromIntegral $ ((n-1) `div` 8) + 1
  272     {-# INLINE memoryUsage #-}

  274     -- maxBound::SatInt = 2^63-1.  This shouldn't be a problem for costing because no
  275     -- realistic input should be that large; however if you're going to use this then be
  276     -- sure to convince yourself that it's safe.

  294 {- Bytestrings: we want the empty bytestring and bytestrings of length 1-8 to have
  295    size 1, bytestrings of length 9-16 to have size 2, etc.  Note that (-1)
  296    `quot` 8 == 0, so the code below gives the correct answer for the empty
  297    bytestring.  -}
  298 instance ExMemoryUsage BS.ByteString where
  299     -- Don't use `div` here!  That gives 0 instead of 1 for the empty bytestring.
  300     memoryUsage bs = singletonRose . unsafeToSatInt $ ((n - 1) `quot` 8) + 1 where
  301         n = BS.length bs
  302     {-# INLINE memoryUsage #-}

  304 instance ExMemoryUsage T.Text where
  305     -- This says that @Text@ allocates 1 'CostingInteger' worth of memory (i.e. 8 bytes) per
  306     -- character, which is a conservative overestimate (i.e. is safe) regardless of whether @Text@
  307     -- is UTF16-based (like it used to when we implemented this instance) or UTF8-based (like it is
  308     -- now).
  309     --
  310     -- Note that the @ExMemoryUsage Char@ instance does not affect this one, this is for performance
  311     -- reasons, since @T.length@ is O(1) unlike @sum . map (memoryUsage @Char) . T.unpack@. We used
  312     -- to have the latter, but changed it to the former for easy performance gains.
  313     --
  314     -- We may want to make this a bit less of an overestimate in future just not to overcharge
  315     -- users.
  316     memoryUsage = singletonRose . fromIntegral . T.length

  332 -- | Add two 'CostRose's. We don't make this into a 'Semigroup' instance, because there exist
  333 -- different ways to add two 'CostRose's (e.g. we could optimize the case when one of the roses
  334 -- contains only one element or we can make the function lazy in the second argument). Here we chose
  335 -- the version that is most efficient when the first argument is a statically known constant (we
  336 -- didn't do any benchmarking though, so it may not be the most efficient one) as we need this
  337 -- below.
  338 addConstantRose :: CostRose -> CostRose -> CostRose
  339 addConstantRose (CostRose cost1 forest1) (CostRose cost2 forest2) =
  340     CostRose (cost1 + cost2) (forest1 ++ forest2)
  341 {-# INLINE addConstantRose #-}

  343 {- A naive traversal for size.  This accounts for the number of nodes in a Data
  344    object and also the sizes of the contents of the nodes.  This is not ideal,
  345    but it seems to be the best we can do.  At present this only comes into play
  346    for 'equalsData', which is implemented using the derived implementation of
  347    '==' (fortunately the costing functions are lazy, so this won't be called for
  348    things like 'unBData' which have constant costing functions because they only
  349    have to look at the top node).  The problem is that when we call 'equalsData'
  350    the comparison will take place entirely in Haskell, so the costing functions
  351    for the contents of 'I' and 'B' nodes won't be called.  Thus if we just
  352    counted the number of nodes the sizes of 'I 2' and 'B <huge bytestring>'
  353    would be the same but they'd take different amounts of time to compare.  It's
  354    not clear how to trade off the costs of processing a node and processing the
  355    contents of nodes: the implementation below compromises by charging four
  356    units per node, but we may wish to revise this after experimentation.
  357 -}
  358 instance ExMemoryUsage Data where
  359     memoryUsage = sizeData where
  360         dataNodeRose = singletonRose 4
  361         {-# INLINE dataNodeRose #-}
  362 
  363         sizeData d = addConstantRose dataNodeRose $ case d of
  364             -- TODO: include the size of the tag, but not just yet. See PLT-1176.
  365             Constr _ l -> CostRose 0 $ l <&> sizeData
  366             Map l      -> CostRose 0 $ l >>= \(d1, d2) -> [d1, d2] <&> sizeData
  367             List l     -> CostRose 0 $ l <&> sizeData
  368             I n        -> memoryUsage n
  369             B b        -> memoryUsage b
