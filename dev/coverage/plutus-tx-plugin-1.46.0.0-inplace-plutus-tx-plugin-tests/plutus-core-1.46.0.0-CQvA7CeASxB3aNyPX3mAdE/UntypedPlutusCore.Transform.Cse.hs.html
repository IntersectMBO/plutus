   34 {- Note [CSE]
   35 
   36 -------------------------------------------------------------------------------
   37 1. Simplifications
   38 -------------------------------------------------------------------------------
   39 
   40 This is a simplified (i.e., not fully optimal) implementation of CSE. The two simplifications
   41 we made are:
   42 
   43 - No alpha equivalence check, i.e., `\x -> x` and `\y -> y` are considered different expressions.
   44 - The builtin function arity information is approximate: rather than using the accurate arities,
   45   we simply use the maximum number of arguments applied to a builtin function in the program
   46   as the builtin function's arity. The arity information is used to determine whether a builtin
   47   application is possibly saturated.
   48 
   49 -------------------------------------------------------------------------------
   50 2. How does it work?
   51 -------------------------------------------------------------------------------
   52 
   53 We use the following example to explain how the implementation works:
   54 
   55 \x y -> (1+(2+x))
   56         +
   57         (case y [ (1+(2+x)) + (3+x)
   58                 , (2+x) + (3+x)
   59                 , 4+x
   60                 ]
   61         )
   62 
   63 The implementation makes several passes on the given term. The first pass collects builtin
   64 arity information as described above.
   65 
   66 In the second pass, we assign a unique ID to each `LamAbs`, `Delay`, and each `Case` branch.
   67 Then, we annotate each subterm with a path, consisting of IDs encountered from the root
   68 to that subterm (not including itself). The reason to do this is because `LamAbs`, `Delay`,
   69 and `Case` branches represent places where computation stops, i.e., subexpressions are not
   70 immediately evaluated, and may not be evaluated at all.
   71 
   72 In the above example, the ID of `\x` is 0, the ID of `\y` is 1, and the IDs of the
   73 three case branches are 2, 3, 4 (the actual numbers don't matter, as long as they are unique).
   74 The path for the first `1+(2+x)` and the first `2+x` is "0.1"; the path for the second
   75 `1+(2+x)` and the second `2+x` is "0.1.2"; the path for `4+x` is "0.1.4".
   76 
   77 In the third pass, we calculate a count for each `(term, path)` pair, where `term` is a
   78 non-workfree term, and `path` is its path. If the same term has two paths, and one is an
   79 ancestor (i.e., prefix) of the other, we increment the count for the ancestor path in both
   80 instances.
   81 
   82 In the above example, there are three occurrences of `2+x`, whose paths are "0.1", "0.1.2"
   83 and "0.1.3", respectively. The first path is an ancestor of the latter two. Therefore,
   84 the count for `(2+x, "0.1")` is 3, while the count for `(2+x, "0.1.2")` and `(2+x, "0.1.3")`
   85 is 0. The following all have a count of 1: `(3+x, "0.1.2")`, `(3+x, "0.1.3")` and
   86 `(4+x, "0.1.4")`.
   87 
   88 Now, each `(term, path)` pair whose count is greater than 1 is a CSE candidate.
   89 In the above example, the CSE candidates are `(2+x, "0.1")` and `(1+(2+x), "0.1")`.
   90 Note that `3+x` is not a CSE candidate, because it has two paths, and neither has a count
   91 greater than 1. `2+` is also not a CSE candidate, because it is workfree.
   92 
   93 The CSE candidates are then processed in descending order of their `termSize`s. For each CSE
   94 candidate, we generate a fresh variable, create a LamAbs for it under its path, and substitute
   95 it for all occurrences in the original term whose paths are descendents (or self) of
   96 the candidate's path. The order is because a bigger expression may contain a small subexpression.
   97 
   98 In the above example, we first process CSE candidate `(1+(2+x), "0.1")`. We create a fresh
   99 variable `cse1` for it, perform substitution, and create a `LamAbs` under path "0.1" (i.e., around
  100 the body of `y`). After processing this CSE candidate, the original term becomes
  101 
  102 \x y -> (\cse1 -> cse1
  103                   +
  104                   (case y [ cse1 + (3+x)
  105                           , (2+x) + (3+x)
  106                           , 4+x
  107                           ]
  108         ) (1+(2+x))
  109 
  110 The second CSE candidate is processed similarly, and the final result is
  111 
  112 \x y -> (\cse2 -> (\cse1 -> cse1
  113                             +
  114                             (case y [ cse1 + (3+x)
  115                                     , cse2 + (3+x)
  116                                     , 4+x
  117                                     ]
  118                   ) (1+cse2)
  119         ) (2+x)
  120 
  121 Here's another example:
  122 
  123 force (force ifThenElse
  124          (lessThanEqualsInteger 0 0)
  125          (delay ((1+2) + (1+2)))
  126          (delay (1+2))
  127       )
  128 
  129 In this case, the first two occurrences of `1+2` can be CSE'd, but the third occurrence
  130 can not. This is ensured by checking the path when substituting `cse1` for `1+2`. The result is
  131 
  132 force (force ifThenElse
  133          (lessThanEqualsInteger 0 0)
  134          (delay ((\cse1 -> cse1 + cse1) (1+2))
  135          (delay (1+2))
  136       )
  137 
  138 -------------------------------------------------------------------------------
  139 3. When should CSE run?
  140 -------------------------------------------------------------------------------
  141 
  142 CSE should run for multiple iterations, and should interleave with inlining. The following
  143 example illustrates why:
  144 
  145 \x ->
  146   f
  147     ((\y -> 1+(y+y)) (0+x))
  148     ((\z -> 2+(z+z)) (0+x))
  149 
  150 There is no inlining opportunity in this term. After the first iteration of CSE, where
  151 the common subepxression is `0+x`, we get:
  152 
  153 \x ->
  154   (\cse1 ->
  155     f
  156       ((\y -> 1+(y+y)) cse1)
  157       ((\z -> 2+(z+z)) cse1)
  158   ) (0+x)
  159 
  160 Now `y` and `z` can be inlined, after which we get
  161 
  162 \x ->
  163   (\cse1 ->
  164     f
  165       (1+(cse1+cse1))
  166       (2+(cse1+cse1))
  167   ) (0+x)
  168 
  169 Now there's a new common subexpression: `cse1+cse1`. So another iteration of CSE is
  170 needed, yielding:
  171 
  172 \x ->
  173   (\cse1 ->
  174     (\cse2 ->
  175       f
  176         (1+cse2)
  177         (2+cse2)
  178     ) (cse1+cse1)
  179   ) (0+x)
  180 
  181 With this example in mind, one may be tempted to make CSE part of the simplifier, and simply
  182 run it along with the rest of the simplifier. That is, however, a bad idea. CSE does the reverse
  183 of inlining; inlining tends to expose more optimization opportunities, and conversely, CSE
  184 tends to destroy optimization opportunities. Running CSE on a not-fully-optimized program
  185 may cause many optimization opportunities to be permanently lost. Give it a try if you want
  186 to see how bad it is!
  187 
  188 Therefore, this is what we do: first run the simplifier iterations. Then, run the CSE iterations,
  189 interleaving with the simplifier. For example, suppose max-simplifier-iterations-uplc=12, and
  190 max-cse-iterations=4. We first run 12 iterations of the simplifier, then run 4 iterations
  191 of CSE, with a simplifier pass after each iteration of CSE (i.e., the simplifier is run for a
  192 total of 16 times).
  193 
  194 Finally, since CSE can change the order or the number of occurrences of effects, it is only run
  195 when conservative optimization is off.
  196 -}

  198 -- | In reverse order, e.g., "1.2.3" is `[3, 2, 1]`.
  199 type Path = [Int]
  200 
  201 isAncestorOrSelf :: Path -> Path -> Bool
  202 isAncestorOrSelf = isSuffixOf

  204 data CseCandidate uni fun ann = CseCandidate
  205   { ccFreshName     :: Name
  206   , ccTerm          :: Term Name uni fun ()
  207   , ccAnnotatedTerm :: Term Name uni fun (Path, ann)
  208   -- ^ `ccTerm` is needed for equality comparison, while `ccAnnotatedTerm` is needed
  209   -- for the actual substitution. They are always the same term barring the annotations.
  210   }
  211 
  212 cse ::
  213   ( MonadQuote m
  214   , Hashable (Term Name uni fun ())
  215   , Rename (Term Name uni fun ann)
  216   , ToBuiltinMeaning uni fun
  217   ) =>
  218   BuiltinSemanticsVariant fun ->
  219   Term Name uni fun ann ->
  220   SimplifierT Name uni fun ann m (Term Name uni fun ann)
  221 cse builtinSemanticsVariant t0 = do
  222   t <- rename t0
  223   let annotated = annotate t
  224       commonSubexprs =
  225         -- Processed the common subexpressions in descending order of `termSize`.
  226         -- See Note [CSE].
  227         sortOn (Down . termSize)
  228           . fmap snd3
  229           -- A subexpression is common if the count is greater than 1.
  230           . filter ((> 1) . thd3)
  231           . join
  232           . Map.elems
  233           $ countOccs builtinSemanticsVariant annotated
  234   result <- mkCseTerm commonSubexprs annotated
  235   recordSimplification t0 CSE result
  236   return result
  237 
  238 -- | The second pass. See Note [CSE].
  239 annotate :: Term name uni fun ann -> Term name uni fun (Path, ann)
  240 annotate = flip evalState 0 . flip runReaderT [] . go
  241   where
  242     -- The integer state is the highest ID assigned so far.
  243     -- The reader context is the current path.
  244     go :: Term name uni fun ann -> ReaderT Path (State Int) (Term name uni fun (Path, ann))
  245     go t = do
  246       path <- ask
  247       case t of
  248         Apply ann fun arg -> Apply (path, ann) <$> go fun <*> go arg
  249         Force ann body -> Force (path, ann) <$> go body
  250         Constr ann i args -> Constr (path, ann) i <$> traverse go args
  251         Constant ann val -> pure $ Constant (path, ann) val
  252         Error ann -> pure $ Error (path, ann)
  253         Builtin ann fun -> pure $ Builtin (path, ann) fun
  254         Var ann name -> pure $ Var (path, ann) name
  255         LamAbs ann n body -> do
  256           freshId <- (+ 1) <$> lift get
  257           lift $ put freshId
  258           LamAbs (path, ann) n <$> local (freshId :) (go body)
  259         Delay ann body -> do
  260           freshId <- (+ 1) <$> lift get
  261           lift $ put freshId
  262           Delay (path, ann) <$> local (freshId :) (go body)
  263         Case ann scrut branches ->
  264           Case (path, ann)
  265             <$> go scrut
  266             <*> ( for branches $ \br -> do
  267                     freshId <- (+ 1) <$> lift get
  268                     lift $ put freshId
  269                     local (freshId :) (go br)
  270                 )
  271 
  272 -- | The third pass. See Note [CSE].
  273 countOccs ::
  274   forall name uni fun ann.
  275   (Hashable (Term name uni fun ()), ToBuiltinMeaning uni fun) =>
  276   BuiltinSemanticsVariant fun ->
  277   Term name uni fun (Path, ann) ->
  278   -- | Here, the value of the inner map not only contains the count, but also contains
  279   -- the annotated term, corresponding to the term that is the key of the outer map.
  280   -- The annotated terms need to be recorded since they will be used for substitution.
  281   HashMap (Term name uni fun ()) [(Path, Term name uni fun (Path, ann), Int)]
  282 countOccs builtinSemanticsVariant = foldrOf termSubtermsDeep addToMap Map.empty
  283   where
  284     addToMap ::
  285       Term name uni fun (Path, ann) ->
  286       HashMap (Term name uni fun ()) [(Path, Term name uni fun (Path, ann), Int)] ->
  287       HashMap (Term name uni fun ()) [(Path, Term name uni fun (Path, ann), Int)]
  288     addToMap t0
  289       -- We don't consider work-free terms for CSE, because doing so may or may not
  290       -- have a size benefit, but certainly doesn't have any cost benefit (the cost
  291       -- will in fact be slightly higher due to the additional application).
  292       | isWorkFree builtinSemanticsVariant t0
  293         || not (isBuiltinSaturated t0)
  294         || isForcingBuiltin t0 =
  295           id
  296       | otherwise =
  297           Map.alter
  298             ( \case
  299                 Nothing -> Just [(path, t0, 1)]
  300                 Just paths -> Just $ combinePaths t0 path paths
  301             )
  302             t
  303       where
  304         t = void t0
  305         path = fst (termAnn t0)
  306 
  307     isBuiltinSaturated =
  308       splitApplication >>> \case
  309         (Builtin _ fun, args) ->
  310           length args >= length (builtinArity (Proxy @uni) builtinSemanticsVariant fun)
  311         _term -> True
  312 
  313     isForcingBuiltin = \case
  314       Builtin{} -> True
  315       Force _ t -> isForcingBuiltin t
  316       _ -> False

  318 -- | Combine a new path with a number of existing (path, count) pairs.
  319 combinePaths ::
  320   forall name uni fun ann.
  321   Term name uni fun (Path, ann) ->
  322   Path ->
  323   [(Path, Term name uni fun (Path, ann), Int)] ->
  324   [(Path, Term name uni fun (Path, ann), Int)]
  325 combinePaths t path = go 1
  326   where
  327     go ::
  328       Int ->
  329       [(Path, Term name uni fun (Path, ann), Int)] ->
  330       [(Path, Term name uni fun (Path, ann), Int)]
  331     -- The new path is not a descendent-or-self of any existing path.
  332     go acc [] = [(path, t, acc)]
  333     go acc ((path', t', cnt) : paths)
  334       -- The new path is an ancestor-or-self of an existing path.
  335       -- Take over all counts of the existing path, remove the existing path,
  336       -- and continue.
  337       | path `isAncestorOrSelf` path' = go (acc + cnt) paths
  338       -- The new path is a descendent-or-self of an existing path.
  339       -- Increment the count for the existing path. There can only be one such
  340       -- existing path, so we don't need to recurse here.
  341       | path' `isAncestorOrSelf` path = (path', t', cnt + 1) : paths
  342       | otherwise = (path', t', cnt) : go acc paths

  344 mkCseTerm ::
  345   forall uni fun ann m.
  346   (MonadQuote m, Eq (Term Name uni fun ())) =>
  347   [Term Name uni fun (Path, ann)] ->
  348   -- | The original annotated term
  349   Term Name uni fun (Path, ann) ->
  350   m (Term Name uni fun ann)
  351 mkCseTerm ts t = do
  352   cs <- traverse mkCseCandidate ts
  353   pure . fmap snd $ Foldable.foldl' (flip applyCse) t cs
  354 
  355 applyCse ::
  356   forall uni fun ann.
  357   (Eq (Term Name uni fun ())) =>
  358   CseCandidate uni fun ann ->
  359   Term Name uni fun (Path, ann) ->
  360   Term Name uni fun (Path, ann)
  361 applyCse c = mkLamApp . transformOf termSubterms substCseVarForTerm
  362   where
  363     candidatePath = fst (termAnn (ccAnnotatedTerm c))
  364 
  365     substCseVarForTerm :: Term Name uni fun (Path, ann) -> Term Name uni fun (Path, ann)
  366     substCseVarForTerm t =
  367       if currTerm == ccTerm c && candidatePath `isAncestorOrSelf` currPath
  368         then Var (termAnn t) (ccFreshName c)
  369         else t
  370       where
  371         currTerm = void t
  372         currPath = fst (termAnn t)
  373 
  374     mkLamApp :: Term Name uni fun (Path, ann) -> Term Name uni fun (Path, ann)
  375     mkLamApp t
  376       | currPath == candidatePath =
  377           Apply
  378             (termAnn t)
  379             (LamAbs (termAnn t) (ccFreshName c) t)
  380             (ccAnnotatedTerm c)
  381       | currPath `isAncestorOrSelf` candidatePath = case t of
  382           Var ann name            -> Var ann name
  383           LamAbs ann name body    -> LamAbs ann name (mkLamApp body)
  384           Apply ann fun arg       -> Apply ann (mkLamApp fun) (mkLamApp arg)
  385           Force ann body          -> Force ann (mkLamApp body)
  386           Delay ann body          -> Delay ann (mkLamApp body)
  387           Constant ann val        -> Constant ann val
  388           Builtin ann fun         -> Builtin ann fun
  389           Error ann               -> Error ann
  390           Constr ann i ts         -> Constr ann i (mkLamApp <$> ts)
  391           Case ann scrut branches -> Case ann (mkLamApp scrut) (mkLamApp <$> branches)
  392       | otherwise = t
  393       where
  394         currPath = fst (termAnn t)

  396 -- | Generate a fresh variable for the common subexpression.
  397 mkCseCandidate ::
  398   forall uni fun ann m.
  399   (MonadQuote m) =>
  400   Term Name uni fun (Path, ann) ->
  401   m (CseCandidate uni fun ann)
  402 mkCseCandidate t = CseCandidate <$> freshName "cse" <*> pure (void t) <*> pure t
