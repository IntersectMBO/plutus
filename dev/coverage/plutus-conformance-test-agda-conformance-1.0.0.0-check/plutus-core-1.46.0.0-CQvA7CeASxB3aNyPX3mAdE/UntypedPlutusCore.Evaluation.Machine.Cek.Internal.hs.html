  109 {- Note [Compilation peculiarities]
  110 READ THIS BEFORE TOUCHING ANYTHING IN THIS FILE
  111 
  112 Don't use @StrictData@, it makes the machine slower by several percent.
  113 
  114 Exporting the 'computeCek' function from this module causes the CEK machine to become slower by
  115 up to 25%. I repeat: just adding 'computeCek' to the export list makes the evaluator substantially
  116 slower. The reason for this is that with 'computeCek' exported the generated GHC Core is much worse:
  117 it contains more lambdas, allocates more stuff etc. While perhaps surprising, this is not an
  118 unusual behavior of the compiler as https://wiki.haskell.org/Performance/GHC explains:
  119 
  120 > Indeed, generally speaking GHC will inline across modules just as much as it does within modules,
  121 > with a single large exception. If GHC sees that a function 'f' is called just once, it inlines it
  122 > regardless of how big 'f' is. But once 'f' is exported, GHC can never see that it's called exactly
  123 > once, even if that later turns out to be the case. This inline-once optimisation is pretty
  124 > important in practice.
  125 >
  126 > So: if you care about performance, do not export functions that are not used outside the module
  127 > (i.e. use an explicit export list, and keep it as small as possible).
  128 
  129 Now clearly 'computeCek' cannot be inlined in 'runCek' whether it's exported or not, since
  130 'computeCek' is recursive. However:
  131 
  132 1. GHC is _usually_ smart enough to perform the worker/wrapper transformation and inline the wrapper
  133    (however experiments have shown that sticking the internals of the CEK machine, budgeting modes
  134    and the API into the same file prevents GHC from performing the worker/wrapper transformation for
  135    some reason likely related to "we've been compiling this for too long, let's leave it at that"
  136 2. GHC seems to be able to massage the definition of 'computeCek' into something more performant
  137    making use of knowing exactly how 'computeCek' is used, essentially tailoring the definition of
  138    'computeCek' for a particular invocation in 'runCek'
  139 
  140 Hence we don't export 'computeCek' and instead define 'runCek' in this file and export it, even
  141 though the rest of the user-facing API (which 'runCek' is a part of) is defined downstream.
  142 
  143 Another problem is handling mutual recursion in the 'computeCek'/'returnCek'/'forceEvaluate'/etc
  144 family. If we keep these functions at the top level, GHC won't be able to pull the constraints out
  145 of the family (confirmed by inspecting Core: GHC thinks that since the superclass constraints
  146 populating the dictionary representing the @Ix fun@ constraint are redundant, they can be replaced
  147 with calls to 'error' in a recursive call, but that changes the dictionary and so it can no longer
  148 be pulled out of recursion). But that entails passing a redundant argument around, which slows down
  149 the machine a tiny little bit.
  150 
  151 Hence we define a all happy-path functions having CEK-machine-specific constraints as local
  152 functions making use of a shared context from their parent function. This also allows GHC to inline
  153 almost all of the machine into a single definition (with a bunch of recursive join points in it).
  154 
  155 In general, it's advised to run benchmarks (and look at Core output if the results are suspicious)
  156 on any changes in this file.
  157 
  158 Finally, it's important to put bang patterns on any 'Int' arguments to ensure that GHC unboxes them:
  159 this can make a surprisingly large difference.
  160 -}

  221 -- 'Values' for the modified CEK machine.
  222 data CekValue uni fun ann =
  223     -- This bang gave us a 1-2% speed-up at the time of writing.
  224     VCon !(Some (ValueOf uni))
  225   | VDelay !(NTerm uni fun ann) !(CekValEnv uni fun ann)
  226   | VLamAbs !NamedDeBruijn !(NTerm uni fun ann) !(CekValEnv uni fun ann)
  227     -- | A partial builtin application, accumulating arguments for eventual full application.
  228     -- We don't need a 'CekValEnv' here unlike in the other constructors, because 'VBuiltin'
  229     -- values always store their corresponding 'Term's fully discharged, see the comments at
  230     -- the call sites (search for 'VBuiltin').
  231   | VBuiltin
  232       !fun
  233       -- ^ So that we know, for what builtin we're calculating the cost. We can sneak this into
  234       -- 'BuiltinRuntime', so that we don't need to store it here, but somehow doing so was
  235       -- consistently slowing evaluation down by half a percent. Might be noise, might be not, but
  236       -- at least we know that removing this @fun@ is not helpful anyway. See this commit reversing
  237       -- the change: https://github.com/IntersectMBO/plutus/pull/4778/commits/86a3e24ca3c671cc27c6f4344da2bcd14f961706
  238       (NTerm uni fun ())
  239       -- ^ This must be lazy. It represents the fully discharged partial application of the builtin
  240       -- function that we're going to run when it's fully saturated.  We need the 'Term' to be able
  241       -- to return it in case full saturation is never achieved and a partial application needs to
  242       -- be returned in the result. The laziness is important, because the arguments are discharged
  243       -- values and discharging is expensive, so we don't want to do it unless we really have
  244       -- to. Making this field strict resulted in a 3-4.5% slowdown at the time of writing.
  245       !(BuiltinRuntime (CekValue uni fun ann))
  246       -- ^ The partial application and its costing function.
  247       -- Check the docs of 'BuiltinRuntime' for details.
  248     -- | A constructor value, including fully computed arguments and the tag.
  249   | VConstr {-# UNPACK #-} !Word64 !(ArgStack uni fun ann)
  250 
  251 deriving stock instance (GShow uni, Everywhere uni Show, Show fun, Show ann, Closed uni)
  252     => Show (CekValue uni fun ann)

  279 {- Note [Cost slippage]
  280 Tracking the budget usage for every step in the machine adds a lot of overhead. To reduce this,
  281 we adopt a technique which allows some overshoot of the budget ("slippage"), but only a bounded
  282 amount.
  283 
  284 To do this we:
  285 - Track all the machine steps of all kinds in an set of counters in a 'StepCounter'
  286 - Actually "spend" the budget when we've done more than some fixed number of steps, or at the end.
  287 
  288 This saves a *lot* of time, at the cost of potentially overshooting the budget by slippage*step_cost,
  289 which is okay so long as we bound the slippage appropriately.
  290 
  291 Note that we're only proposing to do this for machine steps, since it's plausible that we can track
  292 them in an optimized way. Builtins are more complicated (and infrequent), so we can just budget them
  293 properly when we hit them.
  294 
  295 There are two options for how to bound the slippage:
  296 1. As a fixed number of steps
  297 2. As a proportion of the overall budget
  298 
  299 Option 2 initially seems much better as a bound: if we run N scripts with an overall budget of B, then
  300 the potential overrun from 1 is N*slippage, whereas the overrun from 2 is B*slippage. That is, 2
  301 says we always overrun by a fraction of the total amount of time you were expecting, whereas 1 says
  302 it depends how many scripts you run... so if I send you a lot of small scripts, I could cause a lot
  303 of overrun.
  304 
  305 However, it turns out (empirically) that we can pick a number for 1 that gives us most of the speedup, but such
  306 that the maximum overrun is negligible (e.g. much smaller than the "startup cost"). So in the end
  307 we opted for option 1, which also happens to be simpler to implement.
  308 -}

  310 {- Note [Structure of the step counter]
  311 The step counter is kept in a 'MutablePrimArray', which is a fast way of storing a bunch of
  312 mutable 'Word8's.
  313 This suits our purposes, as we need one counter for every step type, and one for the total number.
  314 
  315 We keep the counters for each step in the first indices, so we can index them simply by using
  316 the 'Enum' instance of 'StepKind', and the total counter in the last index.
  317 -}
  318 
  319 -- So that we don't need to update 'NumberOfStepCounters' manually, which would be extremely
  320 -- error-prone and has caused a bug in the past.
  321 type CountConstructorsEnum :: (GHC.Type -> GHC.Type) -> Nat
  322 type family CountConstructorsEnum rep where
  323     CountConstructorsEnum U1         = 1
  324     CountConstructorsEnum (M1 _ _ f) = CountConstructorsEnum f
  325     CountConstructorsEnum (f :+: g)  = CountConstructorsEnum f + CountConstructorsEnum g
  326     CountConstructorsEnum V1         = TypeError ('Text "Cannot be empty")
  327     CountConstructorsEnum (f :*: g)  = TypeError ('Text "Cannot be a non-enumeration type")
  328     CountConstructorsEnum (K1 _ _)   = TypeError ('Text "Cannot be a non-enumeration type")
  329     CountConstructorsEnum (Rec1 _)   = TypeError ('Text "Cannot be a non-enumeration type")
  330     CountConstructorsEnum Par1       =
  331         TypeError ('Text "If you really want a parameterized type, handle this clause")
  332 
  333 -- | The number of step counters that we need, should be the same as the number of constructors
  334 -- of 'StepKind'.
  335 type NumberOfStepCounters = CountConstructorsEnum (Rep StepKind)
  336 -- | The total number of counters that we need, one extra for the total counter.
  337 -- See Note [Structure of the step counter]
  338 type CounterSize = NumberOfStepCounters + 1
  339 -- | The index at which the total step counter is kept.
  340 -- See Note [Structure of the step counter]
  341 type TotalCountIndex = NumberOfStepCounters

  372 {- Note [Implicit parameters in the machine]
  373 The traditional way to pass context into a function is to use 'ReaderT'. However, 'ReaderT' has some
  374 disadvantages.
  375 - It requires threading through the context even where you don't need it (every monadic bind)
  376 - It *can* often be optimized away, but this requires GHC to be somewhat clever and do a lot of
  377   case-of-case to lift all the arguments out.
  378 
  379 Moreover, if your context is global (i.e. constant across the lifetime of the monad, i.e. you don't
  380 need 'local'), then you're buying some extra power (the ability to pass in a different context somewhere
  381 deep inside the computation) which you don't need.
  382 
  383 There are three main alternatives:
  384 - Explicit function parameters. Simple, doesn't get tied up in the Monad operations, *does* still
  385 present the appearance of letting you do 'local'. But a bit cluttered.
  386 - Implicit parameters. A bit esoteric, can be bundled up into a constraint synonym and just piped to
  387 where they're needed, essentially the same as explicit parameters in terms of runtime.
  388 - Constraints via 'reflection'. Quite esoteric, *does* get you global parameters (within their scope),
  389 bit of a hassle threading around all the extra type parameters.
  390 
  391 We're using implicit parameters for now, which seems to strike a good balance of speed and convenience.
  392 I haven't tried 'reflection' in detail, but I believe the main thing it would do is to make the parameters
  393 global - but we already have this for most of the hot functions by making them all local definitions, so
  394 they don't actually take the context as an argument even at the source level.
  395 -}

  426 {- Note [Throwing exceptions in ST]
  427 This note represents MPJ's best understanding right now, might be wrong.
  428 
  429 We use a moderately evil trick to throw exceptions in ST, but unlike the evil trick for catching them, it's hidden.
  430 
  431 The evil is that the 'MonadThrow' instance for 'ST' uses 'unsafeIOToST . throwIO'! Sneaky! The author has marked it
  432 "Trustworthy", no less. However, I believe this to be safe for basically the same reasons as our trick to catch
  433 exceptions is safe, see Note [Catching exceptions in ST]
  434 -}
  435 
  436 {- Note [Catching exceptions in ST]
  437 This note represents MPJ's best understanding right now, might be wrong.
  438 
  439 We use a moderately evil trick to catch exceptions in ST. This uses the unsafe ST <-> IO conversion functions to go into IO,
  440 catch the exception, and then go back into ST.
  441 
  442 Why is this okay? Recall that IO ~= ST RealWorld, i.e. it is just ST with a special thread token. The unsafe conversion functions
  443 just coerce from one to the other. So the thread token remains the same, it's just that we'll potentially leak it from ST, and we don't
  444 get ordering guarantees with other IO actions.
  445 
  446 But in our case this is okay, because:
  447 
  448 1. We do not leak the original ST thread token, since we only pass it into IO and then immediately back again.
  449 2. We don't have ordering guarantees with other IO actions, but we don't care because we don't do any side effects, we only catch a single kind of exception.
  450 3. We *do* have ordering guarantees between the throws inside the ST action and the catch, since they are ultimately using the same thread token.
  451 -}

  463 instance ThrowableBuiltins uni fun =>
  464         MonadError (CekEvaluationException NamedDeBruijn uni fun) (CekM uni fun s) where
  465     -- See Note [Throwing exceptions in ST].
  466     throwError = CekM . throwM
  467 
  468     -- See Note [Catching exceptions in ST].
  469     catchError
  470         :: forall a.
  471            CekM uni fun s a
  472         -> (CekEvaluationException NamedDeBruijn uni fun -> CekM uni fun s a)
  473         -> CekM uni fun s a
  474     a `catchError` h =
  475         -- Here in addition to catching 'CekEvaluationException' we also catch common GHC exceptions
  476         -- in case one of them somehow gets triggered during script execution (which would be a bug
  477         -- on our side). We could probably use @enclosed-exceptions@, but spawning a thread per
  478         -- script is expensive. We could also use type-based disambiguation like @unliftio@ does,
  479         -- but it fails if an exception whose type indicates that it's a sync one gets thrown in an
  480         -- async way.
  481         -- Alexey Kuleshevich told us that the node catches exceptions anyway, so what we're doing
  482         -- here is for easing debugging and error reporting, it's not a proper safety measure. Hence
  483         -- catching several common exception types is enough.
  484         CekM . unsafeIOToST $ aIO `catches`
  485             [ Handler hIO
  486             , panicHandler @IOError
  487             , panicHandler @Exception.ErrorCall
  488             , panicHandler @Exception.ArithException
  489             , panicHandler @Exception.ArrayException
  490             ]
  491       where
  492         aIO = unsafeRunCekM a
  493         hIO = unsafeRunCekM . h
  494 
  495         -- Unsafely run a 'CekM' computation in the 'IO' monad by converting the underlying 'ST' to
  496         -- it.
  497         unsafeRunCekM :: CekM uni fun s a -> IO a
  498         unsafeRunCekM = unsafeSTToIO . unCekM
  499 
  500         panicHandler :: forall e. Exception e => Handler IO a
  501         panicHandler =
  502             Handler $ \(err :: e) -> hIO $
  503                 ErrorWithCause

  523 -- | Instantiate all the free variables of a term by looking them up in an environment.
  524 -- Mutually recursive with dischargeCekVal.
  525 dischargeCekValEnv :: forall uni fun ann. CekValEnv uni fun ann -> NTerm uni fun () -> NTerm uni fun ()
  526 dischargeCekValEnv valEnv = go 0
  527  where
  528   -- The lamCnt is just a counter that measures how many lambda-abstractions
  529   -- we have descended in the `go` loop.
  530   go :: Word64 -> NTerm uni fun () -> NTerm uni fun ()
  531   go !lamCnt =  \case
  532     LamAbs ann name body -> LamAbs ann name $ go (lamCnt+1) body
  533     var@(Var _ (NamedDeBruijn _ ndbnIx)) -> let idx = coerce ndbnIx :: Word64  in
  534         if lamCnt >= idx
  535         -- the index n is less-than-or-equal than the number of lambdas we have descended
  536         -- this means that n points to a bound variable, so we don't discharge it.
  537         then var
  538         else maybe
  539                -- var is free, leave it alone
  540                var
  541                -- var is in the env, discharge its value
  542                dischargeCekValue
  543                -- index relative to (as seen from the point of view of) the environment
  544                (Env.indexOne valEnv $ idx - lamCnt)
  545     Apply ann fun arg    -> Apply ann (go lamCnt fun) $ go lamCnt arg
  546     Delay ann term       -> Delay ann $ go lamCnt term
  547     Force ann term       -> Force ann $ go lamCnt term
  548     t -> t

  550 -- | Convert a 'CekValue' into a 'Term' by replacing all bound variables with the terms
  551 -- they're bound to (which themselves have to be obtain by recursively discharging values).
  552 dischargeCekValue :: CekValue uni fun ann -> NTerm uni fun ()
  553 dischargeCekValue = \case
  554     VCon     val                         -> Constant () val
  555     VDelay   body env                    -> dischargeCekValEnv env $ Delay () (void body)
  556     -- 'computeCek' turns @LamAbs _ name body@ into @VLamAbs name body env@ where @env@ is an
  557     -- argument of 'computeCek' and hence we need to start discharging outside of the reassembled
  558     -- lambda, otherwise @name@ could clash with the names that we have in @env@.
  559     VLamAbs (NamedDeBruijn n _ix) body env ->
  560         -- The index on the binder is meaningless, we put `0` by convention, see 'Binder'.
  561         dischargeCekValEnv env $ LamAbs () (NamedDeBruijn n deBruijnInitIndex) (void body)
  562     -- We only return a discharged builtin application when (a) it's being returned by the machine,
  563     -- or (b) it's needed for an error message.
  564     -- @term@ is fully discharged, so we can return it directly without any further discharging.
  565     VBuiltin _ term _                    -> term
  566     VConstr i es                         -> Constr () i (fmap dischargeCekValue $ stack2list es)
  567       where
  568         stack2list = go []
  569         go acc EmptyStack           = acc
  570         go acc (ConsStack arg rest) = go (arg : acc) rest

  585 {-|
  586 The context in which the machine operates.
  587 
  588 Morally, this is a stack of frames, but we use the "intrusive list" representation so that
  589 we can match on context and the top frame in a single, strict pattern match.
  590 -}
  591 data Context uni fun ann
  592     = FrameAwaitArg !(CekValue uni fun ann) !(Context uni fun ann)
  593     -- ^ @[V _]@
  594     | FrameAwaitFunTerm !(CekValEnv uni fun ann) !(NTerm uni fun ann) !(Context uni fun ann)
  595     -- ^ @[_ N]@
  596     | FrameAwaitFunValue !(CekValue uni fun ann) !(Context uni fun ann)
  597     -- ^ @[_ V]@
  598     | FrameForce !(Context uni fun ann)
  599     -- ^ @(force _)@
  600     -- See Note [Accumulators for terms]
  601     | FrameConstr !(CekValEnv uni fun ann) {-# UNPACK #-} !Word64 ![NTerm uni fun ann] !(ArgStack uni fun ann) !(Context uni fun ann)
  602     -- ^ @(constr i V0 ... Vj-1 _ Nj ... Nn)@
  603     | FrameCases !(CekValEnv uni fun ann) !(V.Vector (NTerm uni fun ann)) !(Context uni fun ann)
  604     -- ^ @(case _ C0 .. Cn)@
  605     | NoFrame
  606 
  607 deriving stock instance (GShow uni, Everywhere uni Show, Show fun, Show ann, Closed uni)
  608     => Show (Context uni fun ann)

  612     memoryUsage = \case
  613         VCon c      -> memoryUsage c
  614         VDelay {}   -> singletonRose 1

  616         VBuiltin {} -> singletonRose 1

  620 {- Note [ArgStack vs Spine]
  621 We use 'ArgStack' for collecting the arguments of a constructor to later pass it to the function in
  622 the appropriate branch. Originally, all arguments of a constructor are terms and hence before we can
  623 pass them to a function they need to be evaluated to values, which means that in case of the CEK
  624 machine the evaluated arguments are going to be reversed: you evaluate the first argument and put
  625 the result into a 'FrameConstr', then the second one and put it in a 'FrameConstr' again, this time
  626 prepending it to the one that is already there etc -- in the end you get the arguments in reversed
  627 order. Which is why 'transferArgStack' is a left fold (just like 'reverse').
  628 
  629 But in case of 'Spine' the builtins machinery directly produces values, not terms. Meaning, a
  630 'Spine' that we get from the builtins machinery isn't reversed, hence we can pass its contents
  631 directly to the head of the application. Which is why 'transferSpine' is a right fold.
  632 -}

  650     :: forall a cost uni fun ann
  651     . ThrowableBuiltins uni fun
  652     => MachineParameters CekMachineCosts fun (CekValue uni fun ann)

  654     -> EmitterMode uni fun
  655     -> (forall s. GivenCekReqs uni fun ann s => CekM uni fun s a)

  657 runCekM (MachineParameters costs runtime) (ExBudgetMode getExBudgetInfo) (EmitterMode getEmitterMode) a = runST $ do

  673 -- See Note [Compilation peculiarities].
  674 -- | The entering point to the CEK machine's engine.
  675 enterComputeCek
  676     :: forall uni fun ann s
  677     . (ThrowableBuiltins uni fun, GivenCekReqs uni fun ann s)
  678     => Context uni fun ann
  679     -> CekValEnv uni fun ann
  680     -> NTerm uni fun ann
  681     -> CekM uni fun s (NTerm uni fun ())
  682 enterComputeCek = computeCek
  683   where
  684     -- | The computing part of the CEK machine.
  685     -- Either
  686     -- 1. adds a frame to the context and calls 'computeCek' ('Force', 'Apply')
  687     -- 2. calls 'returnCek' on values ('Delay', 'LamAbs', 'Constant', 'Builtin')
  688     -- 3. throws 'EvaluationFailure' ('Error')
  689     -- 4. looks up a variable in the environment and calls 'returnCek' ('Var')
  690     computeCek
  691         :: Context uni fun ann
  692         -> CekValEnv uni fun ann
  693         -> NTerm uni fun ann
  694         -> CekM uni fun s (NTerm uni fun ())
  695     -- s ; ρ ▻ {L A}  ↦ s , {_ A} ; ρ ▻ L
  696     computeCek !ctx !env (Var _ varName) = do
  697         stepAndMaybeSpend BVar
  698         val <- lookupVarName varName env
  699         returnCek ctx val
  700     -- s ; ρ ▻ con c  ↦  s ◅ con c
  701     computeCek !ctx !_ (Constant _ val) = do
  702         stepAndMaybeSpend BConst
  703         returnCek ctx (VCon val)
  704     -- s ; ρ ▻ lam x L  ↦  s ◅ lam x (L , ρ)
  705     computeCek !ctx !env (LamAbs _ name body) = do
  706         stepAndMaybeSpend BLamAbs
  707         returnCek ctx (VLamAbs name body env)
  708     -- s ; ρ ▻ delay L  ↦  s ◅ delay (L , ρ)
  709     computeCek !ctx !env (Delay _ body) = do
  710         stepAndMaybeSpend BDelay
  711         returnCek ctx (VDelay body env)
  712     -- s ; ρ ▻ force T  ↦  s , force _ ; ρ ▻ L
  713     computeCek !ctx !env (Force _ body) = do
  714         stepAndMaybeSpend BForce
  715         computeCek (FrameForce ctx) env body
  716     -- s ; ρ ▻ [L M]  ↦  s , [_ (M,ρ)]  ; ρ ▻ L
  717     computeCek !ctx !env (Apply _ fun arg) = do
  718         stepAndMaybeSpend BApply
  719         computeCek (FrameAwaitFunTerm env arg ctx) env fun
  720     -- s ; ρ ▻ builtin bn  ↦  s ◅ builtin bn arity arity [] [] ρ
  721     computeCek !ctx !_ (Builtin _ bn) = do
  722         stepAndMaybeSpend BBuiltin
  723         let meaning = lookupBuiltin bn ?cekRuntime
  724         -- 'Builtin' is fully discharged.
  725         returnCek ctx (VBuiltin bn (Builtin () bn) meaning)
  726     -- s ; ρ ▻ constr I T0 .. Tn  ↦  s , constr I _ (T1 ... Tn, ρ) ; ρ ▻ T0
  727     computeCek !ctx !env (Constr _ i es) = do
  728         stepAndMaybeSpend BConstr
  729         case es of
  730           (t : rest) -> computeCek (FrameConstr env i rest EmptyStack ctx) env t
  731           []         -> returnCek ctx $ VConstr i EmptyStack
  732     -- s ; ρ ▻ case S C0 ... Cn  ↦  s , case _ (C0 ... Cn, ρ) ; ρ ▻ S
  733     computeCek !ctx !env (Case _ scrut cs) = do
  734         stepAndMaybeSpend BCase
  735         computeCek (FrameCases env cs ctx) env scrut
  736     -- s ; ρ ▻ error  ↦  <> A
  737     computeCek !_ !_ (Error _) =
  738         throwingWithCause _OperationalError CekEvaluationFailure $ Error ()
  739 
  740     {- | The returning phase of the CEK machine.
  741     Returns 'EvaluationSuccess' in case the context is empty, otherwise pops up one frame
  742     from the context and uses it to decide how to proceed with the current value v.
  743 
  744       * 'FrameForce': call forceEvaluate
  745       * 'FrameApplyArg': call 'computeCek' over the context extended with 'FrameApplyFun'
  746       * 'FrameApplyFun': call 'applyEvaluate' to attempt to apply the function
  747           stored in the frame to an argument.
  748     -}
  749     returnCek
  750         :: Context uni fun ann
  751         -> CekValue uni fun ann
  752         -> CekM uni fun s (NTerm uni fun ())
  753     --- Instantiate all the free variable of the resulting term in case there are any.
  754     -- . ◅ V           ↦  [] V
  755     returnCek NoFrame val = do
  756         spendAccumulatedBudget
  757         pure $ dischargeCekValue val
  758     -- s , {_ A} ◅ abs α M  ↦  s ; ρ ▻ M [ α / A ]*
  759     returnCek (FrameForce ctx) fun = forceEvaluate ctx fun
  760     -- s , [_ (M,ρ)] ◅ V  ↦  s , [V _] ; ρ ▻ M
  761     returnCek (FrameAwaitFunTerm argVarEnv arg ctx) fun =
  762         computeCek (FrameAwaitArg fun ctx) argVarEnv arg
  763     -- s , [(lam x (M,ρ)) _] ◅ V  ↦  s ; ρ [ x  ↦  V ] ▻ M
  764     -- FIXME: add rule for VBuiltin once it's in the specification.
  765     returnCek (FrameAwaitArg fun ctx) arg =
  766         applyEvaluate ctx fun arg
  767     -- s , [_ V] ◅ lam x (M,ρ)  ↦  s ; ρ [ x  ↦  V ] ▻ M
  768     returnCek (FrameAwaitFunValue arg ctx) fun =
  769         applyEvaluate ctx fun arg
  770     -- s , constr I V0 ... Vj-1 _ (Tj+1 ... Tn, ρ) ◅ Vj  ↦  s , constr i V0 ... Vj _ (Tj+2... Tn, ρ)  ; ρ ▻ Tj+1
  771     returnCek (FrameConstr env i todo done ctx) e = do
  772         let done' = ConsStack e done
  773         case todo of
  774           (next : todo') -> computeCek (FrameConstr env i todo' done' ctx) env next
  775           []             -> returnCek ctx $ VConstr i done'
  776     -- s , case _ (C0 ... CN, ρ) ◅ constr i V1 .. Vm  ↦  s , [_ V1 ... Vm] ; ρ ▻ Ci
  777     returnCek (FrameCases env cs ctx) e = case e of
  778         -- If the index is larger than the max bound of an Int, or negative, then it's a bad index
  779         -- As it happens, this will currently never trigger, since i is a Word64, and the largest
  780         -- Word64 value wraps to -1 as an Int64. So you can't wrap around enough to get an
  781         -- "apparently good" value.
  782         (VConstr i _) | fromIntegral @_ @Integer i > fromIntegral @Int @Integer maxBound ->
  783                         throwingDischarged _StructuralError (MissingCaseBranchMachineError i) e
  784         -- Otherwise, we can safely convert the index to an Int and use it
  785         (VConstr i args) -> case (V.!?) cs (fromIntegral i) of
  786             Just t  -> computeCek (transferArgStack args ctx) env t
  787             Nothing -> throwingDischarged _StructuralError (MissingCaseBranchMachineError i) e
  788         _ -> throwingDischarged _StructuralError NonConstrScrutinizedMachineError e
  789 
  790     -- | Evaluate a 'HeadSpine' by pushing the arguments (if any) onto the stack and proceeding with
  791     -- the returning phase of the CEK machine.
  792     returnCekHeadSpine
  793         :: Context uni fun ann
  794         -> HeadSpine (CekValue uni fun ann)
  795         -> CekM uni fun s (Term NamedDeBruijn uni fun ())
  796     returnCekHeadSpine ctx (HeadOnly  x)    = returnCek ctx x
  797     returnCekHeadSpine ctx (HeadSpine f xs) = returnCek (transferSpine xs ctx) f

  799     -- | @force@ a term and proceed.
  800     -- If v is a delay then compute the body of v;
  801     -- if v is a builtin application then check that it's expecting a type argument,
  802     -- and either calculate the builtin application or stick a 'Force' on top of its 'Term'
  803     -- representation depending on whether the application is saturated or not,
  804     -- if v is anything else, fail.
  805     forceEvaluate
  806         :: Context uni fun ann
  807         -> CekValue uni fun ann
  808         -> CekM uni fun s (NTerm uni fun ())
  809     forceEvaluate !ctx (VDelay body env) = computeCek ctx env body
  810     forceEvaluate !ctx (VBuiltin fun term runtime) = do
  811         -- @term@ is fully discharged, and so @term'@ is, hence we can put it in a 'VBuiltin'.
  812         let term' = Force () term
  813         case runtime of
  814             -- It's only possible to force a builtin application if the builtin expects a type
  815             -- argument next.
  816             BuiltinExpectForce runtime' ->
  817                 -- We allow a type argument to appear last in the type of a built-in function,
  818                 -- otherwise we could just assemble a 'VBuiltin' without trying to evaluate the
  819                 -- application.
  820                 evalBuiltinApp ctx fun term' runtime'
  821             _ ->
  822                 throwingWithCause _StructuralError BuiltinTermArgumentExpectedMachineError term'
  823     forceEvaluate !_ val =
  824         throwingDischarged _StructuralError NonPolymorphicInstantiationMachineError val

  826     -- | Apply a function to an argument and proceed.
  827     -- If the function is a lambda 'lam x ty body' then extend the environment with a binding of @v@
  828     -- to x@ and call 'computeCek' on the body.
  829     -- If the function is a builtin application then check that it's expecting a term argument,
  830     -- and either calculate the builtin application or stick a 'Apply' on top of its 'Term'
  831     -- representation depending on whether the application is saturated or not.
  832     -- If v is anything else, fail.
  833     applyEvaluate
  834         :: Context uni fun ann
  835         -> CekValue uni fun ann   -- lhs of application
  836         -> CekValue uni fun ann   -- rhs of application
  837         -> CekM uni fun s (NTerm uni fun ())
  838     applyEvaluate !ctx (VLamAbs _ body env) arg =
  839         computeCek ctx (Env.cons arg env) body
  840     -- Annotating @f@ and @exF@ with bangs gave us some speed-up, but only until we added a bang to
  841     -- 'VCon'. After that the bangs here were making things a tiny bit slower and so we removed them.
  842     applyEvaluate !ctx (VBuiltin fun term runtime) arg = do
  843         let argTerm = dischargeCekValue arg
  844             -- @term@ and @argTerm@ are fully discharged, and so @term'@ is, hence we can put it
  845             -- in a 'VBuiltin'.
  846             term' = Apply () term argTerm
  847         case runtime of
  848             -- It's only possible to apply a builtin application if the builtin expects a term
  849             -- argument next.
  850             BuiltinExpectArgument f ->
  851                 evalBuiltinApp ctx fun term' $ f arg
  852             _ ->
  853                 throwingWithCause _StructuralError UnexpectedBuiltinTermArgumentMachineError term'
  854     applyEvaluate !_ val _ =
  855         throwingDischarged _StructuralError NonFunctionalApplicationMachineError val

  857     -- | Spend the budget that has been accumulated for a number of machine steps.
  858     spendAccumulatedBudget :: CekM uni fun s ()
  859     spendAccumulatedBudget = do
  860         let ctr = ?cekStepCounter
  861         iforCounter_ ctr spend
  862         resetCounter ctr
  863     -- It's very important for this definition not to get inlined. Inlining it caused performance to
  864     -- degrade by 16+%: https://github.com/IntersectMBO/plutus/pull/5931
  865     {-# OPAQUE spendAccumulatedBudget #-}
  866 
  867     -- Making this a definition of its own causes it to inline better than actually writing it inline, for
  868     -- some reason.
  869     -- Skip index 7, that's the total counter!
  870     -- See Note [Structure of the step counter]
  871     spend !i !w = unless (i == (fromIntegral $ natVal $ Proxy @TotalCountIndex)) $
  872       let kind = toEnum i in spendBudget (BStep kind) (stimes w (cekStepCost ?cekCosts kind))
  873     {-# INLINE spend #-}
  874 
  875     -- | Accumulate a step, and maybe spend the budget that has accumulated for a number of machine steps, but only if we've exceeded our slippage.
  876     stepAndMaybeSpend :: StepKind -> CekM uni fun s ()
  877     stepAndMaybeSpend !kind = do
  878         -- See Note [Structure of the step counter]
  879         -- This generates let-expressions in GHC Core, however all of them bind unboxed things and
  880         -- so they don't survive further compilation, see https://stackoverflow.com/a/14090277
  881         let !counterIndex = fromEnum kind
  882             ctr = ?cekStepCounter
  883             !totalStepIndex = fromIntegral $ natVal (Proxy @TotalCountIndex)
  884         !unbudgetedStepsTotal <-  modifyCounter totalStepIndex (+1) ctr
  885         _ <- modifyCounter counterIndex (+1) ctr
  886         -- There's no risk of overflow here, since we only ever increment the total
  887         -- steps by 1 and then check this condition.
  888         when (unbudgetedStepsTotal >= ?cekSlippage) spendAccumulatedBudget
  889     {-# INLINE stepAndMaybeSpend #-}

  891     -- | Take a possibly partial builtin application and
  892     --
  893     -- - either create a 'CekValue' by evaluating the application if it's saturated (emitting logs,
  894     --    if any, along the way), potentially failing evaluation
  895     -- - or create a partial builtin application otherwise
  896     --
  897     -- and proceed with the returning phase of the CEK machine.
  898     evalBuiltinApp
  899         :: Context uni fun ann
  900         -> fun
  901         -> NTerm uni fun ()
  902         -> BuiltinRuntime (CekValue uni fun ann)
  903         -> CekM uni fun s (Term NamedDeBruijn uni fun ())
  904     evalBuiltinApp ctx fun term runtime = case runtime of
  905         BuiltinCostedResult budgets0 getFXs -> do
  906             let exCat = BBuiltinApp fun
  907                 spendBudgets (ExBudgetLast budget) = spendBudget exCat budget
  908                 spendBudgets (ExBudgetCons budget budgets) =
  909                     spendBudget exCat budget *> spendBudgets budgets
  910             spendBudgets budgets0
  911             case getFXs of
  912                 BuiltinSuccess fXs ->
  913                     returnCekHeadSpine ctx fXs
  914                 BuiltinSuccessWithLogs logs fXs -> do
  915                     ?cekEmitter logs
  916                     returnCekHeadSpine ctx fXs
  917                 BuiltinFailure logs err -> do
  918                     ?cekEmitter logs
  919                     throwBuiltinErrorWithCause term err
  920         _ -> returnCek ctx $ VBuiltin fun term runtime
  921     {-# INLINE evalBuiltinApp #-}

  951 {- Note [Accumulators for terms]
  952 At a couple of points in the CEK machine (notably building the arguments to a constructor value)
  953 we need to compute a list of terms into values.
  954 
  955 Our usual strategy is to make a frame which has an (implicit) "hole" for the value we are computing,
  956 and which stores the other sub-parts of the term as terms or values depending on whether we've computed
  957 them yet or not (see e.g. how applications work).
  958 
  959 We want to do the same sort of strategy here, but it's a bit more complicated. We need a hole
  960 "in the middle" of the list, with computed values on one side and yet-to-be-computed terms on the other.
  961 We also very much want to avoid allocating as much as possible.
  962 
  963 The basic structure that we end up with has three parts:
  964 1. Use the list of sub-terms from the original term as our "todo" queue, a list is a good structure for this.
  965 2. Use an accumulator type with fast snoc to accumulate the values as we compute them.
  966 3. Convert the accumulator quickly into a final type that is fast to process/lookup in later.
  967 
  968 (this process was at one point made explicit with an interface, but in the end we just inlined it all away)
  969 
  970 We tried at least three variants of this:
  971 1. List/MutableVector/Vector
  972 2. List/Seq/Seq
  973 3. List/DList/List
  974 
  975 Suprisingly, option 3 was just as performant as the others, so we opted to go with it for simplicity.
  976 But there may well be a faster version.
