   29 {- Note [Inlining meanings of builtins]
   30 It's vitally important to inline the 'toBuiltinMeaning' method of a set of built-in functions as
   31 that allows GHC to look under lambdas and completely optimize multiple abstractions away.
   32 
   33 There are two ways of doing that: by relying on 'INLINE' pragmas all the way up from the
   34 'ToBuiltinMeaning' instance for the default set of builtins or by ensuring that 'toBuiltinsRuntime'
   35 is compiled efficient by turning it into a one-method class (see
   36 https://github.com/IntersectMBO/plutus/pull/4419 for how that looks like). We chose the former,
   37 because it's simpler. Although it's also less reliable: machine parameters are computed in
   38 multiple places and we need to make sure that benchmarking, cost model calculations and the actual
   39 production path have builtins compiled in the same way, 'cause otherwise performance analysis and
   40 cost predictions can be wrong by a big margin without us knowing. Because of this danger in addition
   41 to putting @INLINE@ pragmas on every relevant definition, we also stick a call to 'inline' at the
   42 call site. We also do not attempt to only compile things efficiently where we need that and instead
   43 inline the meanings of builtins everywhere. Just to be sure.
   44 
   45 Note that a combination of @INLINABLE@ + 'inline' does not result in proper inlining for whatever
   46 reason. It has to be @INLINE@ (and we add 'inline' on top of that for some additional reliability
   47 as we did have cases where sticking 'inline' on something that already had @INLINE@ would fix
   48 inlining).
   49 -}

   51 -- | Produce a 'DefaultMachineParameters' for each of the given semantics variants.
   52 -- The 'CostModelParams' argument is used to update the costing parameters returned by
   53 -- 'cekCostModelForVariant' for each of the semantics variants.
   54 --
   55 -- Whenever you need to evaluate UPLC in a performance-sensitive manner (e.g. in the production,
   56 -- for benchmarking, for cost calibration etc), you MUST use this definition for creating a
   57 -- 'DefaultMachineParameters' and not any other. Changing this definition in absolutely any way,
   58 -- however trivial, requires running the benchmarks and making sure that the resulting GHC Core is
   59 -- still sensible. E.g. you switch the order of arguments -- you run the benchmarks and check the
   60 -- Core; you move this definition as-is to a different file -- you run the benchmarks and check the
   61 -- Core; you change how it's exported (implicitly as a part of a whole-module export or explicitly
   62 -- as a single definition) -- you get the idea.
   63 --
   64 -- This function is very expensive, so its result needs to be cached if it's going to be used
   65 -- multiple times.
   66 mkMachineVariantParametersFor
   67     :: MonadError CostModelApplyError m
   68     => [BuiltinSemanticsVariant DefaultFun]
   69     -> CostModelParams
   70     -> m [(BuiltinSemanticsVariant DefaultFun, DefaultMachineVariantParameters)]
   71 mkMachineVariantParametersFor semVars newCMP = do
   72     res <- for semVars $ \semVar ->
   73         -- See Note [Inlining meanings of builtins].
   74         (,) semVar . inline mkMachineVariantParameters semVar <$>
   75             applyCostModelParams (cekCostModelForVariant semVar) newCMP
   76     -- Force all thunks to pay the cost of creating machine parameters upfront. Doing it here saves
   77     -- us from doing that in every single benchmark runner.
   78     pure $! force res
   79 -- Not marking this function with @INLINE@, since at this point everything we wanted to be inlined
   80 -- is inlined and there's zero reason to duplicate thousands and thousands of lines of Core down
   81 -- the line.
