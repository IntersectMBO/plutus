  117 {- Note [Compilation peculiarities]
  118 READ THIS BEFORE TOUCHING ANYTHING IN THIS FILE
  119 
  120 Don't use @StrictData@, it makes the machine slower by several percent.
  121 
  122 Exporting the 'computeCek' function from this module causes the CEK machine to become slower by
  123 up to 25%. I repeat: just adding 'computeCek' to the export list makes the evaluator substantially
  124 slower. The reason for this is that with 'computeCek' exported the generated GHC Core is much worse:
  125 it contains more lambdas, allocates more stuff etc. While perhaps surprising, this is not an
  126 unusual behavior of the compiler as https://wiki.haskell.org/Performance/GHC explains:
  127 
  128 > Indeed, generally speaking GHC will inline across modules just as much as it does within modules,
  129 > with a single large exception. If GHC sees that a function 'f' is called just once, it inlines it
  130 > regardless of how big 'f' is. But once 'f' is exported, GHC can never see that it's called exactly
  131 > once, even if that later turns out to be the case. This inline-once optimisation is pretty
  132 > important in practice.
  133 >
  134 > So: if you care about performance, do not export functions that are not used outside the module
  135 > (i.e. use an explicit export list, and keep it as small as possible).
  136 
  137 Now clearly 'computeCek' cannot be inlined in 'runCek' whether it's exported or not, since
  138 'computeCek' is recursive. However:
  139 
  140 1. GHC is _usually_ smart enough to perform the worker/wrapper transformation and inline the wrapper
  141    (however experiments have shown that sticking the internals of the CEK machine, budgeting modes
  142    and the API into the same file prevents GHC from performing the worker/wrapper transformation for
  143    some reason likely related to "we've been compiling this for too long, let's leave it at that"
  144 2. GHC seems to be able to massage the definition of 'computeCek' into something more performant
  145    making use of knowing exactly how 'computeCek' is used, essentially tailoring the definition of
  146    'computeCek' for a particular invocation in 'runCek'
  147 
  148 Hence we don't export 'computeCek' and instead define 'runCek' in this file and export it, even
  149 though the rest of the user-facing API (which 'runCek' is a part of) is defined downstream.
  150 
  151 Another problem is handling mutual recursion in the 'computeCek'/'returnCek'/'forceEvaluate'/etc
  152 family. If we keep these functions at the top level, GHC won't be able to pull the constraints out
  153 of the family (confirmed by inspecting Core: GHC thinks that since the superclass constraints
  154 populating the dictionary representing the @Ix fun@ constraint are redundant, they can be replaced
  155 with calls to 'error' in a recursive call, but that changes the dictionary and so it can no longer
  156 be pulled out of recursion). But that entails passing a redundant argument around, which slows down
  157 the machine a tiny little bit.
  158 
  159 Hence we define a all happy-path functions having CEK-machine-specific constraints as local
  160 functions making use of a shared context from their parent function. This also allows GHC to inline
  161 almost all of the machine into a single definition (with a bunch of recursive join points in it).
  162 
  163 In general, it's advised to run benchmarks (and look at Core output if the results are suspicious)
  164 on any changes in this file.
  165 
  166 Finally, it's important to put bang patterns on any 'Int' arguments to ensure that GHC unboxes them:
  167 this can make a surprisingly large difference.
  168 -}

  187 -- | Convert the given 'CekResult' into an 'Either'.
  188 -- This is useful, because in the ledger API we care whether the result is a constant or not, but in
  189 -- tests, executables etc we don't and so handling an either-error-or-term is more natural.
  190 cekResultToEither
  191     :: CekResult name uni fun
  192     -> Either (CekEvaluationException name uni fun) (Term name uni fun ())
  193 cekResultToEither (CekFailure err)             = Left err
  194 cekResultToEither (CekSuccessConstant val)     = Right $ Constant () val
  195 cekResultToEither (CekSuccessNonConstant term) = Right term
  196 
  197 -- | Apply the given function to the 'Term' (if any) stored in the given 'CekResult'.
  198 mapTermCekResult
  199     :: (Term name uni fun () -> Term name' uni fun ())
  200     -> CekResult name uni fun
  201     -> CekResult name' uni fun
  202 mapTermCekResult f (CekFailure err)             = CekFailure $ f <$> err
  203 mapTermCekResult _ (CekSuccessConstant val)     = CekSuccessConstant val
  204 mapTermCekResult f (CekSuccessNonConstant term) = CekSuccessNonConstant $ f term

  252 -- | A LIFO stack of 'CekValue's, used to record multiple arguments that need to be pushed
  253 -- onto the context in reverse order.  Currently used by 'FrameConstr' for collecting the
  254 -- elements of a 'Constr' as it is cheap to prepend new elements in 'ArgStack'.
  255 data ArgStack uni fun ann =
  256   NilStack
  257   | ConsStack !(CekValue uni fun ann) !(ArgStack uni fun ann)
  258 
  259 -- | A non-empty variant of 'ArgStack', used in 'FrameAwaitFunValueN' to store arguments
  260 -- that will be applied to a term. More efficient than 'ArgStack', since this saves one
  261 -- evaluation cycle by ensuring there is no 'NilStack'.
  262 data ArgStackNonEmpty uni fun ann =
  263   LastStackNonEmpty !(CekValue uni fun ann)
  264   | ConsStackNonEmpty !(CekValue uni fun ann) !(ArgStackNonEmpty uni fun ann)
  265 
  266 -- | An alternative version of 'ArgStack' that uses 'ArgNonEmptyStack' when non-empty.
  267 -- Used in 'VConstr'. Once all evaluated elements of 'Constr' is collecting to 'ArgStack'
  268 -- in 'FrameConstr', the collected elements gets reversed and put into 'VConstr' as
  269 -- `EmptyOrMultiStack`. 'VConstr' using `EmptyOrMultiStack` is more efficient than 'ArgStack' when casing,
  270 -- since 'FrameAwaitFunValueN' can be dispatched with a single pattern match.
  271 data EmptyOrMultiStack uni fun ann =
  272   EmptyStack
  273   | MultiStack !(ArgStackNonEmpty uni fun ann)
  274

  279 deriving stock instance (GShow uni, Everywhere uni Show, Show fun, Show ann, Closed uni)

  282 -- 'Values' for the modified CEK machine.
  283 data CekValue uni fun ann =
  284     -- This bang gave us a 1-2% speed-up at the time of writing.
  285     VCon !(Some (ValueOf uni))
  286   | VDelay !(NTerm uni fun ann) !(CekValEnv uni fun ann)
  287   | VLamAbs !NamedDeBruijn !(NTerm uni fun ann) !(CekValEnv uni fun ann)
  288     -- | A partial builtin application, accumulating arguments for eventual full application.
  289     -- We don't need a 'CekValEnv' here unlike in the other constructors, because 'VBuiltin'
  290     -- values always store their corresponding 'Term's fully discharged, see the comments at
  291     -- the call sites (search for 'VBuiltin').
  292   | VBuiltin
  293       !fun
  294       -- ^ So that we know, for what builtin we're calculating the cost. We can sneak this into
  295       -- 'BuiltinRuntime', so that we don't need to store it here, but somehow doing so was
  296       -- consistently slowing evaluation down by half a percent. Might be noise, might be not, but
  297       -- at least we know that removing this @fun@ is not helpful anyway. See this commit reversing
  298       -- the change: https://github.com/IntersectMBO/plutus/pull/4778/commits/86a3e24ca3c671cc27c6f4344da2bcd14f961706
  299       (NTerm uni fun ())
  300       -- ^ This must be lazy. It represents the fully discharged partial application of the builtin
  301       -- function that we're going to run when it's fully saturated.  We need the 'Term' to be able
  302       -- to return it in case full saturation is never achieved and a partial application needs to
  303       -- be returned in the result. The laziness is important, because the arguments are discharged
  304       -- values and discharging is expensive, so we don't want to do it unless we really have
  305       -- to. Making this field strict resulted in a 3-4.5% slowdown at the time of writing.
  306       !(BuiltinRuntime (CekValue uni fun ann))
  307       -- ^ The partial application and its costing function.
  308       -- Check the docs of 'BuiltinRuntime' for details.
  309     -- | A constructor value, including fully computed arguments and the tag.
  310   | VConstr {-# UNPACK #-} !Word64 !(EmptyOrMultiStack uni fun ann)
  311 
  312 deriving stock instance (GShow uni, Everywhere uni Show, Show fun, Show ann, Closed uni)
  313     => Show (CekValue uni fun ann)

  340 {- Note [Cost slippage]
  341 Tracking the budget usage for every step in the machine adds a lot of overhead. To reduce this,
  342 we adopt a technique which allows some overshoot of the budget ("slippage"), but only a bounded
  343 amount.
  344 
  345 To do this we:
  346 - Track all the machine steps of all kinds in an set of counters in a 'StepCounter'
  347 - Actually "spend" the budget when we've done more than some fixed number of steps, or at the end.
  348 
  349 This saves a *lot* of time, at the cost of potentially overshooting the budget by slippage*step_cost,
  350 which is okay so long as we bound the slippage appropriately.
  351 
  352 Note that we're only proposing to do this for machine steps, since it's plausible that we can track
  353 them in an optimized way. Builtins are more complicated (and infrequent), so we can just budget them
  354 properly when we hit them.
  355 
  356 There are two options for how to bound the slippage:
  357 1. As a fixed number of steps
  358 2. As a proportion of the overall budget
  359 
  360 Option 2 initially seems much better as a bound: if we run N scripts with an overall budget of B, then
  361 the potential overrun from 1 is N*slippage, whereas the overrun from 2 is B*slippage. That is, 2
  362 says we always overrun by a fraction of the total amount of time you were expecting, whereas 1 says
  363 it depends how many scripts you run... so if I send you a lot of small scripts, I could cause a lot
  364 of overrun.
  365 
  366 However, it turns out (empirically) that we can pick a number for 1 that gives us most of the speedup, but such
  367 that the maximum overrun is negligible (e.g. much smaller than the "startup cost"). So in the end
  368 we opted for option 1, which also happens to be simpler to implement.
  369 -}

  371 {- Note [Structure of the step counter]
  372 The step counter is kept in a 'MutablePrimArray', which is a fast way of storing a bunch of
  373 mutable 'Word8's.
  374 This suits our purposes, as we need one counter for every step type, and one for the total number.
  375 
  376 We keep the counters for each step in the first indices, so we can index them simply by using
  377 the 'Enum' instance of 'StepKind', and the total counter in the last index.
  378 -}
  379 
  380 -- So that we don't need to update 'NumberOfStepCounters' manually, which would be extremely
  381 -- error-prone and has caused a bug in the past.
  382 type CountConstructorsEnum :: (GHC.Type -> GHC.Type) -> Nat
  383 type family CountConstructorsEnum rep where
  384     CountConstructorsEnum U1         = 1
  385     CountConstructorsEnum (M1 _ _ f) = CountConstructorsEnum f
  386     CountConstructorsEnum (f :+: g)  = CountConstructorsEnum f + CountConstructorsEnum g
  387     CountConstructorsEnum V1         = TypeError ('Text "Cannot be empty")
  388     CountConstructorsEnum (f :*: g)  = TypeError ('Text "Cannot be a non-enumeration type")
  389     CountConstructorsEnum (K1 _ _)   = TypeError ('Text "Cannot be a non-enumeration type")
  390     CountConstructorsEnum (Rec1 _)   = TypeError ('Text "Cannot be a non-enumeration type")
  391     CountConstructorsEnum Par1       =
  392         TypeError ('Text "If you really want a parameterized type, handle this clause")
  393 
  394 -- | The number of step counters that we need, should be the same as the number of constructors
  395 -- of 'StepKind'.
  396 type NumberOfStepCounters = CountConstructorsEnum (Rep StepKind)
  397 -- | The total number of counters that we need, one extra for the total counter.
  398 -- See Note [Structure of the step counter]
  399 type CounterSize = NumberOfStepCounters + 1
  400 -- | The index at which the total step counter is kept.
  401 -- See Note [Structure of the step counter]
  402 type TotalCountIndex = NumberOfStepCounters

  433 {- Note [Implicit parameters in the machine]
  434 The traditional way to pass context into a function is to use 'ReaderT'. However, 'ReaderT' has some
  435 disadvantages.
  436 - It requires threading through the context even where you don't need it (every monadic bind)
  437 - It *can* often be optimized away, but this requires GHC to be somewhat clever and do a lot of
  438   case-of-case to lift all the arguments out.
  439 
  440 Moreover, if your context is global (i.e. constant across the lifetime of the monad, i.e. you don't
  441 need 'local'), then you're buying some extra power (the ability to pass in a different context somewhere
  442 deep inside the computation) which you don't need.
  443 
  444 There are three main alternatives:
  445 - Explicit function parameters. Simple, doesn't get tied up in the Monad operations, *does* still
  446 present the appearance of letting you do 'local'. But a bit cluttered.
  447 - Implicit parameters. A bit esoteric, can be bundled up into a constraint synonym and just piped to
  448 where they're needed, essentially the same as explicit parameters in terms of runtime.
  449 - Constraints via 'reflection'. Quite esoteric, *does* get you global parameters (within their scope),
  450 bit of a hassle threading around all the extra type parameters.
  451 
  452 We're using implicit parameters for now, which seems to strike a good balance of speed and convenience.
  453 I haven't tried 'reflection' in detail, but I believe the main thing it would do is to make the parameters
  454 global - but we already have this for most of the hot functions by making them all local definitions, so
  455 they don't actually take the context as an argument even at the source level.
  456 -}

  514 {- Note [Catching exceptions in ST]
  515 This note represents MPJ's best understanding right now, might be wrong.
  516 
  517 We use a moderately evil trick to catch exceptions in ST. This uses the unsafe ST <-> IO conversion functions to go into IO,
  518 catch the exception, and then go back into ST.
  519 
  520 Why is this okay? Recall that IO ~= ST RealWorld, i.e. it is just ST with a special thread token. The unsafe conversion functions
  521 just coerce from one to the other. So the thread token remains the same, it's just that we'll potentially leak it from ST, and we don't
  522 get ordering guarantees with other IO actions.
  523 
  524 But in our case this is okay, because:
  525 
  526 1. We do not leak the original ST thread token, since we only pass it into IO and then immediately back again.
  527 2. We don't have ordering guarantees with other IO actions, but we don't care because we don't do any side effects, we only catch a single kind of exception.
  528 3. We *do* have ordering guarantees between the throws inside the ST action and the catch, since they are ultimately using the same thread token.
  529 -}

  625 -- | Convert a 'CekValue' into a 'Term' by replacing all bound variables with the terms
  626 -- they're bound to (which themselves have to be obtained by recursively discharging values).
  627 dischargeCekValue :: forall uni fun ann. CekValue uni fun ann -> DischargeResult uni fun
  628 dischargeCekValue (VCon val) = DischargeConstant val
  629 dischargeCekValue value0     = DischargeNonConstant $ goValue value0 where
  630     goValue :: CekValue uni fun ann -> NTerm uni fun ()
  631     goValue = \case
  632         VCon val -> Constant () val
  633         VDelay body env -> Delay () $ goValEnv env 0 body
  634         VLamAbs (NamedDeBruijn n _ix) body env ->
  635             -- The index on the binder is meaningless, we put @0@ by convention, see 'Binder'.
  636             LamAbs () (NamedDeBruijn n deBruijnInitIndex) $ goValEnv env 1 body
  637         -- We only return a discharged builtin application when (a) it's being returned by the
  638         -- machine, or (b) it's needed for an error message.
  639         -- @term@ is fully discharged, so we can return it directly without any further discharging.
  640         VBuiltin _ term _ -> term
  641         VConstr ind args -> Constr () ind . map goValue $ argStackToList args
  642 
  643     -- Instantiate all the free variables of a term by looking them up in an environment.
  644     -- Mutually recursive with @goValue@.
  645     goValEnv :: CekValEnv uni fun ann -> Word64 -> NTerm uni fun ann -> NTerm uni fun ()
  646     goValEnv env = go where
  647         -- @shift@ is just a counter that measures how many lambda-abstractions we have descended
  648         -- into so far.
  649         go :: Word64 -> NTerm uni fun ann -> NTerm uni fun ()
  650         go !shift =  \case
  651           LamAbs _ name body -> LamAbs () name $ go (shift + 1) body
  652           Var _ named@(NamedDeBruijn _ (coerce -> idx)) ->
  653               if shift >= idx
  654               -- the index n is less-than-or-equal than the number of lambdas we have descended
  655               -- this means that n points to a bound variable, so we don't discharge it.
  656               then Var () named
  657               else maybe
  658                      -- var is free, leave it alone
  659                      (Var () named)
  660                      -- var is in the env, discharge its value
  661                      goValue
  662                      -- index relative to (as seen from the point of view of) the environment
  663                      (Env.indexOne env $ idx - shift)
  664           Apply _ fun arg    -> Apply () (go shift fun) $ go shift arg
  665           Delay _ term       -> Delay () $ go shift term
  666           Force _ term       -> Force () $ go shift term
  667           Constant _ val     -> Constant () val
  668           Builtin _ fun      -> Builtin () fun
  669           Error _            -> Error ()
  670           Constr _ ind args  -> Constr () ind $ map (go shift) args
  671           Case _ scrut alts  -> Case () (go shift scrut) $ fmap (go shift) alts

  686 {-|
  687 The context in which the machine operates.
  688 
  689 Morally, this is a stack of frames, but we use the "intrusive list" representation so that
  690 we can match on context and the top frame in a single, strict pattern match.
  691 -}
  692 data Context uni fun ann
  693     = FrameAwaitArg !(CekValue uni fun ann) !(Context uni fun ann)
  694     -- ^ @[V _]@
  695     | FrameAwaitFunTerm !(CekValEnv uni fun ann) !(NTerm uni fun ann) !(Context uni fun ann)
  696     -- ^ @[_ N]@
  697     | FrameAwaitFunConN !(Spine (Some (ValueOf uni))) !(Context uni fun ann)
  698     -- ^ @[_ V]@
  699     | FrameAwaitFunValueN !(ArgStackNonEmpty uni fun ann) !(Context uni fun ann)
  700     -- ^ @[_ V1 .. Vn]@
  701     | FrameForce !(Context uni fun ann)
  702     -- ^ @(force _)@
  703     -- See Note [Accumulators for terms]
  704     | FrameConstr !(CekValEnv uni fun ann) {-# UNPACK #-} !Word64 ![NTerm uni fun ann] !(ArgStack uni fun ann) !(Context uni fun ann)
  705     -- ^ @(constr i V0 ... Vj-1 _ Nj ... Nn)@
  706     | FrameCases !(CekValEnv uni fun ann) !(V.Vector (NTerm uni fun ann)) !(Context uni fun ann)
  707     -- ^ @(case _ C0 .. Cn)@
  708     | NoFrame
  709 
  710 deriving stock instance (GShow uni, Everywhere uni Show, Show fun, Show ann, Closed uni)
  711     => Show (Context uni fun ann)

  723 {- Note [ArgStack vs Spine]
  724 We use 'ArgStack' for collecting the arguments of a constructor to later pass it to the function in
  725 the appropriate branch. Originally, all arguments of a constructor are terms and hence before we can
  726 pass them to a function they need to be evaluated to values, which means that in case of the CEK
  727 machine the evaluated arguments are going to be reversed: you evaluate the first argument and put
  728 the result into a 'FrameConstr', then the second one and put it in a 'FrameConstr' again, this time
  729 prepending it to the one that is already there etc -- in the end you get the arguments in reversed
  730 order.
  731 
  732 But in case of 'Spine' the builtins machinery directly produces values, not terms. Meaning, a
  733 'Spine' that we get from the builtins machinery isn't reversed, hence we can pass its contents
  734 directly to the head of the application. Which is why 'transferSpine' is a right fold.
  735 -}

  737 runCekM
  738     :: forall cost uni fun ann
  739     . ThrowableBuiltins uni fun
  740     => MachineParameters CekMachineCosts fun (CekValue uni fun ann)
  741     -> ExBudgetMode cost uni fun
  742     -> EmitterMode uni fun
  743     -> (forall s. GivenCekReqs uni fun ann s => CekM uni fun s (DischargeResult uni fun))
  744     -> CekReport cost NamedDeBruijn uni fun
  745 runCekM
  746         (MachineParameters caser (MachineVariantParameters costs runtime))
  747         (ExBudgetMode getExBudgetInfo)
  748         (EmitterMode getEmitterMode)
  749         a = runST $ do
  750     ExBudgetInfo{_exBudgetModeSpender, _exBudgetModeGetFinal, _exBudgetModeGetCumulative} <- getExBudgetInfo
  751     CekEmitterInfo{_cekEmitterInfoEmit, _cekEmitterInfoGetFinal} <- getEmitterMode _exBudgetModeGetCumulative
  752     ctr <- newCounter (Proxy @CounterSize)
  753     let ?cekRuntime = runtime
  754         ?cekCaserBuiltin = caser
  755         ?cekEmitter = _cekEmitterInfoEmit
  756         ?cekBudgetSpender = _exBudgetModeSpender
  757         ?cekCosts = costs
  758         ?cekSlippage = defaultSlippage
  759         ?cekStepCounter = ctr
  760     res <- unCekM (tryError a) <&> \case
  761         Left err                          -> CekFailure err
  762         Right (DischargeConstant val)     -> CekSuccessConstant val
  763         Right (DischargeNonConstant term) -> CekSuccessNonConstant term
  764     st <- _exBudgetModeGetFinal
  765     logs <- _cekEmitterInfoGetFinal
  766     pure $ CekReport res st logs
  767 {-# INLINE runCekM #-}

  769 -- See Note [Compilation peculiarities].
  770 -- | The entering point to the CEK machine's engine.
  771 enterComputeCek
  772     :: forall uni fun ann s
  773     . (ThrowableBuiltins uni fun, GivenCekReqs uni fun ann s)
  774     => Context uni fun ann
  775     -> CekValEnv uni fun ann
  776     -> NTerm uni fun ann
  777     -> CekM uni fun s (DischargeResult uni fun)
  778 enterComputeCek = computeCek
  779   where
  780     -- | The computing part of the CEK machine.
  781     -- Either
  782     -- 1. adds a frame to the context and calls 'computeCek' ('Force', 'Apply')
  783     -- 2. calls 'returnCek' on values ('Delay', 'LamAbs', 'Constant', 'Builtin')
  784     -- 3. throws 'EvaluationFailure' ('Error')
  785     -- 4. looks up a variable in the environment and calls 'returnCek' ('Var')
  786     computeCek
  787         :: Context uni fun ann
  788         -> CekValEnv uni fun ann
  789         -> NTerm uni fun ann
  790         -> CekM uni fun s (DischargeResult uni fun)
  791     -- s ; ρ ▻ {L A}  ↦ s , {_ A} ; ρ ▻ L
  792     computeCek !ctx !env (Var _ varName) = do
  793         stepAndMaybeSpend BVar
  794         val <- lookupVarName varName env
  795         returnCek ctx val
  796     -- s ; ρ ▻ con c  ↦  s ◅ con c
  797     computeCek !ctx !_ (Constant _ val) = do
  798         stepAndMaybeSpend BConst
  799         returnCek ctx (VCon val)
  800     -- s ; ρ ▻ lam x L  ↦  s ◅ lam x (L , ρ)
  801     computeCek !ctx !env (LamAbs _ name body) = do
  802         stepAndMaybeSpend BLamAbs
  803         returnCek ctx (VLamAbs name body env)
  804     -- s ; ρ ▻ delay L  ↦  s ◅ delay (L , ρ)
  805     computeCek !ctx !env (Delay _ body) = do
  806         stepAndMaybeSpend BDelay
  807         returnCek ctx (VDelay body env)
  808     -- s ; ρ ▻ force T  ↦  s , force _ ; ρ ▻ L
  809     computeCek !ctx !env (Force _ body) = do
  810         stepAndMaybeSpend BForce
  811         computeCek (FrameForce ctx) env body
  812     -- s ; ρ ▻ [L M]  ↦  s , [_ (M,ρ)]  ; ρ ▻ L
  813     computeCek !ctx !env (Apply _ fun arg) = do
  814         stepAndMaybeSpend BApply
  815         computeCek (FrameAwaitFunTerm env arg ctx) env fun
  816     -- s ; ρ ▻ builtin bn  ↦  s ◅ builtin bn arity arity [] [] ρ
  817     computeCek !ctx !_ (Builtin _ bn) = do
  818         stepAndMaybeSpend BBuiltin
  819         let meaning = lookupBuiltin bn ?cekRuntime
  820         -- 'Builtin' is fully discharged.
  821         returnCek ctx (VBuiltin bn (Builtin () bn) meaning)
  822     -- s ; ρ ▻ constr I T0 .. Tn  ↦  s , constr I _ (T1 ... Tn, ρ) ; ρ ▻ T0
  823     computeCek !ctx !env (Constr _ i es) = do
  824         stepAndMaybeSpend BConstr
  825         case es of
  826           (t : rest) -> computeCek (FrameConstr env i rest NilStack ctx) env t
  827           []         -> returnCek ctx $ VConstr i EmptyStack
  828     -- s ; ρ ▻ case S C0 ... Cn  ↦  s , case _ (C0 ... Cn, ρ) ; ρ ▻ S
  829     computeCek !ctx !env (Case _ scrut cs) = do
  830         stepAndMaybeSpend BCase
  831         computeCek (FrameCases env cs ctx) env scrut
  832     -- s ; ρ ▻ error  ↦  <> A
  833     computeCek !_ !_ (Error _) =
  834         throwErrorWithCause (OperationalError CekEvaluationFailure) (Error ())
  835 
  836     {- | The returning phase of the CEK machine.
  837     Returns 'EvaluationSuccess' in case the context is empty, otherwise pops up one frame
  838     from the context and uses it to decide how to proceed with the current value v.
  839 
  840       * 'FrameForce': call forceEvaluate
  841       * 'FrameApplyArg': call 'computeCek' over the context extended with 'FrameApplyFun'
  842       * 'FrameApplyFun': call 'applyEvaluate' to attempt to apply the function
  843           stored in the frame to an argument.
  844     -}
  845     returnCek
  846         :: Context uni fun ann
  847         -> CekValue uni fun ann
  848         -> CekM uni fun s (DischargeResult uni fun)
  849     --- Instantiate all the free variable of the resulting term in case there are any.
  850     -- . ◅ V           ↦  [] V
  851     returnCek NoFrame val = do
  852         spendAccumulatedBudget
  853         pure $ dischargeCekValue val
  854     -- s , {_ A} ◅ abs α M  ↦  s ; ρ ▻ M [ α / A ]*
  855     returnCek (FrameForce ctx) fun = forceEvaluate ctx fun
  856     -- s , [_ (M,ρ)] ◅ V  ↦  s , [V _] ; ρ ▻ M
  857     returnCek (FrameAwaitFunTerm argVarEnv arg ctx) fun =
  858         computeCek (FrameAwaitArg fun ctx) argVarEnv arg
  859     -- s , [(lam x (M,ρ)) _] ◅ V  ↦  s ; ρ [ x  ↦  V ] ▻ M
  860     -- FIXME (https://github.com/IntersectMBO/plutus-private/issues/1878):
  861     -- add rule for VBuiltin once it's in the specification.
  862     returnCek (FrameAwaitArg fun ctx) arg =
  863         applyEvaluate ctx fun arg
  864     -- s , [_ V] ◅ lam x (M,ρ) ↦ s ; ρ [ x  ↦  V ] ▻ M
  865     returnCek (FrameAwaitFunConN args ctx) fun =
  866       -- In the future, if we want to revert back to more general
  867       -- 'FrameAwaitFunValue (CekValue uni fun ann)', we can use optimization proposed in
  868       -- https://github.com/IntersectMBO/plutus/pull/7288.  #7288 have almost equivalent
  869       -- performance improvement as using 'FrameAwaitFunConN' while keeping more general
  870       -- 'FrameAwaitFunValue'.
  871       case args of
  872         SpineLast arg      -> applyEvaluate ctx fun (VCon arg)
  873         SpineCons arg rest -> applyEvaluate (FrameAwaitFunConN rest ctx) fun (VCon arg)
  874     -- s , [_ V1 .. Vn] ◅ lam x (M,ρ)  ↦  s , [_ V2 .. Vn]; ρ [ x  ↦  V1 ] ▻ M
  875     returnCek (FrameAwaitFunValueN args ctx) fun =
  876         case args of
  877           LastStackNonEmpty arg ->
  878             applyEvaluate ctx fun arg
  879           ConsStackNonEmpty arg rest ->
  880             applyEvaluate (FrameAwaitFunValueN rest ctx) fun arg
  881     -- s , constr I V0 ... Vj-1 _ (Tj+1 ... Tn, ρ) ◅ Vj  ↦  s , constr i V0 ... Vj _ (Tj+2... Tn, ρ)  ; ρ ▻ Tj+1
  882     returnCek (FrameConstr env i todo done ctx) e = do
  883         case todo of
  884           (next : todo') -> computeCek (FrameConstr env i todo' (ConsStack e done) ctx) env next
  885           []             ->
  886             let go acc NilStack         = acc
  887                 go acc (ConsStack x xs) = go (ConsStackNonEmpty x acc) xs
  888             in returnCek ctx $ VConstr i (MultiStack $ go (LastStackNonEmpty e) done)
  889     -- s , case _ (C0 ... CN, ρ) ◅ constr i V1 .. Vm  ↦  s , [_ V1 ... Vm] ; ρ ▻ Ci
  890     returnCek (FrameCases env cs ctx) e = case e of
  891         -- If the index is larger than the max bound of an Int, or negative, then it's a bad index
  892         -- As it happens, this will currently never trigger, since i is a Word64, and the largest
  893         -- Word64 value wraps to -1 as an Int64. So you can't wrap around enough to get an
  894         -- "apparently good" value.
  895         (VConstr i _) | fromIntegral @_ @Integer i > fromIntegral @Int @Integer maxBound ->
  896                         throwErrorDischarged (StructuralError (MissingCaseBranchMachineError i)) e
  897         -- Otherwise, we can safely convert the index to an Int and use it.
  898         (VConstr i args) -> case (V.!?) cs (fromIntegral i) of
  899             Just t  -> case args of
  900               EmptyStack      -> computeCek ctx env t
  901               MultiStack rest -> computeCek (FrameAwaitFunValueN rest ctx) env t
  902             Nothing -> throwErrorDischarged (StructuralError $ MissingCaseBranchMachineError i) e
  903         -- Proceed with caser when expression given is not Constr.
  904         VCon val -> case unCaserBuiltin ?cekCaserBuiltin val cs of
  905             Left err          -> throwErrorDischarged (OperationalError $ CekCaseBuiltinError err) e
  906             Right (HeadOnly fX) -> computeCek ctx env fX
  907             Right (HeadSpine f xs) -> computeCek (FrameAwaitFunConN xs ctx) env f
  908         _ -> throwErrorDischarged (StructuralError NonConstrScrutinizedMachineError) e
  909 
  910     -- | @force@ a term and proceed.
  911     -- If v is a delay then compute the body of v;
  912     -- if v is a builtin application then check that it's expecting a type argument,
  913     -- and either calculate the builtin application or stick a 'Force' on top of its 'Term'
  914     -- representation depending on whether the application is saturated or not,
  915     -- if v is anything else, fail.
  916     forceEvaluate
  917         :: Context uni fun ann
  918         -> CekValue uni fun ann
  919         -> CekM uni fun s (DischargeResult uni fun)
  920     forceEvaluate !ctx (VDelay body env) = computeCek ctx env body
  921     forceEvaluate !ctx (VBuiltin fun term runtime) = do
  922         -- @term@ is fully discharged, and so @term'@ is, hence we can put it in a 'VBuiltin'.
  923         let term' = Force () term
  924         case runtime of
  925             -- It's only possible to force a builtin application if the builtin expects a type
  926             -- argument next.
  927             BuiltinExpectForce runtime' ->
  928                 -- We allow a type argument to appear last in the type of a built-in function,
  929                 -- otherwise we could just assemble a 'VBuiltin' without trying to evaluate the
  930                 -- application.
  931                 evalBuiltinApp ctx fun term' runtime'
  932             _ ->
  933                 throwErrorWithCause (StructuralError BuiltinTermArgumentExpectedMachineError) term'
  934     forceEvaluate !_ val =
  935         throwErrorDischarged (StructuralError NonPolymorphicInstantiationMachineError) val
  936 
  937     -- | Apply a function to an argument and proceed.
  938     -- If the function is a lambda 'lam x ty body' then extend the environment with a binding of @v@
  939     -- to x@ and call 'computeCek' on the body.
  940     -- If the function is a builtin application then check that it's expecting a term argument,
  941     -- and either calculate the builtin application or stick a 'Apply' on top of its 'Term'
  942     -- representation depending on whether the application is saturated or not.
  943     -- If v is anything else, fail.
  944     applyEvaluate
  945         :: Context uni fun ann
  946         -> CekValue uni fun ann   -- lhs of application
  947         -> CekValue uni fun ann   -- rhs of application
  948         -> CekM uni fun s (DischargeResult uni fun)
  949     applyEvaluate !ctx (VLamAbs _ body env) arg =
  950         computeCek ctx (Env.cons arg env) body
  951     -- Annotating @f@ and @exF@ with bangs gave us some speed-up, but only until we added a bang to
  952     -- 'VCon'. After that the bangs here were making things a tiny bit slower and so we removed them.
  953     applyEvaluate !ctx (VBuiltin fun funTerm runtime) arg = do
  954         let argTerm = dischargeResultToTerm $ dischargeCekValue arg
  955             -- @term@ and @argTerm@ are fully discharged, and so @term'@ is, hence we can put it
  956             -- in a 'VBuiltin'.
  957             term' = Apply () funTerm argTerm
  958         case runtime of
  959             -- It's only possible to apply a builtin application if the builtin expects a term
  960             -- argument next.
  961             BuiltinExpectArgument f ->
  962                 evalBuiltinApp ctx fun term' $ f arg
  963             _ ->
  964                 throwErrorWithCause (StructuralError UnexpectedBuiltinTermArgumentMachineError) term'
  965     applyEvaluate !_ val _ =
  966         throwErrorDischarged (StructuralError NonFunctionalApplicationMachineError) val

  968     -- | Spend the budget that has been accumulated for a number of machine steps.
  969     spendAccumulatedBudget :: CekM uni fun s ()
  970     spendAccumulatedBudget = do
  971         let ctr = ?cekStepCounter
  972         iforCounter_ ctr spend
  973         resetCounter ctr
  974     -- It's very important for this definition not to get inlined. Inlining it caused performance to
  975     -- degrade by 16+%: https://github.com/IntersectMBO/plutus/pull/5931
  976     {-# OPAQUE spendAccumulatedBudget #-}
  977 
  978     -- Making this a definition of its own causes it to inline better than actually writing it inline, for
  979     -- some reason.
  980     -- Skip index 7, that's the total counter!
  981     -- See Note [Structure of the step counter]
  982     spend !i !w = unless (i == (fromIntegral $ natVal $ Proxy @TotalCountIndex)) $
  983       let kind = toEnum i in spendBudget (BStep kind) (stimes w (cekStepCost ?cekCosts kind))
  984     {-# INLINE spend #-}
  985 
  986     -- | Accumulate a step, and maybe spend the budget that has accumulated for a number of machine steps, but only if we've exceeded our slippage.
  987     stepAndMaybeSpend :: StepKind -> CekM uni fun s ()
  988     stepAndMaybeSpend !kind = do
  989         -- See Note [Structure of the step counter]
  990         -- This generates let-expressions in GHC Core, however all of them bind unboxed things and
  991         -- so they don't survive further compilation, see https://stackoverflow.com/a/14090277
  992         let !counterIndex = fromEnum kind
  993             ctr = ?cekStepCounter
  994             !totalStepIndex = fromIntegral $ natVal (Proxy @TotalCountIndex)
  995         !unbudgetedStepsTotal <-  modifyCounter totalStepIndex (+1) ctr
  996         _ <- modifyCounter counterIndex (+1) ctr
  997         -- There's no risk of overflow here, since we only ever increment the total
  998         -- steps by 1 and then check this condition.
  999         when (unbudgetedStepsTotal >= ?cekSlippage) spendAccumulatedBudget
 1000     {-# INLINE stepAndMaybeSpend #-}

 1002     -- | Take a possibly partial builtin application and
 1003     --
 1004     -- - either create a 'CekValue' by evaluating the application if it's saturated (emitting logs,
 1005     --    if any, along the way), potentially failing evaluation
 1006     -- - or create a partial builtin application otherwise
 1007     --
 1008     -- and proceed with the returning phase of the CEK machine.
 1009     evalBuiltinApp
 1010         :: Context uni fun ann
 1011         -> fun
 1012         -> NTerm uni fun ()
 1013         -> BuiltinRuntime (CekValue uni fun ann)
 1014         -> CekM uni fun s (DischargeResult uni fun)
 1015     evalBuiltinApp ctx fun term runtime = case runtime of
 1016         BuiltinCostedResult budgets0 getFXs -> do
 1017             let exCat = BBuiltinApp fun
 1018                 spendBudgets (ExBudgetLast budget) = spendBudget exCat budget
 1019                 spendBudgets (ExBudgetCons budget budgets) =
 1020                     spendBudget exCat budget *> spendBudgets budgets
 1021             spendBudgets budgets0
 1022             case getFXs of
 1023                 BuiltinSuccess y ->
 1024                     returnCek ctx y
 1025                 BuiltinSuccessWithLogs logs y -> do
 1026                     ?cekEmitter logs
 1027                     returnCek ctx y
 1028                 BuiltinFailure logs err -> do
 1029                     ?cekEmitter logs
 1030                     throwBuiltinErrorWithCause term err
 1031         _ -> returnCek ctx $ VBuiltin fun term runtime
 1032     {-# INLINE evalBuiltinApp #-}

 1062 {- Note [Accumulators for terms]
 1063 At a couple of points in the CEK machine (notably building the arguments to a constructor value)
 1064 we need to compute a list of terms into values.
 1065 
 1066 Our usual strategy is to make a frame which has an (implicit) "hole" for the value we are computing,
 1067 and which stores the other sub-parts of the term as terms or values depending on whether we've computed
 1068 them yet or not (see e.g. how applications work).
 1069 
 1070 We want to do the same sort of strategy here, but it's a bit more complicated. We need a hole
 1071 "in the middle" of the list, with computed values on one side and yet-to-be-computed terms on the other.
 1072 We also very much want to avoid allocating as much as possible.
 1073 
 1074 The basic structure that we end up with has three parts:
 1075 1. Use the list of sub-terms from the original term as our "todo" queue, a list is a good structure for this.
 1076 2. Use an accumulator type with fast snoc to accumulate the values as we compute them.
 1077 3. Convert the accumulator quickly into a final type that is fast to process/lookup in later.
 1078 
 1079 (this process was at one point made explicit with an interface, but in the end we just inlined it all away)
 1080 
 1081 We tried at least three variants of this:
 1082 1. List/MutableVector/Vector
 1083 2. List/Seq/Seq
 1084 3. List/DList/List
 1085 
 1086 Suprisingly, option 3 was just as performant as the others, so we opted to go with it for simplicity.
 1087 But there may well be a faster version.
 1088 -}
