   59 {- Note [ExMemoryUsage instances for non-constants]
   60 In order to calculate the cost of a built-in function we need to feed the 'ExMemory' of each
   61 argument to the costing function associated with the builtin. For a polymorphic builtin this means
   62 that we need to be able to compute the 'ExMemory' of the AST provided as an argument to the builtin.
   63 How do we do that? Our strategy is:
   64 
   65 1. if the AST is a wrapped constant, then calculate the 'ExMemory' of the constant
   66 2. if the AST is something else, return 1
   67 
   68 This is pretty reasonable: a polymorphic builtin *is* allowed to check if the AST that it got as an
   69 argument is a constant or not, and if it happens to be a constant, the builtin *is* allowed to use
   70 it whatever way it wishes (see Note [Builtins and Plutus type checking] for details). Hence a
   71 builtin may in fact do something ad hoc for constants and we need to account for this possibility in
   72 the costing machinery.
   73 
   74 But if the given AST is not a constant, the builtin can't do anything else with it, hence we simply
   75 return 1, meaning "the costing function can't use this 'ExMemory' in any non-vacuous way".
   76 
   77 See 'HasMeaningIn' for a full list of constraints determining what a builtin can do with values.
   78 
   79 However for all types of values, except the one used by the production evaluator, we implement
   80 'ExMemoryUsage' as a call to 'error'. Not because other evaluators don't compute costs during
   81 evaluation -- the CK machine for example does in fact compute them (because we share the same
   82 builtins machinery between all the evaluators and we want it to be efficient on the production path,
   83 hence it's easier to optimize it for all evaluators than just for the single production evaluator).
   84 And not because the resulting 'ExBudget' is not forced by an evaluator that doesn't care about
   85 costing -- it still gets forced (for the same reason).
   86 
   87 The actual reason why we call 'error' is because at the moment no builtin is supposed to have a
   88 costing function that actually computes the 'ExMemory' of the given AST. Currently, if the builtin
   89 takes an 'Opaque', it's not supposed to actually look inside of it (unlike with 'SomeConstant') and
   90 hence the costing function is supposed to ignore that argument. It is possible that we'll eventually
   91 decide to add such a builtin, so the current approach of throwing an 'error' is a precaution
   92 ensuring that we won't add any weirdness by accident.
   93 
   94 We don't call 'error' on the production path, because we don't want this risk in there. A failing
   95 test is fine, a failing reasonable transaction is not and we don't want to risk it, even if it seems
   96 very unlikely that such a failure could slip in.
   97 
   98 The way we ignore arguments in costing functions is by computing the 'ExMemory' of each of those
   99 arguments lazily. I.e. a call to 'memoryUsage' can only be forced within a costing function and
  100 never outside of one. We have to do this regardless of all the reasoning above: if we compute
  101 the 'ExMemory' of, say, a list strictly, then a builtin prepending an element to a list will
  102 have the complexity of O(length_of_the_list) (because computing the 'ExMemory' of a list requires
  103 traversing the list), while we of course want it to be O(1).
  104 -}

  106 -- | A lazy tree of costs. Convenient for calculating the costs of values of built-in types, because
  107 -- they may have arbitrary branching (in particular a 'Data' object can contain a list of 'Data'
  108 -- objects inside of it).
  109 --
  110 -- 'CostRose' gets collapsed to a lazy linear structure down the pipeline, so that we can
  111 -- stream the costs to the outside where, say, the CEK machine picks them up one by one and handles
  112 -- somehow (in particular, subtracts from the remaining budget).
  113 data CostRose = CostRose {-# UNPACK #-} !CostingInteger ![CostRose]
  114     deriving stock (Show)
  115 
  116 -- | Create a 'CostRose' containing a single cost.
  117 singletonRose :: CostingInteger -> CostRose
  118 singletonRose cost = CostRose cost []
  119 {-# INLINE singletonRose #-}
  120 
  121 -- See Note [Global local functions].
  122 -- This is one way to define the worker. There are many more, see
  123 -- https://github.com/IntersectMBO/plutus/pull/5239#discussion_r1151197471
  124 -- We chose this one, because it's the simplest (no CPS shenanigans) among the safest (retrieving
  125 -- the next element takes O(1) time in the worst case).
  126 --
  127 -- The algorithm is a variation of the defunctionalization technique (see this post in particular:
  128 -- https://www.joachim-breitner.de/blog/778-Don%e2%80%99t_think,_just_defunctionalize), except we
  129 -- don't want a tail-recursive loop and instead emit costs lazily to the outside (as it's the whole
  130 -- point of the lazy costing approach)
  131 flattenCostRoseGo :: CostRose -> [CostRose] -> CostStream
  132 flattenCostRoseGo (CostRose cost1 forest1) forest2 =
  133     case forest1 of
  134         -- The current subtree doesn't have its own subtrees.
  135         [] -> case forest2 of
  136             -- No more elements in the entire tree, emit the last cost.
  137             []                -> CostLast cost1
  138             -- There's at least one unhandled subtree encountered earlier, emit the current cost
  139             -- and collapse the unhandled subtree.
  140             rose2' : forest2' -> CostCons cost1 $ flattenCostRoseGo rose2' forest2'
  141         -- The current subtree has at least one its own subtree.
  142         rose1' : forest1' ->
  143             -- Emit the current cost and handle the list of subtrees of the current subtree.
  144             CostCons cost1 $ case forest1' of
  145                 -- Same as the case below, except this one avoids creating a chain of
  146                 -- @[] ++ ([] ++ ([] ++ <...>))@, which would make retrieving the next element an
  147                 -- O(depth_of_the_tree) operation in the worst case.
  148                 [] -> flattenCostRoseGo rose1' forest2
  149                 -- Add the remaining subtrees of the current subtree (@forest1'@) to the stack of
  150                 -- all other subtrees (@forest2@) and handle the new current subtree (@rose1'@).
  151                 _  -> flattenCostRoseGo rose1' $ forest1' ++ forest2
  152 
  153 -- | Collapse a 'CostRose' to a lazy linear stream of costs. Retrieving the next element takes O(1)
  154 -- time in the worst case regardless of the recursion pattern of the given 'CostRose'.
  155 flattenCostRose :: CostRose -> CostStream
  156 flattenCostRose (CostRose cost [])              = CostLast cost
  157 flattenCostRose (CostRose cost (rose : forest)) = CostCons cost $ flattenCostRoseGo rose forest
  158 {-# INLINE flattenCostRose #-}

  164 {- Note [Alternative memory usage instances].  The `memoryUsage` function provides
  165  a measure of the size of an object for costing purposes, the idea being that
  166  the time taken to execute a builtin, and the memory used to contain its result,
  167  will depend on the size of the inputs.  The name `memoryUsage` is perhaps a
  168  misnomer: it was originally supposed to measure (in 64-bit words) the heap
  169  space required to store an object, but this is not always the correct measure
  170  to use.  For example, the time taken by `AddInteger` or `MultiplyInteger` will
  171  depend on the logarithms of the inputs (and the logarithm is proportional to
  172  the memory occupied by the inputs), and the memory occupied by the result will
  173  be some function of the memory occupied by the inputs, so for these functions
  174  the actual memory usage is a sensible size measure.  However, calling
  175  `replicateByte n b` function allocates a number of bytes which is equal to the
  176  actual value of `n`, which will be exponentially greater than the memory
  177  occupied by `n`, so this case the memory usage is not a sensible size measure.
  178  In most cases the default `memoryUsage` function returns the actual memory
  179  usage, but to deal with cases like `replicateByte` we occasionally use newtype
  180  wrappers which define a different size measure (see `IntegerCostedLiterally`
  181  below).  Polymorphic types require some care though: see Note [ExMemoryUsage
  182  for polymorphic types].
  183 -}
  184 
  185 {- Note [ExMemoryUsage for polymorphic types].  For polymorphic types such as
  186  `pair, `list`, and `array` DO NOT use newtype wrappers to define alternative
  187  size measures.  The denotations of functions which take polymorphic arguments
  188  use `SomeConstant` and this will ignore newtype wrappers and will only use the
  189  default `memoryUsage` function, which could lead to unexpected results.
  190  Furthermore, actual memory usage is typically not a good size measure for
  191  polymorphic arguments: the time taken to process a list, for example, will
  192  typically depend only on the length of the list and not the size of the
  193  contents.  Currently all such functions are parametrically polymorphic and only
  194  manipulate pointers without inspecting the contents of their polymorphic
  195  arguments, so it is reasonable to use size measures which depend only on the
  196  surface structure of polymorphic objects. -}

  198 {- We expect that all builtins which involve pairs will be constant cost and so
  199    their memory usage will never be involved in any computations.  The memory
  200    usage is set to maxBound so that we'll notice if this assumption is ever
  201    violated -}
  202 instance ExMemoryUsage (a, b) where
  203     memoryUsage _ = singletonRose maxBound

  206 {- Note the the `memoryUsage` of an empty list is zero.  This shouldn't cause any
  207  problems, but be sure to check that no costing function involving lists can
  208  return zero for an empty list (or any other input).
  209 -}
  210 {- Calculating the memory usage by processing the entire spine of the list eagerly
  211  is safe because there's no way to cheaply construct a long list: you either
  212  make one using repeated mkCons, which is expensive, or return one from a
  213  builtin, which has to be appropriately expensive too. -}
  214 instance ExMemoryUsage [a] where
  215   memoryUsage l = singletonRose . fromIntegral $ length l
  216   {-# INLINE memoryUsage #-}

  234 {- | Calculate a 'CostingInteger' for the size of the given 'Integer', measured in
  235 64-bit words.  This is the default size measure for `Integer`s.
  236 -}
  237 memoryUsageInteger :: Integer -> CostingInteger
  238 -- integerLog2# is unspecified for 0 (but in practice returns -1)
  239 -- ^ This changed with GHC 9.2: it now returns 0.  It's probably safest if we
  240 -- keep this special case for the time being though.
  241 memoryUsageInteger 0 = 1
  242 -- Assume 64 Int
  243 memoryUsageInteger i = fromIntegral $ I# (integerLog2# (abs i) `quotInt#` integerToInt 64) + 1
  244 -- So that the produced GHC Core doesn't explode in size, we don't win anything by inlining this
  245 -- function anyway.
  246 {-# OPAQUE memoryUsageInteger #-}

  249     memoryUsage i = singletonRose $ memoryUsageInteger i
  250     {-# INLINE memoryUsage #-}

  261 {- | When invoking a built-in function, a value of type `NumBytesCostedAsNumWords`
  262    can be used transparently as a built-in Integer but with a different size
  263    measure: see Note [Integral types as Integer].  This is required by the
  264    `integerToByteString` builtin, which takes an argument `w` specifying the
  265    width (in bytes) of the output bytestring (zero-padded to the desired size).
  266    The memory consumed by the function is given by `w`, *not* the size of `w`.
  267    The `NumBytesCostedAsNumWords` type wraps an Int `w` in a newtype whose
  268    `ExMemoryUsage` is equal to the number of eight-byte words required to
  269    contain `w` bytes, allowing its costing function to work properly.  We also
  270    use this for `replicateByte`.  If this is used to wrap an argument in the
  271    denotation of a builtin then it *MUST* also be used to wrap the same argument
  272    in the relevant budgeting benchmark.
  273 -}
  274 newtype NumBytesCostedAsNumWords = NumBytesCostedAsNumWords { unNumBytesCostedAsNumWords :: Integer }
  275 instance ExMemoryUsage NumBytesCostedAsNumWords where
  276     memoryUsage (NumBytesCostedAsNumWords n) = singletonRose . fromIntegral $ ((n-1) `div` 8) + 1
  277     {-# INLINE memoryUsage #-}
  278     -- Note that this uses `fromIntegral`, which will narrow large values to
  279     -- maxBound::SatInt = 2^63-1.  This shouldn't be a problem for costing because no
  280     -- realistic input should be that large; however if you're going to use this then be
  281     -- sure to convince yourself that it's safe.

  299 {- Bytestrings: we want the empty bytestring and bytestrings of length 1-8 to have
  300    size 1, bytestrings of length 9-16 to have size 2, etc.  Note that (-1)
  301    `quot` 8 == 0, so the code below gives the correct answer for the empty
  302    bytestring.  -}
  303 instance ExMemoryUsage BS.ByteString where
  304     -- Don't use `div` here!  That gives 0 instead of 1 for the empty bytestring.
  305     memoryUsage bs = singletonRose . unsafeToSatInt $ ((n - 1) `quot` 8) + 1 where
  306         n = BS.length bs
  307     {-# INLINE memoryUsage #-}

  309 instance ExMemoryUsage T.Text where
  310     -- This says that @Text@ allocates 1 'CostingInteger' worth of memory (i.e. 8 bytes) per
  311     -- character, which is a conservative overestimate (i.e. is safe) regardless of whether @Text@
  312     -- is UTF16-based (like it used to when we implemented this instance) or UTF8-based (like it is
  313     -- now).
  314     --
  315     -- Note that the @ExMemoryUsage Char@ instance does not affect this one, this is for performance
  316     -- reasons, since @T.length@ is O(1) unlike @sum . map (memoryUsage @Char) . T.unpack@. We used
  317     -- to have the latter, but changed it to the former for easy performance gains.
  318     --
  319     -- We may want to make this a bit less of an overestimate in future just not to overcharge
  320     -- users.
  321     memoryUsage = singletonRose . fromIntegral . T.length

  337 -- | Add two 'CostRose's. We don't make this into a 'Semigroup' instance, because there exist
  338 -- different ways to add two 'CostRose's (e.g. we could optimize the case when one of the roses
  339 -- contains only one element or we can make the function lazy in the second argument). Here we chose
  340 -- the version that is most efficient when the first argument is a statically known constant (we
  341 -- didn't do any benchmarking though, so it may not be the most efficient one) as we need this
  342 -- below.
  343 addConstantRose :: CostRose -> CostRose -> CostRose
  344 addConstantRose (CostRose cost1 forest1) (CostRose cost2 forest2) =
  345     CostRose (cost1 + cost2) (forest1 ++ forest2)
  346 {-# INLINE addConstantRose #-}

  348 {- A naive traversal for size.  This accounts for the number of nodes in a Data
  349    object and also the sizes of the contents of the nodes.  This is not ideal,
  350    but it seems to be the best we can do.  At present this only comes into play
  351    for 'equalsData', which is implemented using the derived implementation of
  352    '==' (fortunately the costing functions are lazy, so this won't be called for
  353    things like 'unBData' which have constant costing functions because they only
  354    have to look at the top node).  The problem is that when we call 'equalsData'
  355    the comparison will take place entirely in Haskell, so the costing functions
  356    for the contents of 'I' and 'B' nodes won't be called.  Thus if we just
  357    counted the number of nodes the sizes of 'I 2' and 'B <huge bytestring>'
  358    would be the same but they'd take different amounts of time to compare.  It's
  359    not clear how to trade off the costs of processing a node and processing the
  360    contents of nodes: the implementation below compromises by charging four
  361    units per node, but we may wish to revise this after experimentation.
  362 -}
  363 instance ExMemoryUsage Data where
  364     memoryUsage = sizeData where
  365         dataNodeRose = singletonRose 4
  366         {-# INLINE dataNodeRose #-}
  367 
  368         sizeData d = addConstantRose dataNodeRose $ case d of
  369             -- TODO: include the size of the tag, but not just yet. See PLT-1176.
  370             Constr _ l -> CostRose 0 $ l <&> sizeData
  371             Map l      -> CostRose 0 $ l >>= \(d1, d2) -> [d1, d2] <&> sizeData
  372             List l     -> CostRose 0 $ l <&> sizeData
  373             I n        -> memoryUsage n
  374             B b        -> memoryUsage b
  375
