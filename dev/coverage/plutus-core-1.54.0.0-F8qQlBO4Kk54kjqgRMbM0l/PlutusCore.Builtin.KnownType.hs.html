   67 {- Note [Performance of ReadKnownIn and MakeKnownIn instances]
   68 It's critically important that 'readKnown' runs in the concrete 'Either' rather than a general
   69 'MonadError'. Changing from the latter to the former gave us a speedup of up to 19%, see
   70 https://github.com/IntersectMBO/plutus/pull/4307
   71 
   72 Replacing the @AsUnliftingError err, AsEvaluationFailure err@ constraints with the dedicated
   73 'BuiltinError' data type gave us a speedup of up to 4%.
   74 
   75 All the same considerations apply to 'makeKnown':
   76 https://github.com/IntersectMBO/plutus/pull/4421
   77 
   78 It's beneficial to inline 'readKnown' and 'makeKnown' not only because we use them directly over
   79 concrete types once 'toBuiltinsRuntime' is inlined, but also because otherwise GHC compiles each of
   80 them to two definitions (one calling the other) for some reason.
   81 So always add an @INLINE@ pragma to all definitions of 'makeKnown' and 'readKnown' unless you have
   82 a specific reason not to.
   83 
   84 Neither 'readKnown' nor 'makeKnown' should appear in the generated Core for builtins. In most cases
   85 they would slow builtins down, but even if a 'readKnown' only throws an error, it still makes sense
   86 to keep it out of Core just not to trigger an investigation on whether it's fine that a call to
   87 'readKnown' is not inlined.
   88 
   89 Some 'readKnown' implementations require inserting a call to 'oneShot'. E.g. if 'oneShot' is not
   90 used in 'readKnownConstant' then 'GHC pulls @gshow uniExp@ out of the 'Nothing' branch, thus
   91 allocating a thunk of type 'String' that is completely redundant whenever there's no error,
   92 which is the majority of cases. And putting 'oneShot' as the outermost call results in
   93 worse Core.
   94 
   95 Any change to an instance of 'ReadKnownIn' or 'MakeKnownIn', even completely trivial, requires
   96 looking into the generated Core, since compilation of these instances is extremely brittle
   97 optimization-wise.
   98 
   99 Things to watch out for are unnecessary sharing (for example, a @let@ appearing outside of a @case@
  100 allocates a thunk and if that thunk is not referenced inside of one of the branches, then it's
  101 wasteful, especially when it's not referenced in the most commonly chosen branch) and type class
  102 methods not being extracted from the dictionary and used directly instead (i.e. if you see
  103 multiple @pure@ and @>>=@ in the code, that's not good). Note that neither @let@ nor @>>=@ are bad
  104 in general, we certainly do need to allocate thunks occasionally, it's just that when it comes to
  105 builtins this is rarely the case as most of the time we want aggressive inlining and specialization
  106 and the "just compute the damn thing" behavior.
  107 -}

  109 {- Note [Unlifting terminology]
  110 This function:
  111 
  112     f :: Integer -> CkValue DefaultUni fun
  113     f = VCon . Some . ValueOf DefaultUniInteger
  114 
  115 lifts an 'Integer' to 'CkValue'. Unlifting is the opposite:
  116 
  117     g :: CkValue DefaultUni fun -> Maybe Integer
  118     g (VCon (Some (ValueOf uni x))) = case uni of
  119         DefaultUniInteger -> Just x
  120         _                 -> Nothing
  121 
  122 The following usages of the "unlift" term are grammatical:
  123 
  124 1. unlift a 'CkValue' to 'Integer'
  125 2. unlift to 'Integer'
  126 3. unlift a 'CkValue' as an 'Integer'
  127 4. unlift from the 'VCon' constructor (or just 'VCon') to 'Integer'
  128 
  129 We call the integer that @g@ returns "the unlifted integer".
  130 -}

  132 {- Note [Unlifting a term as a value of a built-in type]
  133 See Note [Unlifting terminology] first.
  134 
  135 It's trivial to unlift a term to a monomorphic built-in type like 'Integer': just check that the
  136 term is a constant, extract the type tag and check it for equality with the type tag of 'Integer'.
  137 
  138 Things work the same way for a fully monomorphized polymorphic type, i.e. @(Integer, Bool@) is not
  139 any different from just 'Integer' unlifting-wise.
  140 
  141 (TODO: the following explanation needs to be improved, there's PLT-338 for that)
  142 
  143 However there's no sensible way of unlifting to, say, @[a]@ where @a@ in not a built-in type. So
  144 let's say we instantiated @a@ to an @Opaque val rep@ like we do for polymorphic functions that don't
  145 deal with polymorphic built-in types (e.g. @id@, @ifThenElse@ etc). That would mean we'd need to
  146 write a function from a @[a]@ for some arbitrary built-in @a@ to @[Opaque val a]@. Which is really
  147 easy to do: it's just @map makeKnown@. But the problem is, unlifting is supposed to be cheap and
  148 that @map@ is O(n), so for example 'MkCons' would become an O(n) operation making perfectly linear
  149 algorithms quadratic. See https://github.com/IntersectMBO/plutus/pull/4215 for how that would
  150 look like.
  151 
  152 So the problem is that we can't convert in O(1) time a @[a]@ coming from a constant of
  153 statically-unknown type (that @a@ is existential) to @[a']@ where @a'@ is known statically.
  154 Thus it's impossible to instantiate @a@ in Haskell's
  155 
  156     nullList :: [a] -> Bool
  157 
  158 so that there's a 'TypeScheme' for this function.
  159 
  160 One non-solution would be to instantiate @a@, then recurse on the type, construct a new function
  161 that defers to the original @nullList@ but wraps its argument in a specific way (more on that below)
  162 making it possible to assign a 'TypeScheme' to the resulting function. Astonishingly enough, that
  163 could actually work and if we ever write a paper on builtins, we should mention that example, but:
  164 
  165 1. such a trick requires a generic machinery that knows how to check that the head of the builtin
  166    application is a particular built-in type. We used to have that, but it was just way too slow
  167 2. that would only work for functions that don't care about @a@ at all. But for example when
  168    elaborating @cons :: a -> [a] -> [a]@ as a Plutus builtin we need to unlift both the arguments
  169    and check that their @a@s are equal
  170    (See Note [Representable built-in functions over polymorphic built-in types])
  171    and it's either way too complex or even impossible to do that automatically within some generic
  172    machinery
  173 
  174 So what we do is we simply require the user to write
  175 
  176     nullList :: SomeConstant uni [a] -> Bool
  177 
  178 and unlift to @[a]@ manually within the definition of the builtin. This works, because the
  179 existential @a@ never escapes the definition of the builtin. I.e. it's fine to unpack an existential
  180 and use it immediately without ever exposing the existential parts to the outside and it's not fine
  181 to try to return a value having an existential inside of it, which is what unlifting to @[a]@ would
  182 amount to.
  183 
  184 Could we somehow align the unlifting machinery so that it does not construct values of particular
  185 types, but rather feeds them to a continuation or something, so that the existential parts never
  186 try to escape? Maybe, but see point 2 from the above, we do want to get our hands on the particular
  187 universes sometimes and point 1 prevents us from doing that generically, so it doesn't seem like
  188 we could do that within some automated machinery.
  189 
  190 Overall, asking the user to manually unlift a @SomeConstant uni [a]@ is just always going to be
  191 faster than any kind of fancy encoding.
  192 -}

  194 {- Note [Alignment of ReadKnownIn and MakeKnownIn]
  195 We keep 'ReadKnownIn' and 'MakeKnownIn' separate, because values of some types can only be lifted
  196 and not unlifted, for example 'EvaluationResult' and 'Emitter' can't appear in argument position.
  197 
  198 'KnownTypeAst' is not a superclass of ReadKnownIn and MakeKnownIn. This is due to the fact that
  199 polymorphic built-in types are only liftable/unliftable when they're fully monomorphized, while
  200 'toTypeAst' works for polymorphic built-in types that have type variables in them, and so the
  201 constraints are completely different in the two cases and we keep the two concepts apart
  202 (there doesn't seem to be any cons to that).
  203 -}

  205 {- Note [Allowed unlifting and lifting]
  206 Read Note [Alignment of ReadKnownIn and MakeKnownIn] first.
  207 
  208 The following classes of Haskell types represent Plutus types:
  209 
  210 1. monomorphic built-in types such as @Bool@
  211    (assuming @Bool@ is in the universe)
  212 2. polymorphic built-in types such as @(a, b)@ for any @a@ and @b@ representing Plutus types
  213    (assuming @(,)@ is in the universe)
  214 3. @Opaque val rep@ for any @rep@ representing a Plutus type
  215 4. @SomeConstant uni rep@ for any @rep@ representing a Plutus type
  216 5. @Emitter a@ for any @a@ representing a Plutus type
  217 6. @EvaluationResult a@ for any @a@ representing a Plutus type
  218 7. 'TyVarRep', 'TyAppRep', 'TyForallRep' and all similar types mirroring constructors of @Type@
  219 8. @a -> b@ for any @a@ and @b@ representing Plutus types (mirrors 'TyFun')
  220 9. anything else that has a 'KnownTypeAst' instance, for example we express the
  221    @KnownTypeAst DefaultUni Int@ instance in terms of the @KnownType DefaultUni Integer@
  222    one
  223 
  224 Unlifting is allowed to the following classes of types:
  225 
  226 1. monomorphic built-in types such as @Bool@
  227 2. monomorphized polymorphic built-in types such as @(Integer, Text)@
  228 3. @Opaque val rep@ for @rep@ representing a Plutus type
  229 4. @SomeConstant uni rep@ for @rep@ representing a Plutus type
  230 5. anything else that implements 'ReadKnownIn', for example we express the
  231    @ReadKnownIn DefaultUni term Int@ instance in terms of the @ReadKnownIn DefaultUni term Integer@
  232    one, and for another example define an instance for 'Void' in tests
  233 
  234 Lifting is allowed to the following classes of types:
  235 
  236 1. monomorphic built-in types such as @Bool@
  237 2. monomorphized polymorphic built-in types such as @(Integer, Text)@
  238 3. @Opaque val rep@ for @rep@ representing a Plutus type
  239 4. @SomeConstant uni rep@ for @rep@ representing a Plutus type
  240 5. @Emitter a@ for any @a@ that lifting is allowed to
  241 6. @EvaluationResult a@ for any @a@ that lifting is allowed to
  242 7. anything else that implements 'MakeKnownIn', for example we express the
  243    @MakeKnownIn DefaultUni term Int@ instance in terms of the @MakeKnownIn DefaultUni term Integer@
  244    one, and for another example define an instance for 'Void' in tests
  245 -}

  270 -- See Note [Unlifting a term as a value of a built-in type].
  271 -- | Convert a constant embedded into a PLC term to the corresponding Haskell value.
  272 readKnownConstant :: forall val a. KnownBuiltinType val a => val -> ReadKnownM a
  273 -- See Note [Performance of ReadKnownIn and MakeKnownIn instances]
  274 readKnownConstant val = asConstant val >>= oneShot \case
  275     Some (ValueOf uniAct x) -> do
  276         let uniExp = knownUni @_ @(UniOf val) @a
  277         -- 'geq' matches on its first argument first, so we make the type tag that will be known
  278         -- statically (because this function will be inlined) go first in order for GHC to
  279         -- optimize some of the matching away.
  280         case uniExp `geq` uniAct of
  281             Just Refl -> pure x
  282             Nothing   -> throwError $ BuiltinUnliftingEvaluationError $ typeMismatchError uniExp uniAct
  283 {-# INLINE readKnownConstant #-}

  285 -- | A non-empty spine. Isomorphic to 'NonEmpty', except is strict and is defined as a single
  286 -- recursive data type.
  287 data Spine a
  288     = SpineLast a
  289     | SpineCons a (Spine a)
  290     deriving stock (Show, Eq, Foldable, Functor)
  291 
  292 -- | The head-spine form of an iterated application. Provides O(1) access to the head of the
  293 -- application. @NonEmpty a ~ HeadSpine a a@, except is strict and the no-spine case is made a separate
  294 -- constructor for performance reasons (it only takes a single pattern match to access the head when
  295 -- there's no spine this way, while otherwise we'd also need to match on the spine to ensure that
  296 -- it's empty -- and the no-spine case is by far the most common one, hence we want to optimize it).
  297 --
  298 -- Used in built-in functions returning function applications such as 'CaseList'.
  299 data HeadSpine a b
  300     = HeadOnly a
  301     | HeadSpine a (Spine b)
  302     deriving stock (Show, Eq, Functor)
  303 
  304 -- | @HeadSpine@ but the type of head and spine is same
  305 type MonoHeadSpine a = HeadSpine a a
  306 
  307 instance Bifunctor HeadSpine where
  308   bimap headF _ (HeadOnly a)         = HeadOnly $ headF a
  309   bimap headF spineF (HeadSpine a b) = HeadSpine (headF a) (spineF <$> b)
  310 
  311 -- | Construct @HeadSpine@ from head and list.
  312 headSpine :: a -> [b] -> HeadSpine a b
  313 headSpine h [] = HeadOnly h
  314 headSpine h (x:xs) =
  315   -- It's critical to use 'foldr' here, so that deforestation kicks in.
  316   -- See Note [Definition of foldl'] in "GHC.List" and related Notes around for an explanation
  317   -- of the trick.
  318   HeadSpine h $ foldr (\x2 r x1 -> SpineCons x1 $ r x2) SpineLast xs x
  319 {-# INLINE headSpine #-}
  320 
  321 -- |
  322 --
  323 -- >>> import Text.Pretty
  324 -- >>> pretty (SpineCons 'a' $ SpineLast 'b')
  325 -- [a, b]
  326 instance Pretty a => Pretty (Spine a) where pretty = pretty . map Identity . toList
  327 instance PrettyBy config a => DefaultPrettyBy config (Spine a)
  328 deriving via PrettyCommon (Spine a)
  329     instance PrettyDefaultBy config (Spine a) => PrettyBy config (Spine a)
  330 
  331 -- |
  332 --
  333 -- >>> import Text.Pretty
  334 -- >>> pretty (HeadOnly 'z')
  335 -- z
  336 -- >>> pretty (HeadSpine 'f' (SpineCons 'x' $ SpineLast 'y'))
  337 -- f `applyN` [x, y]
  338 instance (Pretty a, Pretty b) => Pretty (HeadSpine a b) where
  339     pretty (HeadOnly x)     = pretty x
  340     pretty (HeadSpine f xs) = pretty f <+> "`applyN`" <+> pretty xs
  341 instance (PrettyBy config a, PrettyBy config b) => DefaultPrettyBy config (HeadSpine a b)
  342 deriving via PrettyCommon (HeadSpine a b)
  343     instance PrettyDefaultBy config (HeadSpine a b) => PrettyBy config (HeadSpine a b)

  345 -- See Note [Performance of ReadKnownIn and MakeKnownIn instances].
  346 class uni ~ UniOf val => MakeKnownIn uni val a where
  347     -- | Convert a Haskell value to the corresponding PLC value.
  348     -- The inverse of 'readKnown'.
  349     makeKnown :: a -> BuiltinResult val
  350     default makeKnown :: KnownBuiltinType val a => a -> BuiltinResult val
  351     -- Everything on evaluation path has to be strict in production, so in theory we don't need to
  352     -- force anything here. In practice however all kinds of weird things happen in tests and @val@
  353     -- can be non-strict enough to cause trouble here, so we're forcing the argument. Looking at the
  354     -- generated Core, the forcing amounts to pulling a @case@ out of the 'fromConstant' call,
  355     -- which doesn't affect the overall cost and benchmarking results suggest the same.
  356     --
  357     -- Note that the value is only forced to WHNF, so care must be taken to ensure that every value
  358     -- of a type from the universe gets forced to NF whenever it's forced to WHNF.
  359     makeKnown x = pure . fromValue $! x
  360     {-# INLINE makeKnown #-}

  396 -- Catching 'EvaluationFailure' here would allow *not* to short-circuit when 'readKnown' fails
  397 -- to read a Haskell value of type @a@. Instead, in the denotation of the builtin function
  398 -- the programmer would be given an explicit 'EvaluationResult' value to handle, which means
  399 -- that when this value is 'EvaluationFailure', a PLC 'Error' was caught.
  400 -- I.e. it would essentially allow us to catch errors and handle them in a programmable way.
  401 -- We forbid this, because it complicates code and isn't supported by evaluation engines anyway.

  403         ( TypeError ('Text "‘BuiltinResult’ cannot appear in the type of an argument")
  404         , uni ~ UniOf val

  406     readKnown _ = throwError underTypeError
  407     {-# INLINE readKnown #-}
