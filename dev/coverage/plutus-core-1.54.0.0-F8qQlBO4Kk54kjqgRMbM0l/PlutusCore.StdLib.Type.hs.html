   22 {- Note [Arity of patterns functors]
   23 The arity of a pattern functor is the number of arguments the pattern functor receives in addition
   24 to the first argument representing the recursive case. So
   25 @f :: * -> *@                           has arity 0
   26 @f :: (k -> *) -> k -> *@               has arity 1
   27 @f :: (k1 -> k2 -> *) -> k1 -> k2 -> *@ has arity 2
   28 etc
   29 -}

   31 {- Note [InterList]
   32 This data type is much like the 'list' data type, but it receives two types arguments rather than one
   33 and "interleaves" them (see 'example_InterList').
   34 
   35     data InterList a b
   36         = InterNil
   37         | InterCons a b (InterList b a)
   38 
   39     example_InterList :: InterList Char Int
   40     example_InterList = InterCons 'a' 1 . InterCons 2 'b' . InterCons 'c' 3 $ InterNil
   41 
   42 The data type is interesting, because we need some way of getting
   43 
   44     fix2 :: ((k1 -> k2 -> *) -> k1 -> k2 -> *) -> k1 -> k2 -> *
   45 
   46 in order to encode it directly, so we use this data type in examples in order to show admissibility
   47 of 'fix2' which is an instance of the more generic
   48 
   49     fix :: (k -> k) -> k
   50 
   51 I.e. we show how the more generic 'fix' can be encoded for any particular 'k' by taking
   52 @k ~ (k1 -> k2 -> *)@ as example and constructing 'fix2'.
   53 -}

   55 {- Note [Natural representation]
   56 Having @fix :: (* -> *) -> *@ we can easily define the @list@ data type as a fixed point of
   57 an appropriate pattern functor:
   58 
   59     listF = \(a :: *) (list :: *) -> all (r :: *). r -> (a -> list -> r) -> r
   60     list  = \(a :: *) -> fix (listF a) a
   61 
   62 There are a few problems with this definition however:
   63 
   64 1. In @listF@ there is no indication that @list@ is supposed to contain elements of type @a@.
   65 So @listF@ binds both @a@ and @list@, but does not specify there is a relation between these two
   66 things. The burden of connecting @a@ and @list@ together is on the caller, which is not a big deal,
   67 because the only callers are @fix@, in terms of which the data type is defined, and @wrap@ that
   68 allows to define the constructors of the data type, but still, this way the code looks strangely
   69 structured.
   70 
   71 2. Related to 1: such encoding diverges from what one would write having a data construction
   72 machinery. A standard Haskell definition would be
   73 
   74     data List a
   75         = Nil
   76         | Cons a (List a)
   77 
   78 In this definition we explicitly apply @List@ to @a@ in the @Cons@ case. Thus, the encoding looks
   79 somewhat unnatural.
   80 
   81 3. @wrap@ constructing a @list@ must carry @listF a@ in the same way @fix@ carries it. This makes
   82 it very hard to construct terms using the AST directly as shown in
   83 @plutus/plutus-core/docs/Holed types.md@.
   84 
   85 4. There are data types that can't be defined this way. See Note [InterList] for one example.
   86 
   87 There is however an approach that allows to encode data types in a "natural" way, does not cause
   88 any trouble while constructing data types and can handle much more data types than what is shown
   89 above. Here is how the @list@ example looks like with it:
   90 
   91     listF = \(list :: * -> *) (a :: *) -> all (r :: *). r -> (a -> list a -> r) -> r
   92     list  = \(a :: *) -> fix listF a
   93 
   94 I.e. instead of tying the knot over @list :: *@ we tie the knot over @list :: * -> *@. This simple
   95 trick solves all the problems described avove.
   96 
   97 But the code is actually ill-kinded. The reason for this is that @fix :: (* -> *) -> *@ is no longer
   98 enough, because we're taking a fixed point of a pattern functor of kind @(* -> *) -> * -> *@
   99 rather than just @* -> *@. Hence we need a more permissive fixed-point operator.
  100 
  101 Read next: Note [The kind of fix].
  102 -}

  104 {- Note [The kind of fix]
  105 In Note [Natural representation] we concluded that @fix :: (* -> *) -> *@ is not enough to encode
  106 @list@ in a satisfying way. In that particular case it would be enough to have
  107 
  108     fix :: ((* -> *) -> * -> *) -> * -> *
  109 
  110 but what about other cases? The example from Note [InterList] requires
  111 
  112     fix :: ((* -> * -> *) -> * -> * -> *) -> * -> * -> *
  113 
  114 and of course we still need
  115 
  116     fix :: (* -> *) -> *
  117 
  118 occasionally. This suggests to change the kind signature of @fix@ to
  119 
  120     fix :: (k -> k) -> k
  121 
  122 which covers all those cases. However, it's not clear how to implement such @fix@.
  123 See @docs/fomega/deep-isorecursive/README.md@ for details.
  124 
  125 But it turns out that
  126 
  127     ifix :: ((k -> *) -> k -> *) -> k -> *
  128 
  129 is enough for all cases.
  130 
  131 Read next: Note [Packing n-ary pattern functors semantically].
  132 -}

  134 {- Note [Packing n-ary pattern functors semantically]
  135 An n-ary pattern functor has the following generic signature:
  136 
  137     patN :: k -> k
  138 
  139 where @k@ is of the @k1 -> k2 -> ... -> *@ form. We need to encode 'patN' as an equivalent 1-ary
  140 pattern functor with this signature:
  141 
  142     pat1 :: ((k' -> *) -> k' -> *
  143 
  144 because that's what 'ifix' accepts.
  145 
  146 @plutus/docs/fomega/mutual-type-level-recursion/IFixIsEnough.agda@ describes the encoding trick
  147 at great detail, but let's look at an example here. The pattern functor of 'InterList'
  148 (see Note [InterList]) is defined as
  149 
  150     interlistF =
  151         \(interlist :: * -> * -> *) (a :: *) (b :: *) ->
  152             all (r :: *). r -> (a -> b -> interlist b a -> r) -> r
  153 
  154 We can't pass the pattern functor to 'ifix', because it's of kind
  155 
  156     ((* -> * -> *) -> * -> * -> *) -> * -> * -> *
  157 
  158 So we're going to "pack" the pattern functor to make it a 1-ary one. The idea is simple:
  159 instead of passing two arguments to the recursive case, we pass a single continuation that applies
  160 a function it receives to those two arguments. Morever, we can define the packed version of
  161 'interlistF' in terms of 'interlistF' itself. It looks like this:
  162 
  163     withSpine =
  164         \(rec :: ((* -> * -> *) -> *) -> *) ->
  165             \(a :: *) (b :: *) -> rec (\(interlist :: * -> * -> *) -> interlist a b)
  166 
  167     interlistF' =
  168         \(rec :: ((* -> * -> *) -> *) -> *) (spine :: (* -> * -> *) -> *) ->
  169             spine (\(a :: *) (b :: *) -> interlistF (withSpine rec) a b)
  170 
  171 Here 'spine' encapsulates 'a' and 'b' as arguments passed to a function 'spine' receives.
  172 This even can be guessed from its signature:
  173 
  174     spine :: (* -> * -> *) -> *
  175 
  176 which can be read as "give me a function of two arguments and I'll provide those arguments and
  177 return the result".
  178 
  179 'withSpine' on the other hand receives
  180 
  181 1. a function that expects a CPS-transformed spine
  182 2. two arguments, 'a' and 'b', which together form a spine that is not CPS-trasformed
  183 
  184 and then 'withSpine' applies said function to the spine by CPSing it first.
  185 
  186 So nothing interesting happens here: we just pack/unpack spines using continuations.
  187 
  188 If we eta-contract @interlistF'@, we'll get
  189 
  190     interlistF' =
  191         \(rec :: ((* -> * -> *) -> *) -> *) (spine :: (* -> * -> *) -> *) ->
  192             spine (interlistF (withSpine rec))
  193 
  194 And this can be generalized to arbitrary n-ary pattern functors:
  195 
  196     toPat1 =
  197         \(withSpine :: ((k -> *) -> *) -> k) (patN :: k -> k) ->
  198             \(rec :: (k -> *) -> *) (spine :: k -> *) ->
  199                 spine (pat (withSpine rec))
  200 
  201 which reads like this: having 'withSpine' constructed for a particular 'k' and an n-ary pattern
  202 functor of kind @k -> k@ we can get a 1-ary pattern functor of kind
  203 
  204     ((k -> *) -> *) -> (k -> *) -> *
  205 
  206 We derive various 'withSpine's automatically on the Haskell side from 'k' itself.
  207 
  208 Read next: Note [Generic fix].
  209 -}

  211 {- Note [Generic fix]
  212 Now that we know how to pack n-ary functors into 1-ary ones
  213 (see [Packing n-ary pattern functors semantically]), only a few tiny steps remain to get the generic
  214 
  215     fix :: (k -> k) -> k
  216 
  217 from just
  218 
  219     ifix :: ((i -> *) -> i -> *) -> i -> *
  220 
  221 Having @pat :: k -> k@ we can pack it as
  222 
  223     toPat1 withSpine patN :: ((k -> *) -> *) -> (k -> *) -> *
  224 
  225 (where 'withSpine' is constructed automatically from 'k' on the Haskell side) and we can apply
  226 'ifix' to this 1-ary pattern functor and get
  227 
  228     ifix (toPat1 withSpine patN) :: (k -> *) -> *
  229 
  230 It only remains to turn something of kind @(k -> *) -> *@ into something of kind @*@, i.e. to define
  231 a type function of kind @((k -> *) -> *) -> k@. But we already have such a function: 'withSpine',
  232 so the final encoding is
  233 
  234     fix = \(patN :: k -> k) -> withSpine (ifix (toPat1 withSpine patN))
  235 
  236 The meaning of 'withSpine' here is the same as we've seen before: we use it to pack @n@ type
  237 arguments as a single CPS-encoded spine and pass it to some function.
  238 
  239 Summarizing, 'fix' receives an n-ary pattern functor and @n@ type arguments, the pattern functor
  240 gets packed as a 1-ary one, the type arguments get packed into a single CPS-encoded spine and
  241 'ifix' gets applied to the 1-ary pattern functor and the spine.
  242 
  243 Read next: Note [Encoded InterList].
  244 -}

  246 {- Note [Encoded InterList]
  247 Let's now look at an example.
  248 
  249 Recall that the pattern functor of 'interlist' is
  250 
  251     interlistF =
  252         \(interlist :: * -> * -> *) (a :: *) (b :: *) ->
  253             all (r :: *). r -> (a -> b -> interlist b a -> r) -> r
  254 
  255 We can apply generic 'fix' (see Note [Generic fix]) to this pattern functor directly:
  256 
  257     fix interlistF :: * -> *
  258 
  259 which after eta-expansion and some reductions becomes
  260 
  261     \(a :: *) -> withSpine (ifix (toPat1 withSpine interlistF)) a
  262 
  263 (as per Note [Generic fix]) which after some more reductions becomes
  264 
  265     -- Two type arguments that the data type receives and the 'ifix' primitive.
  266     \(a :: *) (b -> *) -> ifix
  267         -- The variable responsible for recursion and the variable representing a CPS-encoded spine
  268         -- of two elements. Note that the kind of the argument that the variable responsible for
  269         -- recursion receives is the same as the kind of 'spine', i.e. we always instantiate
  270         -- recursion at some spine.
  271         (\(rec :: ((* -> * -> *) -> *) -> *) -> \(spine :: (* -> * -> *) -> *) ->
  272             -- 'spine' unpacks a CPS-encoded spine and passes all its elements to a continuation.
  273             spine
  274               -- The 'interlistF' pattern functor given above applied to a function that receives
  275               -- two type arguments, packs them as a CPS-encoded spine and passes the spine to the
  276               -- variable responsible for recursion.
  277               (interlistF (\(a :: *) (b :: *) -> rec (\(dat :: * -> * -> *) -> dat a b)))
  278         )
  279         -- The two type arguments packed as a CPS-encoded spine.
  280         (\(dat :: * -> * -> *) -> dat a b)
  281 
  282 We've elaborated the encoding on example, but there is a problem to consider here.
  283 Read next: Note [Denormalization]
  284 -}

  286 {- Note [Denormalization]
  287 Originally, we were binding 'withSpine' and 'patN' (taken from the end of Note
  288 [Packing n-ary pattern functors semantically]) on the Plutus Core side and this
  289 resulted in huge unreadable types being produced. Now we bind 'withSpine',
  290 'patN' and what 'withSpine' receives on the Haskell side, i.e. we use Haskell
  291 lambdas to bind variables and regular function application to eliminate those
  292 lambdas which allows us to defer type reduction business to Haskell.  Here is
  293 how the definition of 'list' looks like:
  294 
  295     \(a :: *) -> ifix
  296         (\(rec :: ((* -> *) -> *) -> *) -> \(spine :: (* -> *) -> *) ->
  297             spine (listF (\(a :: *) -> rec (\(dat :: * -> *) -> dat a)))
  298         )
  299         (\(dat :: * -> *) -> dat a)
  300 
  301 This is pretty readable (once you know how to read it, see Note [Encoded InterList] for a similar
  302 example) and doesn't contain any 'withSpine' or 'patN' variables, but if we inline 'listF', we'll get
  303 
  304     (\(list :: * -> *) -> \(a :: *) -> all (r :: *). r -> (a -> list a -> r) -> r)
  305     (\(a :: *) -> rec (\(dat :: * -> *) -> dat a))
  306 
  307 which is an applied lambda abstraction. This essentially means that in the pattern functor of 'list'
  308 
  309     \(a :: *) -> all (r :: *). r -> (a -> list a -> r) -> r
  310 
  311 'list' is defined as
  312 
  313     \(a :: *) -> rec (\(dat :: * -> *) -> dat a)
  314 
  315 This all is fine, that's how our encoding trick works, but note that we produced a type that is not
  316 in normal form. This is a bit worrying: the user writes something that looks like it's normalized,
  317 but in the end types are not normalized due to how the encoding works. In Plutus Core we have two
  318 modes for type checking:
  319 
  320 1. off-chain, type normalization is allowed
  321 2. on-chain, type normalization is not allowed and types must already be normalized
  322 
  323 Thus, we do care about whether types are normalized or not. In the compilation pipeline we just
  324 explicitly normalize types whenever normalized types are required, but since this module belongs
  325 to a library it better be general and not rely on particular details of downstream code.
  326 
  327 Preserving properties of user-written code is generally a good idea while transforming it,
  328 so we also do not want to remove redexes from user-written code and thus we can't just normalize
  329 everything in sight to overcome the denormalization problem.
  330 
  331 Then the question is whether it's possible to preserve redexes in user-written types and not to
  332 produce new ones while encoding the types. And the answer is "yes, but it's too costly".
  333 
  334 But read Note [Spiney API] first.
  335 -}

  337 {- Note [Spiney API]
  338 Encoding of n-ary pattern functors into 1-ary pattern functors is hidden behind an API that pretends
  339 our types are in head-spine form. See @plutus/docs/fomega/deep-isorecursive/alternatives.md@ for
  340 details and discussion about the head-spine form approach.
  341 
  342 The reasons for providing such API are
  343 
  344 1. it's simple
  345 2. it hides all the gory details in such a way that we can change the representation of types and
  346 not change the API. For example, we can encode pattern functors in different ways (and we, in fact,
  347 do this) or we even can have the head-spine form in the AST and that wouldn't affect the API
  348 
  349 We could have an API like this: the user provides an n-ary pattern functor and we manipulate the AST
  350 directly which may or may not involve deconstruction of the AST depending on how we perform encoding.
  351 However the user might provide something that is not a pattern functor, but computes to a pattern
  352 functor and everything becomes more complicated. Instead, we require that the user provides the name
  353 of the data type being defined, a list of type variables (the ones that the pattern functor binds)
  354 along with their kinds and the body of the pattern functor separately. Having this information is
  355 enought to perform whatever encoding we want. Here is how it looks on the 'interlist' example:
  356 
  357 the n-ary pattern functor of 'interlist' is
  358 
  359     \(interlist :: * -> * -> *) (a :: *) (b :: *) ->
  360         all (r :: *). r -> (a -> b -> interlist b a -> r) -> r
  361 
  362 and we require the user (where "the user" means someone generating Plutus Core or writing it directly,
  363 i.e. either someone writing a compiler to Plutus Core or one of the creators of the language) to split
  364 this type into three components
  365 
  366 1. "interlist"         -- the name of the data type
  367 2. @[a :: *, b :: *]@  -- other type variables the pattern functors binds along with their kinds
  368 3. @all (r :: *). r -> (a -> b -> interlist b a -> r) -> r@  -- the body of the pattern functor
  369 
  370 and pass them to the 'makeRecursiveType' function (which also receives an annotation as its first
  371 argument just so that we have something to place in the AST when needed). Note that we do not require
  372 to provide the kind of 'interlist', because we can compute it from the kinds of other type variables.
  373 
  374 The code constructing the data type itself:
  375 
  376     -- Introduce names in scope.
  377     [a, b, interlist, r] <- traverse freshTyName ["a", "b", "interlist", "r"]
  378 
  379     -- Define some aliases.
  380     let interlistBA = mkIterTyAppNoAnn (TyVar () interlist) [TyVar () b, TyVar () a]
  381         nilElimTy   = TyVar () r
  382         consElimTy  = mkIterTyFun () [TyVar () a, TyVar () b, interlistBA] $ TyVar () r)
  383 
  384     -- Construct the actual data type.
  385     makeRecursiveType () interlist [TyVarDecl () a $ Type (), TyVarDecl () b $ Type ()]
  386         . TyForall () r (Type ())  -- all (r :: *).
  387         . TyFun () nilElimTy       --     r ->
  388         . TyFun () consElimTy      --         (a -> b -> interlist b a -> r) ->
  389         $ TyVar () r               --             r
  390 
  391 So for the user the interface this module provides is rather simple considering how much stuff is
  392 hidden behind it.
  393 
  394 Read next: Note [Packing n-ary pattern functors syntactically]
  395 -}

  397 {- Note [Packing n-ary pattern functors syntactically]
  398 Now that we know how the API looks like (see Note [Spiney API]), we can discuss a solution to
  399 the denormalization problem (see Note [Denormalization]).
  400 
  401 Recall (or see Note [Packing n-ary pattern functors semantically]) that if we pack the pattern
  402 functor of 'list'
  403 
  404     listF = \(list :: * -> *) (a :: *) -> all (r :: *). r -> (a -> list a -> r) -> r
  405 
  406 semantically and normalize the result, we'll get
  407 
  408     semListF =
  409         \(rec :: ((* -> *) -> *) -> *) -> \(spine :: (* -> *) -> *) ->
  410             spine (\(a :: *) -> all (r :: *). r -> (a -> rec (\(dat :: * -> *) -> dat a) -> r) -> r)
  411 
  412 The question is how to get the same without full-scale normalization.
  413 
  414 In this particular case it's easy: since we receive the body of 'listF' separately from the variables
  415 the leading lambdas bind, we can simply enclose the body of 'listF' like this:
  416 
  417     \(rec :: ((* -> *) -> *) -> *) -> \(spine :: (* -> *) -> *) ->
  418                 spine (\(a :: *) -> <body_of_listF>)
  419 
  420 and replace each occurrence of @list a@ by
  421 
  422     rec (\(dat :: * -> *) -> dat a)
  423 
  424 and that's all.
  425 
  426 The slightly tricky part is how exactly we perform the replacement: we need to traverse each sequence
  427 of consecutive function applications in the pattern functor, remember all encountered arguments and
  428 if the head of consecutive applications is 'list' then rename it to 'dat' (and freshen the unique of
  429 the variable, because it's easy and it's nice not to break the global uniqueness condition, but this
  430 is not too important), apply 'dat' to all the remembered arguments and enclose the result by
  431 
  432     rec (\(dat :: * -> *) -> _)
  433 
  434 But that doesn't work in the general case. The user might write
  435 
  436     app = \(f :: * -> *) (a :: *) -> f a
  437     listF = \(list :: * -> *) (a :: *) -> all (r :: *). r -> (a -> app list a -> r) -> r
  438 
  439 i.e. a type containing a non-saturated 'list' and the outlined algorithm can't handle this case,
  440 because it always just restores all the arguments in a sequence of applications (which we have none
  441 in this example) while we need to generate the following:
  442 
  443     semAppListF =
  444         \(rec :: ((* -> *) -> *) -> *) -> \(spine :: (* -> *) -> *) ->
  445             let list = \(a :: *) -> rec (\(dat :: * -> *) -> dat a)
  446                 in spine (\(a :: *) -> all (r :: *). r -> (a -> app list a -> r) -> r)
  447 
  448 (where @let list = ... in ...@ is pseudosyntax introduced for readability)
  449 So if a recursive case is not saturated, we have to generate as many lambdas as there are missing
  450 arguments and prepend the lambdas to the encoding of the recursive case.
  451 
  452 This way we can preserve the user's redexes and not introduce additional ones.
  453 
  454 Read next: Note [Comparing approaches to pattern functor packing]
  455 -}

  457 {- Note [Comparing approaches to pattern functor packing]
  458 Packing n-ary pattern functors semantically (see Note [Packing n-ary pattern functors semantically]):
  459 Pros:
  460     1. easy to get right. Kinds match? You're all set
  461     2. does not require manual manipulations with syntax (which would be very error-prone)
  462     3. does not evaluate redexes written by the user
  463     4. pattern functors with more than one recursive case are smaller being encoded this way than
  464        when everything is fully inlined (in the latter case the overhead is O(n) where 'n' is the
  465        number of recursive occurrences in a pattern functor)
  466 Cons:
  467     1. resulting types contain additional redexes. I.e. we can turn normalized types into
  468        non-normalized ones
  469     2. pattern functors with one recursive case are slightly bigger being encoded this way than
  470        when everything is fully inlined
  471 
  472 Packing n-ary pattern functors syntactically (see Note [Packing n-ary pattern functors syntactically]):
  473 Pros:
  474     1. neither introduces new redexes nor evaluates ones written by the user
  475 Cons:
  476     1. super easy to get wrong. While implementing this approach, I got it wrong three times.
  477        I'm still not sure it's sound
  478     2. requires testing against the other way to encode n-ary pattern functors
  479     3. requires manipulations with uniques which always look fine until you get an incomprehensible
  480        error message after 82 generated test cases pass
  481     4. pattern functors with more than one recursive case are bigger being encoded this way than
  482        with the other approach (the overhead here is O(n) where 'n' is the number of recursive
  483        occurrences in a pattern functor)
  484     5. someone who generates Plutus Core does not care much about whether types are normalized,
  485        because it's a pain and if you want normalized types, just normalize what you generated
  486        in the end.
  487 
  488 Therefore, the costs of encoding n-ary pattern functors as 1-ary pattern functors in normal form
  489 are rather high while the benefits are minor, and thus we go with the semantic packing approach.
  490 -}

  501 -- is not equal to the length of the spine that @_recursiveType@ contains.

  530 -- > spineKindToRecKind _ spineKind = spineKind -> *
  531 spineKindToRecKind :: ann -> Kind ann -> Kind ann
  532 spineKindToRecKind ann spineKind = KindArrow ann spineKind $ Type ann

  534 -- | Make a function that packs a list of 'TyDecl's as a spine using the CPS trick.
  535 --
  536 -- > getToSpine _ =
  537 -- >     \[a1 :: k1, a2 :: k2 ... an :: kn] -> (dat :: k1 -> k2 -> ... kn -> *) -> dat a1 a2 ... an
  538 --
  539 -- For example,
  540 --
  541 -- > getToSpine _ =
  542 -- >     \[a1 :: k1, a2 :: k2] -> (dat :: k1 -> k2 -> *) -> dat a1 a2
  543 getToSpine :: ann -> Quote ([TyDecl TyName uni ann] -> Type TyName uni ann)
  544 getToSpine ann = do
  545     dat <- freshTyName "dat"
  546 
  547     return $ \args ->
  548           TyLam ann dat (argKindsToDataKindN ann $ map _tyDeclKind args)
  549         . mkIterTyApp (TyVar ann dat)
  550         $ map ((ann,) . _tyDeclType) args
  551 
  552 -- | Pack a list of 'TyDecl's as a spine using the CPS trick.
  553 --
  554 -- > getSpine _ [a1 :: k1, a2 :: k2 ... an :: kn] =
  555 -- >     \(dat :: k1 -> k2 -> ... kn -> *) -> dat a1 a2 ... an
  556 --
  557 -- For example,
  558 --
  559 -- > getSpine _ [a1 :: k1, a2 :: k2] =
  560 -- >     \(dat :: k1 -> k2 -> *) -> dat a1 a2
  561 getSpine :: ann -> [TyDecl TyName uni ann] -> Quote (Type TyName uni ann)
  562 getSpine ann args = ($ args) <$> getToSpine ann

  564 -- See Note [Packing n-ary pattern functors semantically].
  565 -- | Having a list of type variables along with their kinds, make a function that receives
  566 --
  567 -- 1. a function expecting a spine in CPS form
  568 -- 2. a sequence of types
  569 --
  570 -- packs the types into a CPS-encoded spine and passes the spine to the function.
  571 --
  572 -- > getWithSpine _ [v1 :: k1, v2 :: k2 ... vn :: kn] =
  573 -- >     \(cont :: ((k1 -> k2 -> ... -> kn -> *) -> *) -> *)
  574 -- >      (v1 :: k1) (v2 :: k2) ... (vn :: kn) ->
  575 -- >          cont \(dat :: k1 -> k2 -> ... kn -> *) -> dat v1 v2 ... vn
  576 --
  577 -- For example,
  578 --
  579 -- > getWithSpine _ [v1 :: k1, v2 :: k2] =
  580 -- >     \(cont : ((k1 -> k2 -> *) -> *) -> *) (v1 :: k1) (v2 :: k2) ->
  581 -- >          cont \(dat :: k1 -> k2 -> *) -> dat v1 v2
  582 getWithSpine
  583     :: ann
  584     -> [TyVarDecl TyName ann]
  585     -> Quote ((Type TyName uni ann -> Type TyName uni ann) -> Type TyName uni ann)
  586 getWithSpine ann argVars = do
  587     spine <- getSpine ann $ map tyDeclVar argVars
  588     return $ \k -> mkIterTyLam argVars $ k spine

  590 -- See Note [Spiney API].
  591 type FromDataPieces uni ann a
  592     =  ann                     -- ^ An annotation placed everywhere we do not have annotations.
  593     -> TyName                  -- ^ The name of the data type being defined.
  594     -> [TyVarDecl TyName ann]  -- ^ A list of @n@ type variables bound in a pattern functor.
  595     -> Type TyName uni ann     -- ^ The body of the n-ary pattern functor.
  596     -> Quote a
  597 
  598 -- See Note [Packing n-ary pattern functors semantically].
  599 -- | Pack the body of an n-ary pattern functor and make the corresponding 1-ary pattern functor.
  600 --
  601 -- > packPatternFunctorBodyN _ dataName [v1 :: k1, v2 :: k2 ... vn :: kn] patBodyN =
  602 -- >     let patN = \(dataName :: k1 -> k2 -> ... -> kn -> *) (v1 :: k1) (v2 :: k2) ... (vn :: kn) ->
  603 -- >                     patBodyN
  604 -- >         in \(rec :: ((k1 -> k2 -> ... kn -> *) -> *) -> *)
  605 -- >             (spine :: (k1 -> k2 -> ... -> kn -> *) -> *) ->
  606 -- >                 spine (patN (withSpine rec))
  607 --
  608 -- For example,
  609 --
  610 -- > packPatternFunctorBodyN _ dataName [v1 :: k1, v2 :: k2] patBodyN =
  611 -- >     let patN = \(dataName :: k1 -> k2 -> *) (v1 :: k1) (v2 :: k2) ->
  612 -- >                     patBodyN
  613 -- >         in \(rec :: ((k1 -> k2 -> *) -> *) -> *)
  614 -- >             (spine :: (k1 -> k2 -> *) -> *) ->
  615 -- >                 spine (patN (withSpine rec))
  616 packPatternFunctorBodyN :: FromDataPieces uni ann (Type TyName uni ann)
  617 packPatternFunctorBodyN ann dataName argVars patBodyN = do
  618     let dataKind  = argKindsToDataKindN ann $ map _tyVarDeclKind argVars
  619         spineKind = dataKindToSpineKind ann dataKind
  620         recKind   = spineKindToRecKind  ann spineKind
  621         vDat = TyVarDecl ann dataName dataKind
  622         patN = mkIterTyLam (vDat : argVars) patBodyN
  623 
  624     withSpine <- getWithSpine ann argVars
  625 
  626     rec   <- freshTyName "rec"
  627     spine <- freshTyName "spine"
  628 
  629     return
  630         . TyLam ann rec recKind
  631         . TyLam ann spine spineKind
  632         . TyApp ann (TyVar ann spine)
  633         . TyApp ann patN
  634         . withSpine
  635         . TyApp ann
  636         $ TyVar ann rec
  637 
  638 -- | Construct a data type out of pieces.
  639 getPackedType :: FromDataPieces uni ann (Type TyName uni ann)
  640 getPackedType ann dataName argVars patBodyN = do
  641     withSpine <- getWithSpine ann argVars
  642     withSpine . TyIFix ann <$> packPatternFunctorBodyN ann dataName argVars patBodyN
  643 
  644 -- | An auxiliary type for returning a polymorphic @wrap@. Haskell's support for impredicative
  645 -- polymorphism isn't good enough to do without this.
  646 newtype PolyWrap uni fun ann = PolyWrap
  647     (forall term. TermLike term TyName Name uni fun => [Type TyName uni ann] -> term ann -> term ann)
  648 
  649 -- | Make a generic @wrap@ that takes a spine of type arguments and the rest of a term, packs
  650 -- the spine using the CPS trick and passes the spine and the term to 'IWrap' along with a 1-ary
  651 -- pattern functor constructed from pieces of a data type passed as arguments to 'getWrap'.
  652 getPackedWrap :: FromDataPieces uni ann (PolyWrap uni fun ann)
  653 getPackedWrap ann dataName argVars patBodyN = do
  654     pat1 <- packPatternFunctorBodyN ann dataName argVars patBodyN
  655     toSpine <- getToSpine ann
  656     let instVar v ty = TyDecl ann ty $ _tyVarDeclKind v
  657     return $ PolyWrap $ \args ->
  658         let argVarsLen = length argVars
  659             argsLen = length args
  660             in if argVarsLen == argsLen
  661                 then iWrap ann pat1 . toSpine $ zipWith instVar argVars args
  662                 else throw . IndicesLengthsMismatchException argVarsLen argsLen $ dataName
  663 
  664 {- Note [Special cases]
  665 The notes above describe how the general case is compiled, however for the 0-ary and 1-ary cases
  666 there are more efficient compilation schemes.
  667 
  668 Compiled generally, @fix0 (pat :: * -> *)@ looks like this:
  669 
  670     ifix
  671         (\(rec :: (* -> *) -> *) (spine :: * -> *) -> spine (pat (rec (\(dat :: *) -> dat))))
  672         (\(dat :: *) -> dat)
  673 
  674 A more efficient compilation scheme is
  675 
  676     ifix
  677         (\(rec :: (* -> *) -> *) (f :: * -> *) -> f (rec f))
  678         pat
  679 
  680 This is the one that we use in 'makeRecursiveType0'. Note however that 'makeRecursiveType*'
  681 functions always receive the body of a pattern functor and not the pattern functor itself,
  682 so in the 0-ary case we turn the body of the pattern functor into the pattern functor by
  683 prepending a lambda binding a variable with the same name as that of the data type being
  684 defined. Note also that the pattern functor of the resulting 'ifix' is always the same --
  685 it's the index that differs.
  686 
  687 Compiled generally, @fix1 (pat :: (k -> *) -> k -> *) (arg :: *)@ looks like this:
  688 
  689     \(v :: k) ->
  690         ifix
  691             (\(rec :: (* -> *) -> *) (spine :: * -> *) ->
  692                 spine (pat (\(x :: v) -> rec (\(dat :: k -> *) -> dat v))))
  693             (\(dat :: k -> *) -> dat v)
  694 
  695 But we clearly don't need to represent a single type argument as a CPS-encoded spine,
  696 because the only point of representing things like that is to get a single type argument out of
  697 @n@ type arguments and when @n = 1@ no such tricks are needed.
  698 
  699 So the encoding that we use is
  700 
  701     \(v :: k) -> ifix pat v
  702 
  703 And as in the 0-ary case we need to turn the body of a pattern functor into a pattern functor,
  704 this time not only by prepending a lambda binding a variable with the same name as that of
  705 the data type being defined, but also by prepending a lambda binding a variable representing the
  706 index.
  707 -}
  708 
  709 -- See Note [Special cases].
  710 -- | Construct a 'RecursiveType' by passing a 0-ary pattern functor to 'TyIFix' and 'IWrap'
  711 -- /as an index/.
  712 makeRecursiveType0
  713     :: ann                  -- ^ An annotation placed everywhere we do not have annotations.
  714     -> TyName               -- ^ The name of the data type being defined.
  715     -> Type TyName uni ann  -- ^ The body of the pattern functor.
  716     -> Quote (RecursiveType uni fun ann)
  717 makeRecursiveType0 ann dataName patBody0 = do
  718     rec <- freshTyName "rec"
  719     f   <- freshTyName "f"
  720     let argKind = KindArrow ann (Type ann) $ Type ann
  721         recKind = KindArrow ann argKind $ Type ann
  722         pat1
  723             = TyLam ann rec recKind
  724             . TyLam ann f   argKind
  725             . TyApp ann (TyVar ann f)
  726             . TyApp ann (TyVar ann rec)
  727             $ TyVar ann f
  728         arg = TyLam ann dataName (Type ann) patBody0
  729         -- recType =
  730         --     ifix
  731         --         (\(rec :: (* -> *) -> *) (f :: * -> *) -> f (rec f))
  732         --         (\(dataName :: *) -> patBody0)
  733         recType = TyIFix ann pat1 arg
  734         wrap args = case args of
  735             [] -> iWrap ann pat1 arg
  736             _  -> throw . IndicesLengthsMismatchException 0 (length args) $ dataName
  737     return $ RecursiveType recType wrap
  738 
  739 -- See Note [Special cases].
  740 -- | Construct a 'RecursiveType' by passing a 1-ary pattern functor to 'TyIFix' and 'IWrap'.
  741 makeRecursiveType1
  742     :: ann                   -- ^ An annotation placed everywhere we do not have annotations.
  743     -> TyName                -- ^ The name of the data type being defined.
  744     -> TyVarDecl TyName ann  -- ^ The index type variable.
  745     -> Type TyName uni ann   -- ^ The body of the pattern functor.
  746     -> Quote (RecursiveType uni fun ann)
  747 makeRecursiveType1 ann dataName argVar patBody1 = do
  748     let varName = _tyVarDeclName argVar
  749         varKind = _tyVarDeclKind argVar
  750     varName' <- freshenTyName varName
  751     let
  752         recKind = KindArrow ann varKind $ Type ann
  753         pat1 = TyLam ann dataName recKind $ TyLam ann varName varKind patBody1
  754         -- recType = \(v :: k) -> ifix (\(dataName :: k -> *) (v :: k) -> patBody1) v
  755         recType = TyLam ann varName' varKind . TyIFix ann pat1 $ TyVar ann varName'
  756         wrap args = case args of
  757             [arg] -> iWrap ann pat1 arg
  758             _     -> throw . IndicesLengthsMismatchException 1 (length args) $ dataName
  759     return $ RecursiveType recType wrap

  761 -- See all the Notes above.
  762 -- | Construct a 'RecursiveType' by encoding an n-ary pattern functor as the corresponding 1-ary one
  763 -- and passing it to 'TyIFix' and 'IWrap'. @n@ type arguments get packaged together as a CPS-encoded
  764 -- spine.
  765 makeRecursiveTypeN :: FromDataPieces uni ann (RecursiveType uni fun ann)
  766 makeRecursiveTypeN ann dataName argVars patBodyN = do
  767     recType <- getPackedType ann dataName argVars patBodyN
  768     PolyWrap wrap <- getPackedWrap ann dataName argVars patBodyN
  769     return $ RecursiveType recType wrap

  771 -- | Construct a 'RecursiveType' out of its name, variables bound in its pattern functor
  772 -- and the body of the pattern functor. The 0- and 1-ary pattern functors are special-cased,
  773 -- while in the general case the pattern functor and type arguments get encoded into a 1-ary
  774 -- form first.
  775 makeRecursiveType :: FromDataPieces uni ann (RecursiveType uni fun ann)
  776 makeRecursiveType ann dataName []       = makeRecursiveType0 ann dataName
  777 makeRecursiveType ann dataName [argVar] = makeRecursiveType1 ann dataName argVar
  778 makeRecursiveType ann dataName argVars  = makeRecursiveTypeN ann dataName argVars
