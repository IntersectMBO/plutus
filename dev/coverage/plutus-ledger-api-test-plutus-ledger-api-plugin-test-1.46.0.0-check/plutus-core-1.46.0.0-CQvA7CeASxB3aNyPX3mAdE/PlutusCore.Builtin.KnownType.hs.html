   64 {- Note [Performance of ReadKnownIn and MakeKnownIn instances]
   65 It's critically important that 'readKnown' runs in the concrete 'Either' rather than a general
   66 'MonadError'. Changing from the latter to the former gave us a speedup of up to 19%, see
   67 https://github.com/IntersectMBO/plutus/pull/4307
   68 
   69 Replacing the @AsUnliftingError err, AsEvaluationFailure err@ constraints with the dedicated
   70 'BuiltinError' data type gave us a speedup of up to 4%.
   71 
   72 All the same considerations apply to 'makeKnown':
   73 https://github.com/IntersectMBO/plutus/pull/4421
   74 
   75 It's beneficial to inline 'readKnown' and 'makeKnown' not only because we use them directly over
   76 concrete types once 'toBuiltinsRuntime' is inlined, but also because otherwise GHC compiles each of
   77 them to two definitions (one calling the other) for some reason.
   78 So always add an @INLINE@ pragma to all definitions of 'makeKnown' and 'readKnown' unless you have
   79 a specific reason not to.
   80 
   81 Neither 'readKnown' nor 'makeKnown' should appear in the generated Core for builtins. In most cases
   82 they would slow builtins down, but even if a 'readKnown' only throws an error, it still makes sense
   83 to keep it out of Core just not to trigger an investigation on whether it's fine that a call to
   84 'readKnown' is not inlined.
   85 
   86 Some 'readKnown' implementations require inserting a call to 'oneShot'. E.g. if 'oneShot' is not
   87 used in 'readKnownConstant' then 'GHC pulls @gshow uniExp@ out of the 'Nothing' branch, thus
   88 allocating a thunk of type 'String' that is completely redundant whenever there's no error,
   89 which is the majority of cases. And putting 'oneShot' as the outermost call results in
   90 worse Core.
   91 
   92 Any change to an instance of 'ReadKnownIn' or 'MakeKnownIn', even completely trivial, requires
   93 looking into the generated Core, since compilation of these instances is extremely brittle
   94 optimization-wise.
   95 
   96 Things to watch out for are unnecessary sharing (for example, a @let@ appearing outside of a @case@
   97 allocates a thunk and if that thunk is not referenced inside of one of the branches, then it's
   98 wasteful, especially when it's not referenced in the most commonly chosen branch) and type class
   99 methods not being extracted from the dictionary and used directly instead (i.e. if you see
  100 multiple @pure@ and @>>=@ in the code, that's not good). Note that neither @let@ nor @>>=@ are bad
  101 in general, we certainly do need to allocate thunks occasionally, it's just that when it comes to
  102 builtins this is rarely the case as most of the time we want aggressive inlining and specialization
  103 and the "just compute the damn thing" behavior.
  104 -}

  106 {- Note [Unlifting terminology]
  107 This function:
  108 
  109     f :: Integer -> CkValue DefaultUni fun
  110     f = VCon . Some . ValueOf DefaultUniInteger
  111 
  112 lifts an 'Integer' to 'CkValue'. Unlifting is the opposite:
  113 
  114     g :: CkValue DefaultUni fun -> Maybe Integer
  115     g (VCon (Some (ValueOf uni x))) = case uni of
  116         DefaultUniInteger -> Just x
  117         _                 -> Nothing
  118 
  119 The following usages of the "unlift" term are grammatical:
  120 
  121 1. unlift a 'CkValue' to 'Integer'
  122 2. unlift to 'Integer'
  123 3. unlift a 'CkValue' as an 'Integer'
  124 4. unlift from the 'VCon' constructor (or just 'VCon') to 'Integer'
  125 
  126 We call the integer that @g@ returns "the unlifted integer".
  127 -}
  128 
  129 {- Note [Unlifting a term as a value of a built-in type]
  130 See Note [Unlifting terminology] first.
  131 
  132 It's trivial to unlift a term to a monomorphic built-in type like 'Integer': just check that the
  133 term is a constant, extract the type tag and check it for equality with the type tag of 'Integer'.
  134 
  135 Things work the same way for a fully monomorphized polymorphic type, i.e. @(Integer, Bool@) is not
  136 any different from just 'Integer' unlifting-wise.
  137 
  138 (TODO: the following explanation needs to be improved, there's PLT-338 for that)
  139 
  140 However there's no sensible way of unlifting to, say, @[a]@ where @a@ in not a built-in type. So
  141 let's say we instantiated @a@ to an @Opaque val rep@ like we do for polymorphic functions that don't
  142 deal with polymorphic built-in types (e.g. @id@, @ifThenElse@ etc). That would mean we'd need to
  143 write a function from a @[a]@ for some arbitrary built-in @a@ to @[Opaque val a]@. Which is really
  144 easy to do: it's just @map makeKnown@. But the problem is, unlifting is supposed to be cheap and
  145 that @map@ is O(n), so for example 'MkCons' would become an O(n) operation making perfectly linear
  146 algorithms quadratic. See https://github.com/IntersectMBO/plutus/pull/4215 for how that would
  147 look like.
  148 
  149 So the problem is that we can't convert in O(1) time a @[a]@ coming from a constant of
  150 statically-unknown type (that @a@ is existential) to @[a']@ where @a'@ is known statically.
  151 Thus it's impossible to instantiate @a@ in Haskell's
  152 
  153     nullList :: [a] -> Bool
  154 
  155 so that there's a 'TypeScheme' for this function.
  156 
  157 One non-solution would be to instantiate @a@, then recurse on the type, construct a new function
  158 that defers to the original @nullList@ but wraps its argument in a specific way (more on that below)
  159 making it possible to assign a 'TypeScheme' to the resulting function. Astonishingly enough, that
  160 could actually work and if we ever write a paper on builtins, we should mention that example, but:
  161 
  162 1. such a trick requires a generic machinery that knows how to check that the head of the builtin
  163    application is a particular built-in type. We used to have that, but it was just way too slow
  164 2. that would only work for functions that don't care about @a@ at all. But for example when
  165    elaborating @cons :: a -> [a] -> [a]@ as a Plutus builtin we need to unlift both the arguments
  166    and check that their @a@s are equal
  167    (See Note [Representable built-in functions over polymorphic built-in types])
  168    and it's either way too complex or even impossible to do that automatically within some generic
  169    machinery
  170 
  171 So what we do is we simply require the user to write
  172 
  173     nullList :: SomeConstant uni [a] -> Bool
  174 
  175 and unlift to @[a]@ manually within the definition of the builtin. This works, because the
  176 existential @a@ never escapes the definition of the builtin. I.e. it's fine to unpack an existential
  177 and use it immediately without ever exposing the existential parts to the outside and it's not fine
  178 to try to return a value having an existential inside of it, which is what unlifting to @[a]@ would
  179 amount to.
  180 
  181 Could we somehow align the unlifting machinery so that it does not construct values of particular
  182 types, but rather feeds them to a continuation or something, so that the existential parts never
  183 try to escape? Maybe, but see point 2 from the above, we do want to get our hands on the particular
  184 universes sometimes and point 1 prevents us from doing that generically, so it doesn't seem like
  185 we could do that within some automated machinery.
  186 
  187 Overall, asking the user to manually unlift a @SomeConstant uni [a]@ is just always going to be
  188 faster than any kind of fancy encoding.
  189 -}

  191 {- Note [Alignment of ReadKnownIn and MakeKnownIn]
  192 We keep 'ReadKnownIn' and 'MakeKnownIn' separate, because values of some types can only be lifted
  193 and not unlifted, for example 'EvaluationResult' and 'Emitter' can't appear in argument position.
  194 
  195 'KnownTypeAst' is not a superclass of ReadKnownIn and MakeKnownIn. This is due to the fact that
  196 polymorphic built-in types are only liftable/unliftable when they're fully monomorphized, while
  197 'toTypeAst' works for polymorphic built-in types that have type variables in them, and so the
  198 constraints are completely different in the two cases and we keep the two concepts apart
  199 (there doesn't seem to be any cons to that).
  200 -}

  202 {- Note [Allowed unlifting and lifting]
  203 Read Note [Alignment of ReadKnownIn and MakeKnownIn] first.
  204 
  205 The following classes of Haskell types represent Plutus types:
  206 
  207 1. monomorphic built-in types such as @Bool@
  208    (assuming @Bool@ is in the universe)
  209 2. polymorphic built-in types such as @(a, b)@ for any @a@ and @b@ representing Plutus types
  210    (assuming @(,)@ is in the universe)
  211 3. @Opaque val rep@ for any @rep@ representing a Plutus type
  212 4. @SomeConstant uni rep@ for any @rep@ representing a Plutus type
  213 5. @Emitter a@ for any @a@ representing a Plutus type
  214 6. @EvaluationResult a@ for any @a@ representing a Plutus type
  215 7. 'TyVarRep', 'TyAppRep', 'TyForallRep' and all similar types mirroring constructors of @Type@
  216 8. @a -> b@ for any @a@ and @b@ representing Plutus types (mirrors 'TyFun')
  217 9. anything else that has a 'KnownTypeAst' instance, for example we express the
  218    @KnownTypeAst DefaultUni Int@ instance in terms of the @KnownType DefaultUni Integer@
  219    one
  220 
  221 Unlifting is allowed to the following classes of types:
  222 
  223 1. monomorphic built-in types such as @Bool@
  224 2. monomorphized polymorphic built-in types such as @(Integer, Text)@
  225 3. @Opaque val rep@ for @rep@ representing a Plutus type
  226 4. @SomeConstant uni rep@ for @rep@ representing a Plutus type
  227 5. anything else that implements 'ReadKnownIn', for example we express the
  228    @ReadKnownIn DefaultUni term Int@ instance in terms of the @ReadKnownIn DefaultUni term Integer@
  229    one, and for another example define an instance for 'Void' in tests
  230 
  231 Lifting is allowed to the following classes of types:
  232 
  233 1. monomorphic built-in types such as @Bool@
  234 2. monomorphized polymorphic built-in types such as @(Integer, Text)@
  235 3. @Opaque val rep@ for @rep@ representing a Plutus type
  236 4. @SomeConstant uni rep@ for @rep@ representing a Plutus type
  237 5. @Emitter a@ for any @a@ that lifting is allowed to
  238 6. @EvaluationResult a@ for any @a@ that lifting is allowed to
  239 7. anything else that implements 'MakeKnownIn', for example we express the
  240    @MakeKnownIn DefaultUni term Int@ instance in terms of the @MakeKnownIn DefaultUni term Integer@
  241    one, and for another example define an instance for 'Void' in tests
  242 -}

  245     :: PrettyParens (SomeTypeIn uni)
  246     => uni (Esc a)
  247     -> uni (Esc b)

  249 typeMismatchError uniExp uniAct =

  251         [ "Type mismatch: "
  252         , "expected: " ++ displayBy botRenderContext (SomeTypeIn uniExp)
  253         , "; actual: " ++ displayBy botRenderContext (SomeTypeIn uniAct)

  258 -- Normally it's a good idea for an exported abstraction not to be a type synonym, since a @newtype@
  259 -- is cheap, looks good in error messages and clearly emphasize an abstraction barrier. However we
  260 -- make 'ReadKnownM' a type synonym for convenience: that way we don't need to derive all the
  261 -- instances (and add new ones whenever we need them), wrap and unwrap all the time (including in
  262 -- user code), which can be non-trivial for such performance-sensitive code (see e.g. 'coerceVia'
  263 -- and 'coerceArg') and there is no abstraction barrier anyway.
  264 -- | The monad that 'readKnown' runs in.
  265 type ReadKnownM = Either BuiltinError

  267 -- See Note [Unlifting a term as a value of a built-in type].
  268 -- | Convert a constant embedded into a PLC term to the corresponding Haskell value.
  269 readKnownConstant :: forall val a. KnownBuiltinType val a => val -> ReadKnownM a
  270 -- See Note [Performance of ReadKnownIn and MakeKnownIn instances]
  271 readKnownConstant val = asConstant val >>= oneShot \case
  272     Some (ValueOf uniAct x) -> do
  273         let uniExp = knownUni @_ @(UniOf val) @a
  274         -- 'geq' matches on its first argument first, so we make the type tag that will be known
  275         -- statically (because this function will be inlined) go first in order for GHC to
  276         -- optimize some of the matching away.
  277         case uniExp `geq` uniAct of
  278             Just Refl -> pure x
  279             Nothing   -> throwing _UnliftingEvaluationError $ typeMismatchError uniExp uniAct
  280 {-# INLINE readKnownConstant #-}

  282 -- | A non-empty spine. Isomorphic to 'NonEmpty', except is strict and is defined as a single
  283 -- recursive data type.
  284 data Spine a
  285     = SpineLast a
  286     | SpineCons a (Spine a)
  287     deriving stock (Show, Eq, Foldable, Functor)
  288 
  289 -- | The head-spine form of an iterated application. Provides O(1) access to the head of the
  290 -- application. Isomorphic to @NonEmpty@, except is strict and the no-spine case is made a separate
  291 -- constructor for performance reasons (it only takes a single pattern match to access the head when
  292 -- there's no spine this way, while otherwise we'd also need to match on the spine to ensure that
  293 -- it's empty -- and the no-spine case is by far the most common one, hence we want to optimize it).
  294 --
  295 -- Used in built-in functions returning function applications such as 'CaseList'.
  296 data HeadSpine a
  297     = HeadOnly a
  298     | HeadSpine a (Spine a)
  299     deriving stock (Show, Eq, Functor, Foldable)
  300 
  301 -- |
  302 --
  303 -- >>> import Text.Pretty
  304 -- >>> pretty (SpineCons 'a' $ SpineLast 'b')
  305 -- [a, b]
  306 instance Pretty a => Pretty (Spine a) where pretty = pretty . map Identity . toList
  307 instance PrettyBy config a => DefaultPrettyBy config (Spine a)
  308 deriving via PrettyCommon (Spine a)
  309     instance PrettyDefaultBy config (Spine a) => PrettyBy config (Spine a)
  310 
  311 -- |
  312 --
  313 -- >>> import Text.Pretty
  314 -- >>> pretty (HeadOnly 'z')
  315 -- z
  316 -- >>> pretty (HeadSpine 'f' (SpineCons 'x' $ SpineLast 'y'))
  317 -- f `applyN` [x, y]
  318 instance Pretty a => Pretty (HeadSpine a) where
  319     pretty (HeadOnly x)     = pretty x
  320     pretty (HeadSpine f xs) = pretty f <+> "`applyN`" <+> pretty xs
  321 instance PrettyBy config a => DefaultPrettyBy config (HeadSpine a)
  322 deriving via PrettyCommon (HeadSpine a)
  323     instance PrettyDefaultBy config (HeadSpine a) => PrettyBy config (HeadSpine a)

  325 -- See Note [Performance of ReadKnownIn and MakeKnownIn instances].
  326 class uni ~ UniOf val => MakeKnownIn uni val a where
  327     -- | Convert a Haskell value to the corresponding PLC value.
  328     -- The inverse of 'readKnown'.
  329     makeKnown :: a -> BuiltinResult (HeadSpine val)
  330     default makeKnown :: KnownBuiltinType val a => a -> BuiltinResult (HeadSpine val)
  331     -- Everything on evaluation path has to be strict in production, so in theory we don't need to
  332     -- force anything here. In practice however all kinds of weird things happen in tests and @val@
  333     -- can be non-strict enough to cause trouble here, so we're forcing the argument. Looking at the
  334     -- generated Core, the forcing amounts to pulling a @case@ out of the 'fromConstant' call,
  335     -- which doesn't affect the overall cost and benchmarking results suggest the same.
  336     --
  337     -- Note that the value is only forced to WHNF, so care must be taken to ensure that every value
  338     -- of a type from the universe gets forced to NF whenever it's forced to WHNF.
  339     makeKnown x = pure . HeadOnly . fromValue $! x
  340     {-# INLINE makeKnown #-}
  341 
  342 type MakeKnown val = MakeKnownIn (UniOf val) val

  356 -- | Same as 'makeKnown', but allows for neither emitting nor storing the cause of a failure.
  357 makeKnownOrFail :: MakeKnownIn uni val a => a -> EvaluationResult (HeadSpine val)
  358 makeKnownOrFail x = case makeKnown x of
  359     BuiltinSuccess val           -> EvaluationSuccess val
  360     BuiltinSuccessWithLogs _ val -> EvaluationSuccess val
  361     BuiltinFailure _ _           -> EvaluationFailure

  364 -- | Same as 'readKnown', but the cause of a potential failure is the provided term itself.
  365 readKnownSelf
  366     :: (ReadKnown val a, AsUnliftingEvaluationError err, AsEvaluationFailure err)
  367     => val -> Either (ErrorWithCause err val) a
  368 readKnownSelf val = fromRightM (throwBuiltinErrorWithCause val) $ readKnown val

  371 instance MakeKnownIn uni val a => MakeKnownIn uni val (BuiltinResult a) where
  372     makeKnown res = res >>= makeKnown

  375 -- Catching 'EvaluationFailure' here would allow *not* to short-circuit when 'readKnown' fails
  376 -- to read a Haskell value of type @a@. Instead, in the denotation of the builtin function
  377 -- the programmer would be given an explicit 'EvaluationResult' value to handle, which means
  378 -- that when this value is 'EvaluationFailure', a PLC 'Error' was caught.
  379 -- I.e. it would essentially allow us to catch errors and handle them in a programmable way.
  380 -- We forbid this, because it complicates code and isn't supported by evaluation engines anyway.
  381 instance
  382         ( TypeError ('Text "‘BuiltinResult’ cannot appear in the type of an argument")
  383         , uni ~ UniOf val
  384         ) => ReadKnownIn uni val (BuiltinResult a) where
  385     readKnown _ = throwUnderTypeError
  386     {-# INLINE readKnown #-}

  390         , uni ~ UniOf val
  391         ) => MakeKnownIn uni val (EvaluationResult a) where
  392     makeKnown _ = throwUnderTypeError
  393     {-# INLINE makeKnown #-}
