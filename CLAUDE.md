# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Development Environment

This project **requires Nix** for reproducible tooling. Enter the dev shell with:

```bash
nix develop          # Default shell (GHC 9.6, the primary development version)
nix develop .#ghc912 # GHC 9.12 shell (needed for plugin work on that version)
nix develop .#ghc910 # Other supported GHC versions: 9.8, 9.10, 9.14
nix develop .#profiled # Shell with profiling-enabled dependencies (slow to build)
```

Inside the shell, your prompt changes to `[plutus-shell]` and a menu of available commands is displayed.

## Build Commands

```bash
cabal update                   # Run this first if cabal complains about missing packages
cabal build all                # Build all packages
cabal build plutus-core        # Build a specific package
cabal test all                 # Run all tests (sometimes needs `cabal build all` first)
cabal test plutus-core         # Test a specific package
cabal test plutus-core-test --test-options="-p MyTest"  # Run tests matching a pattern
cabal run plc -- evaluate -h   # Run an executable (typed Plutus Core)
cabal run uplc -- evaluate -h  # Run an executable (untyped Plutus Core)
```

## Code Formatting and Linting

Formatting is enforced by pre-commit hooks and CI. To run manually:

```bash
pre-commit run stylish-haskell --all-files  # Format Haskell source
pre-commit run cabal-fmt --all-files        # Format cabal files
```

There are two `.hlint.yaml` files: `./` for editor hints, `.github/` for CI-enforced rules. CI builds with `-Werror` — fix all warnings.

## Regenerating Golden Tests

After changing the evaluator or compiler:

```bash
./scripts/regen-goldens.sh  # Regenerates all golden test outputs

# Or per-package:
cabal run plutus-core-test -- --accept
cabal run untyped-plutus-core-test -- --accept
cabal run plutus-tx-plugin-tests -- --accept  # Must run for EACH supported GHC version

# For plugin golden tests, run in the GHC-specific shell:
nix develop ".#ghc96"  --command cabal test plutus-tx-plugin --test-options=--accept
nix develop ".#ghc912" --command cabal test plutus-tx-plugin --test-options=--accept
```

## Architecture Overview

The compilation pipeline is: **Haskell → GHC-Core → PIR → TPLC → UPLC**

### Packages

**`plutus-core`** — The central package containing three sub-libraries:
- `plutus-core/plutus-core/` — Typed Plutus Core (TPLC): AST, parser, pretty-printer, type checker, renamer, CK machine. Module root: `PlutusCore`.
- `plutus-core/plutus-ir/` — PlutusIR (PIR): higher-level IR that compiles down to TPLC. Module root: `PlutusIR`.
- `plutus-core/untyped-plutus-core/` — Untyped Plutus Core (UPLC): the actual on-chain language with the **CEK machine** evaluator and cost model. Module root: `UntypedPlutusCore`.

**`plutus-tx`** — Runtime Haskell library used in on-chain code. Provides `PlutusTx.Prelude`, `PlutusTx.IsData`, `PlutusTx.Lift`, etc. This is what end users import in their validator scripts.

**`plutus-tx-plugin`** — GHC plugin (`PlutusTx.Plugin`) that intercepts GHC compilation and compiles annotated Haskell to UPLC. **Only supported on GHC 9.6 and 9.12**; the plugin is marked unbuildable on other versions via cabal conditionals. Golden test outputs are stored per-GHC-version.

**`plutus-ledger-api`** — Integration layer for the Cardano ledger. Provides versioned script APIs (`V1`, `V2`, `V3`) and the `MachineParameters` for on-chain execution context.

**`plutus-metatheory`** — Formal mechanized metatheory of Plutus Core written in **Agda**. The Agda code is compiled to Haskell via MAlonzo for use in `plutus-conformance`. Located in `plutus-metatheory/src/`.

**`plutus-conformance`** — Test suite validating that the Haskell CEK machine and the Agda CEK machine produce identical results. Test cases are `.uplc` files under `test-cases/`.

**`plutus-benchmark`** — Performance benchmarks for on-chain scripts. Used for tracking regressions.

**`cardano-constitution`** — The Cardano governance constitution validator.

**`agda-certificates/`** — Agda certification files (`.agda-cert`) generated by the Agda certifier for compiled Plutus scripts.

### Key Architectural Concepts

**Global Uniqueness Property**: After renaming, every name in a TPLC/UPLC program is bound exactly once. Many passes require this invariant; transformations may break it and require a re-renaming pass.

**Cost model**: UPLC execution is metered by the CEK machine using `ExBudget` (CPU + memory). Cost model parameters live in `plutus-core/cost-model/`. The `ExMemoryUsage` typeclass defines memory costs for built-in arguments.

**Builtins**: Built-in functions are defined in `PlutusCore.Default.Builtins` and `UntypedPlutusCore.Evaluation.Machine.Cek.`. Each builtin has a denotation (Haskell implementation) and a cost function.

**Universe (uni)**: Plutus Core is parameterized by a "universe" type `uni` (the set of base types). `DefaultUni` is the standard universe used in production.

## Haskell Style Conventions

Key points from `STYLEGUIDE.adoc`:
- 2-space indentation; full-indent `where`.
- Lines ≤ 100 characters.
- Every top-level binding must have a type signature; most `let`/`where` bindings too.
- Kind signatures using `StandaloneKindSignatures` for types with non-`Type` kind parameters.
- Use `StrictData`; make fields strict by default.
- Always use deriving strategies (`deriving stock`, `deriving newtype`, `deriving anyclass`).
- Use `Text` instead of `String`.
- Derive `Show`, define `Pretty` for human-readable output; never define `Show` manually.
- Avoid punning: data constructors should not have the same name as the type (use `MkFoo`, not `Foo`).
- Use `{- Note [Name of note] -}` multi-line comments for design decisions, cross-referenced with `-- See Note [Name of note]` at use sites (GHC convention).

## Changelogs

Published packages (`plutus-core`, `plutus-ledger-api`, `plutus-tx`, `plutus-tx-plugin`) require changelog entries using `scriv`. Create a fragment in `<package>/changelog.d/`:

```bash
cd plutus-core && scriv create  # Creates a markdown fragment
```

CI checks for changelog entries unless the 'No Changelog Required' label is applied.

## Dependency Management

Packages come from **Hackage** and **CHaP** (Cardano Haskell Packages). Both index states are pinned in `cabal.project`. To update:

```bash
nix flake update hackage    # Update Hackage index
nix flake update CHaP       # Update CHaP index
cabal update                # Tell cabal about the new index state
```

When adding `source-repository-package` entries, update the `sha256` mapping in `nix/project.nix`. Avoid `source-repository-package` for anything that should be released.

## Troubleshooting

If `nix develop` or `cabal build` fails unexpectedly:
1. First try: `git pull --rebase origin master` then re-enter `nix develop`.
2. If still failing: `rm -r ~/.cabal/{store,packages} plutus-metatheory/_build dist dist-newstyle`

If `cabal test all` fails but building succeeds, run `cabal build all` first.

For profiling build errors with `recursionzmschemeszm...`, add to `cabal.project.local`:
```
package plutus-core
  ghc-options: -fexternal-interpreter
```
